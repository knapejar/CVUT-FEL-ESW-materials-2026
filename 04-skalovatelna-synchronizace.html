Škálovatelná synchronizace

Proč je synchronizace na vícejádrových (a do budoucna mnohajádrových) CPU nutná a zároveň velkým zdrojem výkonnostních problémů. Postupně si od naivního (rozbitého) zámku ukážeme šest způsobů, jak se to může pokazit, a jak je opraví atomické operace, volatile, kompilátorové a paměťové bariéry. Pak přijdou reálné zámky (mutex, futex, RW-lock) a jejich režie na cache-koherentním hardwaru. Závěrem pro read-mostly zátěže odvodíme a vysvětlíme Read-Copy-Update (RCU): škálovatelnou čtecí stranu bez atomických instrukcí, která „převádí problém škálovatelnosti na problém garbage collectoru“. Doplňkem je transakční paměť.

Proč na synchronizaci záleží a proč je složitá

Vícejádrové CPU jsou dnes normou a mnohajádrové přijdou zítra. Aby program takový hardware využil, musí běžet paralelně (vícevláknově). Naprosto nezávislá vlákna jsou ale k ničemu – vlákna spolu musí komunikovat (synchronizovat se). Právě synchronizace přitom bývá velkým zdrojem výkonnostních problémů.

💬 Paul McKenney

„Naprogramovat korektní synchronizační primitivum je jako spáchat dokonalý zločin. Existuje nejméně 50 věcí, které se mohou pokazit, a jste-li mimořádně zkušený génius, možná zvládnete předvídat a ošetřit 25 z nich.“

Vícejádrový systém je ve skutečnosti distribuovaný systém. Každé CPU má vlastní cache a store buffer a s ostatními komunikuje přes sdílený interconnect (sběrnici).

CPU0store buf. · L1$ · L2$
CPU1store buf. · L1$ · L2$
CPU2store buf. · L1$ · L2$
CPU3store buf. · L1$ · L2$
CPU interconnect – sdílená sběrnice mezi všemi CPU
Paměť – sdílená hlavní paměť
Vícejádrový systém jako distribuovaný systém: každé CPU má vlastní store buffer a cache (L1$/L2$), komunikuje přes sběrnici.
  • Data jsou cachována na mnoha místech (distribuovaný systém).
  • Ostatní CPU nevidí změny okamžitě.
  • Mimopořadové (out-of-order) vykonávání.
  • Protokoly cache koherence (např. MESIF).

Uspořádání událostí v distribuovaných systémech

  • Událost = operace čtení nebo zápisu do paměti.
  • Události lze uspořádat relací happens before („stalo se dříve“).
  • Události v jednom vlákně mají úplné (totální) uspořádání – vždy lze říct, která byla první.
  • Relace „happens before“ je obecně jen částečné uspořádání: mohou existovat dvě události P a Q, kde ani jedna nepředchází druhé.
  • Události v různých vláknech lze uspořádat, jen pokud mezi nimi existuje relace happens before. Tu poskytují jen některé CPU instrukce – viz atomické operace s release a acquire sémantikou.

Příklad happens-before řetězce vytvořeného mutexem: mutex.unlock() ve vlákně 1 se stane dříve než mutex.lock() ve vlákně 2, takže vlákno 2 vidí změny vlákna 1.

Naivní synchronizace a její 6 problémů

Data má v jeden okamžik měnit nanejvýš jedno vlákno. Naivní pokus o zámek vypadá takto:

bool locked = false;
void func() {
  while (locked == true)
  { /* busy wait */ }
  locked = true;
  data++;
  locked = false;
}

Kód v „zamčené“ oblasti se nazývá kritická sekce. Tento kód má šest problémů:

  1. Kontrola a nastavení zámku není atomická.
  2. Kompilátor může všechny přístupy k locked optimalizovat pryč.
  3. Kompilátor může přesunout přístup k data mimo kritickou sekci.
  4. Hardware může přeskupit paměťové přístupy, i když to kompilátor neudělá.
  5. Snadno vznikne deadlock.
  6. Aktivní čekání (busy waiting) plýtvá energií.

Každý z těchto problémů přednáška postupně řeší.

Atomické operace a jejich cena

Problém 1 ukazuje neatomický inkrement. data++; se v assembleru přeloží na inc ($data), což je atomické (nepřerušitelné) jen v rámci jednoho CPU. Hardwarově jde o čtení sběrnice → ALU → zápis sběrnice. Když dvě CPU dělají data++ najednou, dojde k ztracenému zápisu (výsledek je 1 místo 2):

CPU0CPU1data
bus read0
ALUbus read0
bus writeALU1
bus write1

Atomické operace zajišťují, že operace (typicky read-modify-write) je atomická i na úrovni hardwaru (sběrnice). Typická atomická operace je compare-and-swap (CAS), na x86 instrukce cmpxchg. V C: TYPE __atomic_exchange_n(TYPE *ptr, TYPE val, int memorder) – atomicky nastaví *ptr na val a vrátí starou hodnotu. Naivní zámek se s atomickou výměnou opraví:

void lock() {
  while (__atomic_exchange_n(&locked, true, …) == true)
  { /* busy wait */ }
}

Atomické operace v C a C++

  • Dlouho nebyly atomické operace v C/C++ standardizované (nekompatibilní rozšíření, inline assembler).
  • C11 a C++11 zavedly paměťový model s vědomím vláken a platformově nezávislé atomické operace.
  • C11: #include <stdatomic.h>, funkce atomic_*.
  • C++11: #include <atomic>, šablona std::atomic: 
    std::atomic<int> x;
x++; // atomický inkrement

⚠️ Atomické operace jsou pomalé

Atomické operace jsou pomalejší než neatomické! CPU provádějící atomickou operaci musí zajistit, že žádné jiné CPU nemá stejnou hodnotu v cache. To vyžaduje komunikaci (synchronizaci) s ostatními CPU. Atomická operace totiž potřebuje výhradní přístup (z pohledu protokolu cache koherence) k paměťovému místu, což v nejhorším případě znamená broadcast přes interconnect.

Kompilátorové optimalizace a kompilátorové bariéry

Naivní kód (vlevo) je rozbitý a kompilátor ho může rozbít ještě víc. Oprava (vpravo) přidá volatile a kompilátorové bariéry:

bool locked;

while (locked) {}
locked = true;
data++;
locked = false;

→

#define barrier() asm volatile("" : : : "memory")

volatile bool locked;

while (locked) {}
locked = true;
barrier();
data++;
barrier();
locked = false;
  • Kompilátor předpokládá, že paměť mění jen jednovláknový program, který právě překládá (pokud nepoužívá atomiky).
  • Proměnná locked se jeví zbytečná → optimalizována pryč. Brání tomu klíčové slovo volatile.
  • Kompilátor smí přeskupovat operace, dokud zůstane stejný výsledek jednovláknového výpočtu. barrier() mu brání přesouvat paměťové operace přes bariéru.

Jen vybrané přístupy jako volatile

Označit proměnnou volatile učiní všechny přístupy „volatile“, tedy pomalé. Někdy potřebujeme volatile sémantiku jen u určitých přístupů:

#define ACCESS_ONCE(x) (*(volatile typeof(x) *)&(x))
#define LOAD_SHARED(p) ACCESS_ONCE(p)
#define STORE_SHARED(x, v) ({ ACCESS_ONCE(x) = (v); })
#define barrier() asm volatile("" : : : "memory")

⚠️ barrier() je jen kompilátorová bariéra

Makro barrier() je pouze kompilátorová bariéra, nikoli hardwarová – kompilátor sice instrukce nepřeskupí, ale CPU je při běhu přeskupit může.

Důsledek na x86: x86 může přeskupit stores za loads, takže proměnná data může být přečtena dřív, než ostatní CPU uvidí locked nastavené na true! Proč? Zápisy mohou ve store bufferu čekat na vlastnictví cache-line; nečekat s následnými čteními zvyšuje výkon. Řešením je vložit instrukce paměťové bariéry.

Bariéry (fence) – instrukce

  • Zajistí, že všechny paměťové operace zadané před bariérou dokončí dřív, než se zadá libovolná operace za bariérou.
  • Plné bariéry brání přeskupení všech paměťových operací.
  • Slabé bariéry brání přeskupení jen loadů, nebo jen storeů.
  • Příklady: mfence (plná bariéra), lfence (slabá) na x86, dmb na ARM.
  • Implementace RCU používají platformově nezávislý obal smp_mb().

Specifikace paměťového uspořádání (memory ordering) v C/C++

Parametr order u atomic::load/store určuje, jak se kolem atomické operace uspořádají běžné (neatomické) paměťové přístupy:

relaxed
Žádná režie, žádná záruka pořadí (jen atomicita).
consume
Mezistupeň závislého uspořádání.
acquire
Žádné čtení ani zápisy aktuálního vlákna nelze přeskupit před tento load. Všechny zápisy z jiných vláken, která release stejnou atomickou proměnnou, jsou viditelné.
release
Žádné čtení ani zápisy aktuálního vlákna nelze přeskupit za tento store. Všechny zápisy aktuálního vlákna jsou viditelné ve vláknech, která acquire stejnou proměnnou.
acq_rel
Kombinace acquire i release.
seq_cst
Vysoká režie, sekvenční konzistence.

Podle architektury způsobí různý order vygenerování bariérových instrukcí (např. lfence na x86).

Příklad happens-before vytvořeného acquire/release: release store ve vlákně 1 se stane dříve než odpovídající acquire load ve vlákně 2, takže vlákno 2 garantovaně vidí data == 1:

int data;
std::atomic<int> owner;

// Thread 1                                          // Thread 2
data = 1;
owner.store(2, std::memory_order_release);  -------->  if (owner.load(std::memory_order_acquire) == 2)
                                                       read(data);

Deadlock a spinlock

Příklad deadlocku: jednojádrový systém, aktivně čekající (spinning) zámek, dvě vlákna – nízkoprioritní (LP) a vysokoprioritní (HP).

LP_thread: lock()nízká priorita získá spinlock
LP_thread: data++;
preempce
HP_thread: lock()deadlock – donekonečna spinuje na zámku, který drží neběžící LP
Jednojádro + spinlock: vysokoprioritní HP preemptuje LP držící zámek a navždy spinuje.

LP vlákno drží spinlock; je preemptováno vysokoprioritním HP vláknem, které donekonečna spinuje na zámku, jejž drží (teď neběžící) LP vlákno.

Řešení: když zámek není dostupný, požádej plánovač OS, ať tě uspí a probudí, jakmile bude zámek volný. Problém: atomicita kontroly zámku a uspání → vyžaduje implementaci v jádře OS.

Spinlocky

  • Zámek, který nezapojuje plánovač OS a čeká spinováním (busy waiting), je spin lock.
  • Aplikace spinlocky téměř nikdy nepotřebují – plýtvají energií, CPU časem a hrozí deadlock.
  • Používají se v jádrech OS tam, kde se nesmí spát:
    • Spát mohou jen vlákna (úlohy, procesy).
    • Obsluhy přerušení spát nemohou → ovladače zařízení často používají spinlocky.
    • Spinlock musí použít i plánovač OS, protože nemůže uspat sám sebe.

Reálná synchronizace: mutex, semafory, futex

Mutex

  • Nejčastější synchronizační primitivum, jednoduché API (lock/unlock).
  • Lze implementovat různě: kernel semafor, futex, …
  • Všechny implementace (aspoň na vícejádrech) zapojují atomické instrukce – a ty mohou být pomalé!
  • Spaní se realizuje interakcí s plánovačem OS.

Kernel semafory

  • Implementace zámku uvnitř jádra OS.
  • Každé systémové volání přidává režii (≈ 100 hodinových cyklů na moderním HW).
  • Vyplatí se jemné (fine-grain) zamykání – zámky chrání co nejméně dat, aby se předešlo lock contention.
  • I když je jemné zamykání účinné a vlákna málokdy spí, vždy se platí režie syscallu! Řešením v Linuxu je futex.

Futex

  • Nezahlcený (uncontended) mutex nikdy nejde do jádra.
  • Futex = atomický čítač v uživatelské paměti + fronta čekajících v jádře.
  • Čítač se aktualizuje atomickými instrukcemi v user-space: cmpxchg(val, expct, new) → prev.
  • futex_wait() a futex_wake() jsou syscally volané jen při kolizi (contention), kdy je nutné spát.
class mutex {
public:
  mutex() : val (0) { }

  void lock() {
    int c;
    if ((c = cmpxchg (val, 0, 1)) != 0) {
      if (c != 2)
        c = xchg (val, 2);
      while (c != 0) {
        futex_wait (&val, 2);
        c = xchg (val, 2);
      }
    }
  }
  void unlock() {
    if (atomic_dec (val) != 1) {
      val = 0;
      futex_wake (&val, 1);
    }
  }
private:
  int val;
};

Reference: U. Drepper, Futexes Are Tricky, 2011.

Futex je primitivum, ze kterého se staví: mutexy, semafory, podmínkové proměnné, thread barriers, read-write locky. Dnešní pthread_mutex_t v Linuxu je implementovaný přes futex; synchronizace v JVM přes Mark word a thin/fat locking je koncepčně totéž co futex.

💬 Proč jsou futexy záludné

Co když je futex sdílený mezi procesy a jeden proces nečekaně spadne, zatímco drží zamčený mutex? Deadlock! User-space čítač zůstane zamčený, protože jediný, kdo ho mohl odemknout, spadl. Jádro o zamčení (v uncontended případě) neví. Linux to řeší tzv. „Robust futex“: adresy čítačů se registrují u jádra; když proces spadne, jádro prohledá jeho user-space paměť po zamčených mutexech a odemkne je.

Cena atomik (fyzika), režie zamykání a škálování

Jeden sdílený atomický řádek serializuje stroj a je omezen fyzikou šíření signálu:

  • Světlo za 1 cyklus @ 3 GHz urazí round-trip jen 5 cm.
  • Rychlost elektronů v tranzistorech: 0,03–0,3 C (zlomek rychlosti světla).

Dva scénáře na NUMA topologii (4 packages × 4 CPU):

  • Všechna CPU atomicky inkrementují globální proměnnou: každé CPU má cache miss (ostatní k proměnné také sahají) ⇒ sekvenční vykonání, žádné škálování.
  • Všechna CPU atomicky inkrementují per-cpu proměnnou: žádný cache miss ⇒ mnohem rychlejší a paralelní.

Pointa: kolize na jediném sdíleném cache-line serializuje stroj a je omezena fyzikou; per-CPU data cache miss nevolají a škálují.

Režie zamykání

Klasické zámky se implementují atomickými instrukcemi a brání přeskupení manipulace se zámkem vůči obsahu kritické sekce:

pthread_mutex_lock(mutex);
x++;
pthread_mutex_unlock(mutex);
  • Uncontended (nezahlcený) případ: při lock() není mutex v cache; při unlock() už v cache je.
  • Contended (zahlcený) případ: mutex není v cache ani při unlock, protože pravděpodobně jiné CPU se snaží mutex zamknout a „krade“ ho z cache vlastníka.

Časování pro jednoinstrukční kritickou sekci (vlastní práce x++ je 1 cyklus, dominuje jí provoz cache kolem lock/unlock):

Lock~257 cyklů (cache miss dominuje)
x++1 cyklus
Unlocklevný – mutex už v cache
Uncontended (nezahlcený): drahý Lock + 1cyklový x++ + levný Unlock.
Lock~257 cyklů
x++1 cyklus
Unlock~257 cyklů – mutex „ukraden“ z cache
Contended (zahlcený, bez spinování): drahý Lock i Unlock, mezi nimi 1cyklový x++.
  • Uncontended: Lock ≈ 257 cyklů (worst case) + x++ ≈ 1 cyklus + levný Unlock.
  • Contended (bez spinování): Lock ≈ 257 cyklů i Unlock ≈ 257 cyklů, mezi nimi x++ ≈ 1 cyklus.

Read-mostly zátěž a Read-Write lock

Vzájemné vyloučení v masivně paralelní read-mostly zátěži má dvě ceny:

  1. Režie lock/unlock.
  2. Mrtvý čas (dead time) během aktualizací.

Jediný mutex pustí do kritické sekce vždy jen jedno CPU, takže i čistí čtenáři se serializují a vzniká nečinný „Dead Time“:

CPU 0Reader → čekání / blokováno
CPU 1blokováno → Reader
CPU 2Reader → Dead Time!!! → Reader
CPU 3blokováno → Updater
Jediný mutex serializuje i čisté čtenáře → vzniká nečinný „Dead Time“.

Mnoho reálných zátěží je „read-mostly“: cachování (web cache, CDN, souborový systém, DNS) – většina uživatelů jen čte a cache se aktualizuje jen občas.

Read-Write lock

  • Klasické řešení: read-write lock.
  • Více čtenářů může číst současně.
  • Aktualizace blokuje všechny čtenáře.
  • Lze postavit nad mutexem; škáluje špatně kvůli atomickým instrukcím v lock/unlock.
CPU 0lock/Reader/unlock … → Wait → lock/Reader(updated)/unlock
CPU 1lock/Reader/unlock … → Wait → Reader(updated)
CPU 2lock/Reader/unlock … → Wait → Reader(updated)/unlock
CPU 3lock/Reader/unlock → Wait → lock/Updater/unlock → lock/Reader(updated)/unlock
RW-lock: více čtenářů paralelně, ale aktualizace (Updater) blokuje všechny čtenáře (Wait).

Jednoduchá implementace RW-locku přes semafor – první čtenář získá semafor (zablokuje zapisovatele), poslední ho uvolní:

struct rwlock {
  semaphore_t sem;
  mutex_t readers_lock;
  unsigned readers_active;
};

void write_lock(rwlock &rwl) {
  rwl.sem.wait();
}
void write_unlock(rwlock &rwl) {
  rwl.sem.post();
}

void read_lock(rwlock &rwl) {
  rwl.readers_lock.lock();
  rwl.readers_active++;
  if (rwl.readers_active == 1)
    rwl.sem.wait();
  rwl.readers_lock.unlock();
}
void read_unlock(rwlock &rwl) {
  rwl.readers_lock.lock();
  rwl.readers_active--;
  if (rwl.readers_active == 0)
    rwl.sem.post();
  rwl.readers_lock.unlock();
}

Varianta přes mutex + podmínkovou proměnnou s preferencí zapisovatele (read_lock čeká, dokud je wr_waiting nebo wr_active):

struct rwlock {
  mutex_t lock;
  cond_t cond;
  unsigned rd_active;
  unsigned wr_waiting;
  bool wr_active;
};

void write_lock(rwlock &rwl) {
  rwl.lock.wait();
  rwl.wr_waiting++;
  while (rwl.rd_active || rwl.wr_active)
    cond_wait(rwl.cond, rwl.lock);
  rwl.wr_waiting--;
  rwl.wr_active = true;
  rwl.lock.unlock();
}
void write_unlock(rwlock &rwl) {
  rwl.lock.wait();
  rwl.wr_active = false;
  cond_broadcast(rwl.cond);
  rwl.lock.unlock();
}

void read_lock(rwlock &rwl) {
  rwl.lock.lock();
  while (rwl.wr_waiting || rwl.wr_active)
    cond_wait(rwl.cond, rwl.lock);
  rwl.rd_active++;
  rwl.lock.unlock();
}
void read_unlock(rwlock &rwl) {
  rwl.lock.lock();
  rwl.rd_active--;
  if (rwl.rd_active == 0)
    cond_broadcast(rwl.cond);
  rwl.lock.unlock();
}

Co bychom chtěli

Ideál: čtenáři běží bez režie a aktualizátor neblokuje čtenáře. Čtenáři, kteří začali před aktualizací, vidí starou verzi; ti po aktualizaci vidí novou; obě skupiny i aktualizátor mohou běžet souběžně. Je to možné? Ano – přesně to poskytuje RCU.

Ideál RCU: čtenáři běží bez režie a aktualizátor (Updater) je neblokuje – vše souběžně.

Škálovatelnost čtecí strany

Graf počet jader (0–70) vs počet čtení/s (0 až ~9×10⁹) řadí primitiva od nejlepší po nejhorší škálovatelnost čtecí strany:

  1. QSBRCU – quiescent-state-based RCU
  2. Signal-based RCU
  3. General-purpose RCU
  4. Per-thread mutex
  5. pthread mutex
  6. pthread reader-writer lock
  • RCU škáluje – typicky téměř lineárně do stovek až tisíců CPU (křivky RCU strmě rostou, dosahují miliard čtení/s).
  • Je to možné, protože RCU na čtecí straně nepoužívá atomické instrukce.
  • Atomiky nejsou nutné, protože žádná dvě vlákna nezapisují do stejné proměnné.
  • Zamykání neškáluje – křivky pthread mutex / RW-lock zůstávají u dna. Po přiblížení (y do 9×10⁶) obě spadnou z pár milionů čtení/s na ~1×10⁶, jakmile soutěží víc jader, a zůstanou ploché. RW-lock je trvale nejníž. Zamykání se s více jádry zhoršuje, ne zlepšuje.

Aktualizace RCU-chráněného seznamu

Aktualizace jednosměrného seznamu A → B → C, kde prvek B nahradíme aktualizovaným prvkem D, probíhá v 5 krocích:

1Původní seznamA → B → C
2Zkopíruj BA → B → C; kopie B(copy) → C
3Uprav kopii na DA → B → C; D(updated copy) → C
rcu_assign_pointer
4Publikuj DA → D → C (ukazatel A přehozen z B na D)
grace period
5Uvolni BA → D → C
Aktualizace RCU-chráněného seznamu: B nahrazeno prvkem D; B se uvolní až po grace period.
  1. Původní seznam A → B → C.
  2. Zkopíruj B – alokuj nový uzel a zkopíruj obsah B; jeho next už ukazuje na C.
  3. Uprav kopii na D – kopii změň na novou hodnotu D (stále D → C); B je nezměněné a stále dosažitelné.
  4. Zviditelni D čtenářům – atomicky přehoď ukazatel z A z B na D (A → D → C). Noví čtenáři vidí D; ti, co už načetli ukazatel na B, dál používají B.
  5. Počkej, až všichni čtenáři přestanou používat B, a uvolni ho – po grace period uvolni B, zůstane A → D → C.

💡 Klíčový poznatek

Kroky 1–4 jsou triviální; učinit efektivním krok 5 je důvod, proč RCU existuje! RCU převádí problém škálovatelnosti na problém garbage collectoru.

Jak RCU pozná, že žádný čtenář už nepřistupuje ke starým datům?

  • Žádné explicitní (a drahé) sledování každého čtenáře – žádné reference counting, žádné zámky ve čtenářích.
  • RCU používá nepřímý způsob určení konce všech čtecích sekcí.
  • V některých implementacích (QSBR) má čtecí strana nulovou režii.

Tři hlavní mechanismy RCU

  1. Publikace aktualizací (krok 3→4) – zajistit, že aktualizovaná data se dostanou do paměti dříve než aktualizovaný ukazatel. Realizuje kompilátorová + paměťová bariéra (bez atomik) v rcu_assign_pointer().
  2. Přístup k novým verzím dat – zajistit, že uvidíme všechny aktualizace před publikací. Kompilátorová + paměťová bariéra v rcu_dereference().
  3. Čekání, až všichni čtenáři skončí – ta záludná část! ⇒ synchronize_rcu().

💬 RCU a Java

Má smysl používat RCU v Javě? Ne. JVM rozhoduje o uvolnění objektů trasovacím garbage collectorem, který má mnohem vyšší režii než RCU (stop-the-world, režie značkování, …).

RCU API, grace period a pravidla pro čtenáře

Terminologie RCU:

Read-side critical section
Ohraničená rcu_read_lock() / rcu_read_unlock(); uvnitř se ukazatele načítají přes rcu_dereference().
Quiescent state (klidový stav)
Jakýkoli kód mimo čtecí kritickou sekci.
rcu_assign_pointer()
Aktualizátor jím publikuje novou verzi.
Grace period
Interval spuštěný a vyčkaný funkcí synchronize_rcu(). Začíná po publikaci a končí, jakmile každé CPU prošlo aspoň jedním klidovým stavem, tj. žádný čtenář už nemůže držet odkaz na stará data.
„Must not happen!“
Čtecí kritická sekce nesmí přesáhnout přes celou grace period tak, aby ještě odkazovala na uvolněná data po Free. Čtenář, který začal před publikací, musí skončit před koncem grace period – design garantuje, že grace period čeká dost dlouho.

Abstraktní rozhraní RCU:

void rcu_read_lock();
void rcu_read_unlock();
void synchronize_rcu();
void rcu_assign_pointer(type *ptr, type *new_ptr);
type *rcu_dereference(type *ptr);

Strana čtenáře

rcu_read_lock(); /* Start kritické sekce. */
p = rcu_dereference(cptr);
/* *p garantovaně existuje. */
do_something_with(p);
rcu_read_unlock(); /* Konec kritické sekce. */
/* Od teď může aktualizátor *p uvolnit!!! */
  • rcu_read_lock()/unlock() a rcu_dereference() jsou levné, někdy prázdné funkce (nop).
  • Aktualizátoři jsou výrazně těžší.

Proč rcu_dereference() a ne prosté čtení ukazatele? Naivní čtení je nebezpečné ze tří důvodů:

  1. Na DEC Alpha se může čtení ukazatele provést až po čtení skrz něj (dependent-load reordering).
  2. Kompilátor může vynechat opětovné načtení a znovu použít starou hodnotu.
  3. Kompilátor může cptr načíst vícekrát – pokud se mezi přístupy změní, použijí různou hodnotu p.

Strana aktualizátora

pthread_mutex_lock(&updater_lock);
old_p = cptr;
*new_p = *old_p;                 // kopie, je-li třeba
new_p->... = ...;
rcu_assign_pointer(cptr, new_p); // aktualizace
pthread_mutex_unlock(&updater_lock);
synchronize_rcu();               // počkej na konec grace period
free(old_p);
  • Aktualizátor může používat zámky, protože jde o read-mostly zátěž s vzácnými aktualizacemi.
  • Při jediném aktualizátoru zámky nejsou potřeba.
  • synchronize_rcu() lze nahradit neblokujícím call_rcu(), který po konci grace period zavolá callback.

Implementace finální fáze: QSBR a general-purpose RCU

Možných implementací je mnoho; RCU je populární díky použití v jádře Linuxu. Následují user-space implementace, mezi nimiž je kompromis mezi režií čtecí strany a omezeními struktury aplikace: QSBR a general-purpose.

Quiescent-state based reclamation (QSBR)

Globální čítač rcu_gp_ctr inkrementují aktualizátoři; per-thread čítače rcu_reader.ctr aktualizují čtenáři v rcu_quiescent_state. Vlákno se registruje, periodicky volá rcu_quiescent_state (zkopíruje globální čítač do lokálního) a aktualizátor v synchronize_rcu čeká, až všechny per-thread čítače dohoní novou globální hodnotu.

// Chrání registry před souběžnými přístupy
pthread_mutex_t rcu_gp_lock =
    PTHREAD_MUTEX_INITIALIZER;
LIST_HEAD(registry);

struct rcu_reader {
  // bit 0 = online, bity 1-63 = čítač
  unsigned long ctr;
  struct list_head node;
  pthread_t tid;
};

// per-thread proměnná
struct rcu_reader __thread rcu_reader;

void rcu_register_thread(void) {
  rcu_reader.tid = pthread_self();
  mutex_lock(&rcu_gp_lock);
  list_add(&rcu_reader.node, ®istry);
  mutex_unlock(&rcu_gp_lock);
  rcu_thread_online();
}

void rcu_unregister_thread(void) {
  rcu_thread_offline();
  mutex_lock(&rcu_gp_lock);
  list_del(&rcu_reader.node);
  mutex_unlock(&rcu_gp_lock);
}

#define RCU_GP_ONLINE 0x1
#define RCU_GP_CTR    0x2

// globální čítač - pozn.: nepřistupuje se atomickými instr.
unsigned long rcu_gp_ctr = RCU_GP_ONLINE;

static inline void rcu_read_lock(void) {}
static inline void rcu_read_unlock(void) {}

// Každé vlákno musí tuto funkci volat periodicky
// mimo čtecí kritickou sekci.
// Pozn. 1: Žádná atomická instrukce, bariéry jsou
//          levnější než atomické instrukce.
// Pozn. 2: Nevolá se na každou čtecí sekci, jen
//          "čas od času", např. v hlavní smyčce programu.
static inline void rcu_quiescent_state(void) {
  smp_mb();
  STORE_SHARED(rcu_reader.ctr, LOAD_SHARED(rcu_gp_ctr));
  smp_mb();
}

Funkce rcu_read_lock()/rcu_read_unlock() jsou prázdné – čtecí strana QSBR má skutečně nulovou režii.

Kolem blokujících syscallů (např. epoll_wait()) se vlákno musí označit jako offline. Hodnota čítače 0 značí offline (vlákno nemůže držet odkazy), takže grace period na něj nemusí čekat:

// volej před blokujícím systémovým voláním
static inline void rcu_thread_offline(void) {
  smp_mb();
  STORE_SHARED(rcu_reader.ctr, 0);
}
// volej po návratu z blokujícího systémového volání
static inline void rcu_thread_online(void) {
  STORE_SHARED(rcu_reader.ctr, LOAD_SHARED(rcu_gp_ctr));
  smp_mb();
}

Finální fáze synchronizace – synchronize_rcu a update_counter_and_wait:

// Toto volá aktualizátor
void synchronize_rcu(void) {
  unsigned long was_online = rcu_reader.ctr;
  smp_mb();
  if (was_online) STORE_SHARED(rcu_reader.ctr, 0);
  mutex_lock(&rcu_gp_lock);
  update_counter_and_wait();
  mutex_unlock(&rcu_gp_lock);
  if (was_online) STORE_SHARED(rcu_reader.ctr, LOAD_SHARED(rcu_gp_ctr));
  smp_mb();
}

static void update_counter_and_wait(void) {
  struct rcu_reader *index;
  STORE_SHARED(rcu_gp_ctr, rcu_gp_ctr + RCU_GP_CTR);
  barrier();
  list_for_each_entry(index, ®istry, node) {
    while (rcu_gp_ongoing(&index->ctr))
      msleep(10);
  }
}

static inline int rcu_gp_ongoing(unsigned long *ctr) {
  unsigned long v = LOAD_SHARED(*ctr);
  return v && (v != rcu_gp_ctr);
}

Aktualizátor zvýší globální čítač, pak cyklicky kontroluje každého registrovaného čtenáře (mezi kontrolami spí 10 ms), dokud každý čtenář není buď offline (v == 0), nebo neuvidí novou hodnotu globálního čítače (v == rcu_gp_ctr).

⚠️ Vlastnosti a omezení QSBR

  • synchronize_rcu nemusí skončit ihned, jakmile poslední CPU zavolá rcu_quiescent_state! Tolerujeme režii aktualizátorů, abychom snížili režii čtenářů.
  • Vlákna musí rcu_quiescent_state() volat čas od času (např. v hlavní smyčce) a kolem blokujících volání volat rcu_thread_off/online. Jinak grace period nikdy neskončí.
  • Grace periody se nesdílejí; dlouhé čekání → vyšší spotřeba paměti.
  • Funguje jen na 64bitových architekturách – čítač nesmí přetéct.

General-purpose RCU

Vlastnosti: neomezuje strukturu aplikace, není třeba volat rcu_quiescent_state ani rcu_thread_(on|off)line, nemá problém s přetečením (jiný mechanismus s pouhým 1bitovým čítačem fáze), podporuje vnořování čtecích sekcí (16bitový nesting counter). Cena: vyšší režie čtecí strany – jedna paměťová bariéra (stále méně než atomické instrukce u zámků).

// bit 16 = 1bitový čítač fáze
#define RCU_GP_CTR_PHASE 0x10000
// bity 0-15 = 16bitový čítač vnoření
#define RCU_NEST_MASK 0x0ffff
#define RCU_NEST_COUNT 0x1
unsigned long rcu_gp_ctr = RCU_NEST_COUNT;

static inline void rcu_read_lock(void)
{
  unsigned long tmp;
  tmp = rcu_reader.ctr;
  if (!(tmp & RCU_NEST_MASK)) {
    STORE_SHARED(rcu_reader.ctr, LOAD_SHARED(rcu_gp_ctr));
    smp_mb(); // instrukce paměťové bariéry (fence)
  } else {
    // Vnořená kritická sekce nepotřebuje bariéru
    STORE_SHARED(rcu_reader.ctr, tmp + RCU_NEST_COUNT);
  }
}

static inline void rcu_read_unlock(void)
{
  smp_mb(); // instrukce paměťové bariéry
  STORE_SHARED(rcu_reader.ctr, rcu_reader.ctr - RCU_NEST_COUNT);
}

Grace period má dvě fáze – synchronize_rcu dvakrát překlopí 1bitovou fázi a počká na čtenáře:

void synchronize_rcu(void)
{
  // Grace period má dvě fáze
  smp_mb(); // instrukce paměťové bariéry
  mutex_lock(&rcu_gp_lock);
  update_counter_and_wait(); // fáze 1
  barrier(); // kompilátorová bariéra
  update_counter_and_wait();
  mutex_unlock(&rcu_gp_lock); // fáze 2
  smp_mb(); // instrukce paměťové bariéry
}

static void update_counter_and_wait(void)
{
  struct rcu_reader *index;
  STORE_SHARED(rcu_gp_ctr, rcu_gp_ctr ^ RCU_GP_CTR_PHASE); // překlop bit fáze (^ = XOR)
  barrier(); // kompilátorová bariéra
  list_for_each_entry(index, ®istry, node) {
    while (rcu_gp_ongoing(&index->ctr))
      msleep(10);
  }
}

static inline int rcu_gp_ongoing(unsigned long *ctr)
{
  unsigned long v;
  v = LOAD_SHARED(*ctr);
  // GP fáze probíhá = čítač vnoření > 0 a globální (1bit) čítač != lokální čítač
  // => GP fáze končí když čítač vnoření == 0 (klidový stav) nebo globální (1bit) čítač == lokální
  return (v & RCU_NEST_MASK) > 0 && ((v ^ rcu_gp_ctr) & RCU_GP_CTR_PHASE);
}

Proč nestačí jediná fáze? Synchronizace by selhala v jednom okrajovém případě (viz str. 7 supplemental material URCU paperu).

Porovnání: RCU vs RW-lock vs reference counting

RCU vs RW-lock

HlediskoRW-lockRCU

Čtecí strana – škálováníŠpatné, padá z pár M na ~1×10⁶ čtení/s; s více jádry se zhoršujeTéměř lineární do stovek/tisíců CPU, miliardy čtení/s

Na 1024 CPU při častých čteních je RCU výhodné, protože čtecí strana nepoužívá atomické instrukce na sdílenou cache-line – nevznikají cache miss a koordinace přes interconnect, takže čtení škálují téměř lineárně, kdežto RW-lock se serializuje na atomikách v lock/unlock a propustnost s rostoucím počtem CPU klesá.

Shrnutí update benchmarků (logaritmické grafy z URCU paperu, 8-core Intel Xeon a 64-core POWER5): RCU má na čtecí straně trvale vysokou propustnost, do značné míry necitlivou na aktivitu aktualizací, kdežto zámkové schéma degraduje. RCU aktualizace nesou těžší cenu (čekání na grace period), což je u read-mostly zátěží přijatelné.

Transakční paměť (HW/SW)

Transakční paměť přenáší myšlenku databázových transakcí do přístupu k paměti: blok kódu se provede jako atomická transakce – buď se promítnou všechny jeho paměťové změny najednou, nebo se transakce zruší (abort) a zopakuje při konfliktu s jiným vláknem. Programátor tak nemusí ručně skládat zámky a vyhne se deadlockům; jde o optimistickou synchronizaci (předpokládá, že ke konfliktu obvykle nedojde).

  • HW transakční paměť (HTM) – konflikty mezi transakcemi detekuje hardware pomocí protokolu cache koherence (sledují se přečtené a zapsané cache-line). Příkladem je Intel TSX. Je rychlá, ale velikost transakce je omezena kapacitou cache a transakce může selhat z vnějšího důvodu (přerušení, příliš velký working set).
  • SW transakční paměť (STM) – konflikty řeší knihovna/runtime softwarově (verzování, logy, ověření při commitu). Nemá HW limity, ale má vyšší režii.

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka RCU vs RW-lock: v čem se liší? Porovnej výkonnost a škálování a) read-side b) write-side.

Princip: RW-lock dovolí více čtenářů současně, ale aktualizace blokuje všechny čtenáře; lock/unlock na obou stranách používá atomické instrukce. RCU naopak umožní čtenářům i aktualizátorovi běžet souběžně – aktualizátor zkopíruje, upraví, publikuje a starý objekt uvolní až po grace period.

a) Read-side: RW-lock škáluje špatně – kvůli atomikám na sdílené cache-line padá z pár milionů na ~1×10⁶ čtení/s a s rostoucím počtem jader se zhoršuje. RCU na čtecí straně nepoužívá atomiky (QSBR má nulovou režii, general-purpose jen paměťovou bariéru), takže škáluje téměř lineárně do stovek/tisíců CPU (miliardy čtení/s).

b) Write-side: RW-lock zapisuje přímo, ale výhradně (blokuje čtenáře). RCU má zápis dražší – aktualizátor musí počkat na konec grace period (synchronize_rcu), což zvyšuje latenci a spotřebu paměti. Proto je RCU výhodné jen pro read-mostly zátěže se vzácnými aktualizacemi.

zkoušková otázka RCU: popiš, jak implementovat finální fázi synchronizace, co nesmí čtenáři dělat, aby RCU dobře škálovalo, a jak je daná implementace efektivní.

Finální fáze = synchronize_rcu(), která čeká na konec grace period (až každé CPU projde aspoň jedním klidovým stavem). V QSBR: aktualizátor zvýší globální rcu_gp_ctr a v update_counter_and_wait() cyklicky kontroluje každého registrovaného čtenáře (mezi kontrolami msleep(10)), dokud každý není buď offline (ctr == 0), nebo nedohnal novou hodnotu globálního čítače (ctr == rcu_gp_ctr). Teprve pak lze starý objekt uvolnit. (General-purpose RCU dělá totéž ve dvou fázích s 1bitovým překlápěným čítačem fáze.)

Co čtenáři NESMÍ: aby grace period vůbec skončila a režie čtenářů zůstala nulová, vlákna musí periodicky procházet klidovým stavem – v QSBR volat rcu_quiescent_state() (např. v hlavní smyčce) a kolem blokujících syscallů volat rcu_thread_offline/online. Čtenář tedy nesmí setrvat „věčně“ v čtecí kritické sekci bez návratu do klidového stavu, jinak by grace period nikdy neskončila a paměť by se nemohla uvolnit. Uvnitř sekce také nesmí spoléhat na to, že *p zůstane platné po rcu_read_unlock().

Proč je to efektivní: čtecí strana nepoužívá atomické instrukce (jen levné bariéry, nebo v QSBR nic), takže žádná dvě vlákna nezapisují do stejné proměnné a nevzniká cache kontence. Vědomě tolerujeme vyšší režii aktualizátorů (čekání na grace period), abychom minimalizovali režii častých čtenářů – to je důvod, proč RCU škáluje na stovky/tisíce CPU.

zkoušková otázka Reference counting vs RCU: porovnej výkon čtení i aktualizace, výhody/nevýhody a kdy RCU uvolňuje objekty. (5 b)

Čtení: reference counting na každém čtenáři atomicky inkrementuje a dekrementuje sdílený čítač → cache provoz na společné cache-line, špatné škálování. RCU čtenář nedělá žádné atomiky → škáluje téměř lineárně.

Aktualizace: reference counting je levnější – objekt se uvolní okamžitě, jakmile čítač klesne na 0. RCU je dražší – musí počkat na grace period.

Kdy RCU uvolňuje objekty: až po skončení grace period, tedy když každé CPU prošlo aspoň jedním klidovým stavem a žádný čtenář už nemůže držet odkaz na starou verzi (krok 5 aktualizace seznamu / po synchronize_rcu() nebo callbackem call_rcu()).

Výhody/nevýhody: RCU má skvělé škálování čtení a neblokuje čtenáře, ale draží aktualizace a větší spotřebu paměti při dlouhém čekání; reference counting je jednoduchý a uvolňuje hned, ale sdílený atomický čítač je bodem kontence a hrozí cykly.

zkoušková otázka Porovnání RCU vs RW-lock na 1024 CPU – proč je RCU výhodné při častých čteních.

Při 1024 CPU a častých čteních RW-lock kolabuje: každé read_lock/read_unlock provádí atomickou operaci na sdílené cache-line, což vyvolá cache miss a koordinaci přes interconnect (v nejhorším broadcast). Stroj se tím fakticky serializuje a propustnost čtení s počtem CPU neroste, spíš klesá. RCU naopak na čtecí straně nepoužívá žádné atomiky – žádná dvě vlákna nezapisují do téhož místa, takže nevznikají cache miss a čtení škálují téměř lineárně i na tisíce CPU. Cenu platí jen vzácní aktualizátoři (grace period), což u read-mostly zátěže nevadí.

P4 Jakých 6 problémů má naivní busy-wait zámek?

1) Kontrola a nastavení zámku není atomická; 2) kompilátor optimalizuje locked pryč; 3) kompilátor přesune data mimo kritickou sekci; 4) hardware přeskupí paměťové přístupy; 5) snadný deadlock (jednojádro + priority); 6) busy waiting plýtvá energií. Opravy: atomická výměna (CAS), volatile + barrier(), paměťové bariéry (mfence/smp_mb()), uspání přes plánovač OS.

P4 Proč jsou atomické operace pomalé a o kolik?

CPU musí získat výhradní přístup k cache-line (z pohledu cache koherence), tj. zajistit, že stejnou hodnotu nemá v cache žádné jiné CPU – to v nejhorším vyžaduje broadcast přes interconnect. Na Intel X5550 (Nehalem) jsou atomiky 19–266× pomalejší než neatomické přístupy (off-socket CAS ~96 ns vs 0,4 ns clock period). Bariéry bývají levnější, zvlášť slabé.

P4 Co je memory ordering (acquire/release) a paměťová bariéra?

Parametr order určuje uspořádání běžných přístupů kolem atomické operace: relaxed (jen atomicita) < consume < acquire/release (vytvoří happens-before hranu) < acq_rel < seq_cst (nejvyšší režie). acquire brání přeskupení čtení/zápisů před load, release za store. Bariéra je instrukce, která zajistí, že operace před ní dokončí dřív než operace za ní: plná (mfence) brání přeskupení všeho, slabá (lfence) jen loadů/storeů; smp_mb() je platformově nezávislý obal. barrier() je jen kompilátorová bariéra.

P4 Jak funguje futex a proč je rychlejší než kernel semafor?

Kernel semafor platí režii syscallu (~100 cyklů) i v nezahlceném případě. Futex má atomický čítač v user-space + frontu čekajících v jádře: nezahlcený mutex vyřeší vše atomickou instrukcí v user-space a nikdy nejde do jádra; syscally futex_wait/wake volá jen při kontenci. „Robust futex“ navíc odemkne mutex, když jeho vlastník spadne.

P4 Vysvětli 5 kroků aktualizace RCU-chráněného seznamu.

Pro A → B → C a nahrazení B prvkem D: 1) původní seznam; 2) zkopíruj B; 3) uprav kopii na D (D → C); 4) publikuj – atomicky přehoď ukazatel A z B na D (A → D → C), noví čtenáři vidí D, staří dál B; 5) po grace period uvolni B. Kroky 1–4 jsou triviální, těžký je krok 5 – RCU „převádí škálovatelnost na garbage collection“.

P4 Proč se ke čtení ukazatele používá rcu_dereference() místo prostého čtení?

Prosté čtení je nebezpečné: 1) na DEC Alpha může proběhnout dependent-load reordering (čtení ukazatele až po čtení skrz něj); 2) kompilátor může vynechat opětovné načtení a použít starou hodnotu; 3) kompilátor může cptr načíst vícekrát a použít různé hodnoty. rcu_dereference() všem třem hazardům brání kompilátorovou a paměťovou bariérou.

P4 Rozdíl QSBR vs general-purpose RCU.

QSBR: prázdné rcu_read_lock/unlock → nulová režie čtení, ale vlákna musí periodicky procházet klidovým stavem a značit se offline kolem blokujících volání; funguje jen na 64 bitech (čítač nesmí přetéct). General-purpose: neomezuje strukturu aplikace, nepotřebuje rcu_quiescent_state ani off/online, podporuje vnořování, nemá problém s přetečením (1bitový čítač fáze, dvoufázová grace period kvůli jednomu rohovému případu), ale platí jednu paměťovou bariéru na čtecí sekci.

P4 Co je transakční paměť a jaký je rozdíl HTM vs STM?

Blok kódu se provede jako atomická transakce – buď se promítnou všechny změny, nebo se transakce zruší a zopakuje při konfliktu (optimistická synchronizace, bez ručního skládání zámků a bez deadlocků). HTM detekuje konflikty hardwarem přes cache koherenci (Intel TSX) – rychlé, ale omezené kapacitou cache; STM řeší konflikty softwarově (verzování, logy) – bez HW limitů, ale s vyšší režií.

Spinlock busy-waituje (jádro OS, přerušení); mutex umí uspat přes plánovač.

RCU
Read-side bez atomik; copy → publish → grace period → free; „škálovatelnost → garbage collection“.

Interval čekaný synchronize_rcu(); končí, až každé CPU prošlo klidovým stavem.

📌 Co si zapamatovat

  • Naivní busy-wait zámek je rozbitý 6 způsoby; opravují ho atomiky (CAS), volatile, kompilátorové a paměťové bariéry, uspání přes OS.
  • Atomiky jsou 19–266× pomalejší (potřebují výhradní cache-line, případně broadcast); jeden sdílený řádek serializuje stroj, per-CPU data škálují.
  • barrier() = jen kompilátorová bariéra; mfence/lfence/smp_mb() = hardwarové. x86 přeskupuje stores za loads (TSO).
  • Mutex = atomiky; kernel semafor platí ~100cyklový syscall vždy → Linux používá futex (do jádra jen při kontenci).
  • Pro read-mostly zátěž: mutex serializuje i čtenáře, RW-lock škáluje špatně (padá na ~1×10⁶ čtení/s).
  • RCU: čtenáři bez režie a bez atomik, aktualizátor neblokuje čtenáře; uvolnění objektu až po grace period.
  • QSBR = nulová režie čtení (klidové stavy, 64 bitů); general-purpose = jedna bariéra, vnořování, dvoufázová grace period.
  • RCU škáluje téměř lineárně na stovky/tisíce CPU; vs RW-lock i vs reference counting vyhrává na čtecí straně, prohrává na write-side (čeká na grace period).
