Bentley: datové struktury

Writing Efficient Programs (Jon Bentley, 1982), aktualizováno v MIT kurzu 6.172, je katalog zdrojových optimalizačních vzorů. První rodina se týká volby a tvaru datových struktur. Čtyři široké strategie, každá vyměňuje jeden zdroj za jiný:

  • Ubrat bity (packing) – vejít se víc dat do cache line (64 B). Encoding = reprezentovat data méně bity (datum jako celé číslo místo řetězce). Packing = naskládat víc malých hodnot do jednoho strojového slova.
  • Obětovat paměť (augmentace) – přidat redundantní data pro zrychlení častých operací. Např. pole size u spojového seznamu změní počítání z O(N) na O(1). Cena: víc paměti a složitější aktualizace.
  • Obětovat čas (precomputace) – spočítat výsledky dopředu a uložit. Např. lookup tabulky (LUT) pro goniometrii nebo počítání bitů. Cena: startovní čas a paměť.
  • Vyhnout se práci (řídkost / caching) – neukládat to, co tam není; znovu použít, co už máte. Sparsity: Compressed Row Storage (CSR) pro matice. Caching: ukládat nedávné výsledky.

Packing a encoding: komprese dat v paměti

Příklad – uložení data. Stejné datum ve čtyřech reprezentacích od největší po nejmenší:

ReprezentaceVelikostPoznámka
1. Řetězec "February 23, 2026"~18 BF e b r u a r y 2 3 , 2 0 2 6 \0
2. Struktura {int y; int m; int d;}12 By=2026, m=2, d=23
3a. Packed (těsně)4 Byear:13 | m:4 | d:5 → 22 bitů, 10 bitů nevyužito
3b. Packed (zarovnaně)4 Byear:16 | mon:8 | day:8 → plných 32 bitů

Úvaha: celkový rozsah dní ≈ 3·10⁶ (±4096 let), ⌈log₂(3·10⁶)⌉ = 22 bitů → vejde se do 4 bajtů. Implementace pomocí bit-fieldů:

typedef struct {
  unsigned int year  : 13; // 0-8192
  unsigned int month :  4; // 1-12
  unsigned int day   :  5; // 1-31
} date_t; // Uses 22 bits, fits in 4B

Zarovnání a padding: na pořadí záleží

Hardware vyžaduje zarovnání (např. int na hranici 4 B). Kompilátor proto vkládá neviditelné výplňové bajty (padding), aby zarovnání splnil → plýtvání pamětí a kapacitou cache. Pravidlo: řaď členy struktury podle velikosti od největšího po nejmenší. Porovnání pro char c; int i; char d;:

Naivně (12 B) 0: c | pad 4: i (4 bytes) 8: d | pad
Seřazeno (8 B) 0: i (4 bytes) 4: c | d | pad
Řazení členů od největšího po nejmenší šetří padding.

Úspora: 33 % (4 bajty).

  • Array of Structs (AoS): [{x,y,z}, {x,y,z}] – tradiční OOP. Rozložení: x1 y1 z1 x2 y2 z2. Pokud saháte jen na jednu složku → 33% efektivita cache.

Přínosy: plné cache line (každý načtený bajt se použije), SIMD (data jsou souvislá pro vektorové jednotky), prefetching (hardwarový prefetcher miluje sekvenční proudy). Cena: přístup ke všem polím entity i vyžaduje víc lookupů do různých polí.

Precomputace: binomické koeficienty

Pro C(n,k) = n! / (k!·(n-k)!) přímý vzorec rychle přeteče (13! > 2³²) a naivní rekurze C(n,k) = C(n-1,k-1) + C(n-1,k) stojí O(2ⁿ) volání. Řešení: předpočítat tabulku (Pascalův trojúhelník) v době inicializace nebo překladu → lookup O(1).

  {1, 0, ...}, {1, 1, ...},

Caching: nepřepočítávej nedávné výsledky

// Bez cache
double hypotenuse(double A, double B) {
  return sqrt(A*A + B*B);
}

// S cache
double hypotenuse(double A, double B) {
  if (A == cached_A && B == cached_B)
    return cached_h;
  cached_A = A; cached_B = B;
  return cached_h = sqrt(A*A + B*B);
}

Tok: [MISS] Call(3,4) → zkontroluj → spočítej → ulož → vrať 5; [HIT] Call(3,4) → zkontroluj → vrať 5. → 30 % rychlejší při hit rate 2/3.

Řídkost (sparsity) a CSR

„Nejrychlejší způsob, jak něco spočítat, je nepočítat to vůbec.“ Neukládat a nepočítat s nulami. Příklad matice 4×4 (8 nenulových prvků):

Husté násobení potřebuje n² = 16 operací. Nenulových prvků je jen 8 → řídké násobení vystačí s 8 operacemi. Compressed Sparse Row (CSR) uloží tři pole (n=4, nnz=8):

Bentley: logika

Constant folding: vyhodnoť výrazy při překladu

Konstantní výrazy ve zdrojáku vyhodnotit už při překladu, aby se nepočítaly za běhu. Moderní kompilátory to dělají automaticky pro literály; pomocí constexpr (C23) / consteval (C++) to lze vynutit.

// Before: Runtime Math
double earth_circumference(void) {
    double radius = 6371000.0;
    double diameter = 2 * radius;
    double circ = M_PI * diameter;
    return circ;
}

// After: Compile-time Constant (constexpr)
constexpr double radius = 6371000.0;
constexpr double diameter = 2 * radius;
constexpr double circ = M_PI * diameter;
double earth_circumference(void) {
    return circ;
}

Eliminace společných podvýrazů (CSE)

Stejný výraz spočítat jen jednou a výsledek znovu použít.

// Before
a = b + c;
b = a - d;
c = b + c;
d = a - d; // <-- a - d spočítáno dvakrát!

// After
a = b + c;
b = a - d;
c = b + c;
d = b;     // <-- znovu použije výsledek b

Pozor: třetí řádek nelze nahradit c = a, protože hodnota b se na druhém řádku změnila.

Klíčová identita: √u ≤ v ⟺ u ≤ v² (pro v ≥ 0). Místo drahé odmocniny porovnáme druhé mocniny:

// After: Compare Squares
bool collides(ball_t *b1, ball_t *b2) {
  double d2 =
    sq(b1->x - b2->x) +
    sq(b1->y - b2->y);
  return d2 <= sq(b1->r + b2->r);
}

Short-circuiting: přestaň vyhodnocovat, jakmile znáš výsledek

C zaručuje u && a || vyhodnocení zleva doprava se zkrácením. Místo sečíst všechny prvky a pak porovnat se zastavíme, jakmile součet překročí limit:

// After — stop, jakmile překročíme limit
bool sum_exceeds(int *A, int n, int limit) {
  int sum = 0;
  for (int i = 0; i < n; i++) {
    sum += A[i];
    if (sum > limit) return true;
  }
  return false;
}

Řazení testů podle frekvence a ceny

Nejpravděpodobnější / nejlevnější test dej dopředu. U is_whitespace je nejčastější mezera, tak ji testuj první:

// After: Frequent First
bool is_whitespace(char c) {
  if (c == ' '  || c == '\n' ||
      c == '\t' || c == '\r')
    return true;
  return false;
}

Rychlá cesta (fast path)

Před drahý výpočet vlož levný test pro brzký výstup. Běžný případ (~90 %) – míče daleko od sebe – odbavíme levným AABB testem (axis-aligned bounding box, 3 porovnání); jen vzácný případ (~10 %) spouští drahý přesný výpočet vzdálenosti.

bool collides(ball_t *b1, ball_t *b2) {
  // 1. Fast path — AABB overlap? (levné porovnání)
  if (abs(b1->x - b2->x) > b1->r + b2->r || abs(b1->y - b2->y) > b1->r + b2->r ||
      abs(b1->z - b2->z) > b1->r + b2->r) {
      return false; // 90 % vyskočí zde
  }
  // 2. Slow path — přesná vzdálenost (drahá násobení)
  double d2 = square(b1->x - b2->x) + square(b1->y - b2->y) + square(b1->z - b2->z);
  return d2 <= square(b1->r + b2->r);
}

Bentley: smyčky

Hoisting (LICM): vytáhni invariantní práci ze smyčky

Loop Invariant Code Motion – co se v každé iteraci nemění, spočítej jednou před smyčkou, ne pokaždé znovu.

// Before
void scale(double *X, double *Y, int N) {
  for (int i = 0; i < N; i++)
    Y[i] = X[i] * exp(sqrt(M_PI/2));
}

// After: Precomputed Once
void scale(double *X, double *Y, int N) {
  double factor = exp(sqrt(M_PI/2));
  for (int i = 0; i < N; i++)
    Y[i] = X[i] * factor;
}

Kompilátory to umí dobře, ale vedlejší efekty (např. errno u sqrt) jim to často blokují – proto je jistější hoistovat ručně.

Sentinely: zjednoduš podmínku smyčky

Sentinel je speciální zarážková hodnota na okraji dat, která zjednoduší test ukončení a odstraní kontrolu mezí z každé iterace. Klasické hledání dělá 2n kontrol (i<n? + test prvku); se sentinelem stačí n+1 kontrol (jen test prvku).

// After: One Check (Sentinel)
bool contains(int *a, int n, int key) {
  if (n == 0) return false;
  int last = a[n-1];
  a[n-1] = key;
  int i = 0;
  while (a[i] != key) i++;
  a[n-1] = last;
  return (i < n-1) || (last == key);
}

Loop unrolling: méně režie, víc ILP

Rozvinutí smyčky spojí víc iterací → méně skoků a víc instruction-level parallelismu (ILP). Svinutá smyčka má vysoký poměr kontrol k práci (2:1 – jeden i<n? na jedno sum += A[i]); rozvinutí na čtyři iterace (4 jednotky práce na jednu kontrolu) tento poměr sníží.

// After: Partial Unroll (4×)
int sum = 0, j;
for (j = 0; j < n-3; j += 4) {
  sum += A[j] + A[j+1] +
         A[j+2] + A[j+3];
}
for (int i = j; i < n; ++i)
  sum += A[i];

Loop fusion: lepší časová lokalita

Fúze smyček spojí dvě smyčky přes stejný rozsah → menší režie a data zůstanou v cache. U dvou oddělených smyček mohou být A[i], B[i] z první smyčky vyhozeny z cache, než doběhne druhá → načítají se dvakrát. Sloučená smyčka je drží v L1 → jednou.

// After: One Pass, Data Hot
for (int i = 0; i < n; ++i) {
  C[i] = (A[i] <= B[i]) ? A[i] : B[i];
  D[i] = (A[i] <= B[i]) ? B[i] : A[i];
}
// A[i], B[i] načteno jednou, zůstává v L1.

Eliminace zbytečných iterací: zužování mezí

Úpravou mezí smyčky se vyhneme prázdným iteracím (např. trojúhelníkové smyčky). Místo plného čtverce s testem if (i > j) iterujeme jen dolní trojúhelník:

// After: O(n^2/2), bez kontroly
for (int i = 1; i < n; ++i) {
  for (int j = 0; j < i; ++j) {
    swap(A[i][j], A[j][i]);
  }
}

Bentley: funkce

Inlining: odstraň režii volání funkce

Inlining nahradí volání funkce jejím tělem. Odpadne režie volání (push argumentů, vytvoření a uvolnění zásobníkového rámce, skok a návrat) a kompilátor může optimalizovat napříč hranicí funkce – což zase umožní vektorizaci a unrolling.

// Before
double square(double x) { return x*x; }
double sum_of_squares(double *A, int n) {
  double sum = 0.0;
  for (int i = 0; i < n; ++i) {
    sum += square(A[i]);
  }
  return sum;
}

// After
double sum_of_squares(double *A, int n) {
  double sum = 0.0;
  for (int i = 0; i < n; ++i) {
    double temp = A[i];
    sum += temp*temp;
  }
  return sum;
}

Cena: inlining zvětší kód → tlak na instrukční cache. Zablokovat lze __attribute__((noinline)).

Eliminace koncové rekurze (tail call)

Tail Call Optimization znovu použije současný zásobníkový rámec pro poslední rekurzivní volání → rekurze se změní ve smyčku, zásobník neroste.

// After: Jump (Loop)
void quicksort(int *A, int n) {
  while (n > 1) {
    int r = partition(A, n);
    quicksort(A, r);
    A += r + 1;
    n -= r + 1;
  }
}

Zhrubnutí rekurze (coarsening)

Zvětšíme základní případ a pod prahem ho odbavíme efektivnějším kódem bez režie volání. Quicksort rozděluje (partition) jen nad prahem THRESHOLD = 10, pod ním převezme práci insertion sort:

// After
#define THRESHOLD 10
void quicksort(int *A, int n) {
  while (n > THRESHOLD) { // <-- změněná podmínka
    int r = partition(A, n);
    quicksort(A, r);
    A += r + 1;
    n -= r + 1;
  }
  // base case: insertion sort
  insertion_sort(A, n);   // <-- nový base case
}

Redukce práce je jen heuristika

Vykoná-li program méně instrukcí, je vždy rychlejší? Ne nutně. Algoritmický návrh může práci dramaticky snížit (např. Θ(n log n) řazení místo Θ(n²)), ale redukce práce automaticky nesnižuje dobu běhu kvůli složitosti hardwaru:

  • instruction-level parallelism (ILP),
  • caching a paměťová hierarchie,
  • vektorizace (SIMD),
  • spekulace a predikce skoků.

Přesto je redukce práce dobrá heuristika pro zkrácení celkové doby běhu.

CPU je hardwarový interpret strojového kódu. Opakovaně načítá z paměti, čte/zapisuje registry, počítá v ALU a zapisuje zpět. 💬 Základní věta softwarového inženýrství (D. Wheeler)

„Všechny problémy v informatice lze vyřešit další úrovní nepřímosti (indirection)… kromě problému příliš mnoha úrovní nepřímosti.“ Vrstvy abstrakce vznikaly postupně: těžko psatelný binární kód → assembler (1:1 serializér) → makra, návěští → linker (symbolické zástupce → adresy) → C („přenositelný assembler“: strukturované řízení toku, typy, funkce) → moderní jazyky (C++, Rust, Go: výjimky, RAII, šablony – ale těžší mapování na strojový kód).

Kompilátor = program, který transformuje zdrojový kód na spustitelný kód. Naivní kompilátor má dvě části:

Parsuje syntaxi konkrétního jazyka (C, C++, Rust)Generuje assembler/strojový kód pro konkrétní CPU (x86-64, ARM, RISC-V, MIPS)
Produkuje Abstract Syntax Tree (AST) – strukturovanou reprezentaci zdrojePoužívá šablony pro řízení toku (smyčky, podmínky, volání)
Řeší symboly – jména → ukazatele na interní struktury kompilátoruMapuje operace na instrukce CPU (sčítání → add)
Řeší metaprogramování – šablony, makra, constexpr, reflexi
„Desugaruje“ do základních jazykových konstrukcí

Frontend a AST

Frontend rozparsuje zdrojový kód do abstraktního syntaktického stromu (AST). Parser čte zdroj a staví strom – odstraní formátování, ale zachová strukturu. Každý uzel = jedna sémantická akce. Zdrojový kód je vlastně jen serializační formát AST. Pro zdroj:

int square(int x) {
    return x * x;
}

vznikne AST:

FunctionDeclsquare : int(int)
ParmVarDeclx : int
ReturnStmt
BinaryOp '*': int
DeclRefExpr 'x' : intDeclRefExpr 'x' : int
AST funkce square: ReturnStmt → BinaryOp '*' má dva operandy x.

Uzel je v kompilátoru obyčejná datová struktura:

struct BinaryOp {
    Operator op;
    Type type;
    Expr* left;
    Expr* right;
}

Naivní backend

Naivní backend překládá uzly AST přímo na šablony instrukcí: alokuje stack sloty (jeden na proměnnou a dočasnou hodnotu), překládá výrazy a používá šablony pro řízení toku. Výraz a + b se přeloží na:

mov eax, [rbp - 4]    ; load a
mov ebx, [rbp - 8]    ; load b
add eax, ebx          ; a + b
mov [rbp - 12], eax   ; store result

Problém: tento přístup je hloupý. Pro jediné násobení v square naivní backend vygeneruje 7 instrukcí a 2 přístupy do paměti, zatímco optimum jsou 3 instrukce a 0 přístupů do paměti:

; Naive (template translation)       ; Optimal (what we want)
square:                              square:
    push   rbp                           mov    eax, edi
    mov    rbp, rsp                      imul   eax, edi
    mov    [rbp-4], edi                  ret
    mov    eax, [rbp-4]              ; 3 instructions, 0 memory accesses
    imul   eax, [rbp-4]
    pop    rbp
    ret
; 7 instructions, 2 memory accesses

Chceme přenositelné C, ale optimální assembler. Jak na to?

IR a middle-end (LLVM)

Kde optimalizovat – v AST, nebo v assembleru? Ani jedno není ideální. AST je strukturovaný strom (ForStmt, IfStmt…), assembler plochý seznam instrukcí se skoky. Řešením je mezireprezentace (IR) a control-flow graph (CFG).

Control Flow Graph (CFG)

CFG reprezentuje, jak se kód vykonává, ne jak byl napsán. Seskupí instrukce do základních bloků (uvnitř žádný skok), bloky spojí hranami (skoky) a odhalí smyčky i mrtvý kód. Uzly = bloky, hrany = tok řízení.

BB0 (entry)sum = 0; i = 0;
BB1 (loop header)if (i < n) goto BB2
true
BB2 (loop body)sum += a[i]; i++;
false → BB3 (exit): return sum;
BB3 (exit)return sum;
back edge BB2 → BB1 (smyčka)

Problém M×N a LLVM IR

Podporovat M jazyků a N architektur naivně znamená M×N kompilátorů (4 jazyky × 4 cíle = 16 implementací). Sdílená mezireprezentace to změní na M+N: každý jazyk se přeloží do IR, každý cíl z IR (4 + 4 = 8 implementací).

JazykyC · C++ · Rust · Swift
LLVM IRsdílená mezireprezentace
Cílex86-64 · ARM64 · RISC-V · WASM
Sdílené IR mění M×N na M+N implementací.

Middle-end = optimalizátor pro virtuální CPU: místo pro machine-independent optimalizace sdílené napříč jazyky i architekturami. IR je „assembler“ tohoto virtuálního CPU – příklady LLVM IR, GCC GIMPLE. Celá pipeline:

Source
FrontendClang, GCC, Rustc · AST / sémantika

IR
BackendCode gen · Regalloc, scheduling

Frontend = jazykově specifický | Middle-end = machine-independent | Backend = machine-specific

Stejné IR se na každé architektuře přeloží do jiného assembleru (jiné registry, kódování, volací konvence), ale optimalizace v middle-endu prospějí všem cílům:

; LLVM IR (shared)
define i32 @square(i32 %x) {
  %1 = alloca i32
  store i32 %x, ptr %1
  %2 = load i32, ptr %1
  %3 = load i32, ptr %1
  %4 = mul nsw i32 %2, %3
  ret i32 %4
}

Optimalizační pasy

💡 Optimalizace nejsou magie

Jsou to jednoduché vyhledávání vzorů a jejich nahrazování (pattern matching & replacement). Těžké je rozhodnout, kdy optimalizovat (heuristiky, většina problémů je NP-těžká, heuristika se může mýlit) a jak rychle (chytré datové struktury a efektivní algoritmy). Napsat rychlý kompilátor je samo o sobě cvičení v Efektivním softwaru.

Která Bentleyho pravidla zvládne kompilátor sám?

Kompilátor zvládne většinu pravidel automaticky; ta, která obecně nezvládne (vyžadují lidskou/návrhovou znalost), jsou v pravém sloupci.

LogikaConstant folding a propagace; CSE; algebraické identity; short-circuiting; řazení testů; slučování testůVytvoření rychlé cesty (fast path)
SmyčkyHoisting; loop unrolling; loop fusion; eliminace zbytečných iteracíSentinely

Kompilátor jde i nad rámec Bentleyho pravidel: datové struktury – register allocation, mem2reg, scalar replacement of aggregates (sroa), zarovnání; smyčky – vektorizace, unswitching, idiom replacement, loop fission/fusion, tiling, interchange; logika – strength reduction, dead-code elimination, branch reordering, global value numbering; funkce – unswitching, argument elimination. Je to pohyblivý cíl – vývojáři přidávají nové optimalizace.

Desítky specializovaných pasů

Moderní kompilátory aplikují desítky pasů. High-level (machine-independent) – názvy v LLVM -passes=…: constant propagation (constprop), dead code elimination (dce), dead store elimination (dse), inlining (inline), LICM (licm), loop unrolling (loop-unroll), loop deletion (loop-deletion), memcpy opt (memcpyopt), tail call elimination (tailcallelim), vektorizace (loop-vectorize), global value numbering (gvn), jump threading (jump-threading), scalar replacement of aggregates (sroa), promote memory to register (mem2reg), sparse conditional constant propagation (sccp), simplify CFG (simplifycfg), merge duplicate constants (constmerge), reassociate (reassociate).

Low-level (machine-specific): instruction selection (DAG → MachineInstr), peephole, register allocation (greedy), prologue/epilogue insertion, branch probability block placement, machine CSE, machine LICM, machine InstCombiner, X86 LEA optimize, stack slot coloring, shrink wrapping, control flow optimizer, tail duplication, copy propagation, instruction scheduling.

Příklady jednotlivých pasů

Constant propagation + folding – nahradí proměnné známými konstantami a vyhodnotí výrazy při překladu. Oba pasy se navzájem živí a běží iterativně:

// Before                  // After (-O2)
int x = 5;                 return 16;
int y = x + 3;  // → 8
return y * 2;   // → 16
; LLVM IR: 10 instructions → 1 (ret i32 16)

Dead Store Elimination – odstraní zápisy do paměti, které se přepíšou dřív, než se přečtou („nepočítej, co nepotřebuješ“):

// Before          // After (-O2)
int a = 1;  // dead  return 2;
a = 2;
return a;

Loop-Invariant Code Motion – vytáhne neměnný výpočet ze smyčky do bloku preheader:

// Before                       // After (-O2)
for (int i = 0; i < n; i++) {    int tmp = x * y;  // hoisted
    a[i] = x * y + i;            for (int i = 0; i < n; i++) {
}                                    a[i] = tmp + i;
                                 }

Function inlining – nahradí volání tělem a odemkne další pasy (zde constant folding):

// Before                            // After (-O2)
int square(int x){ return x*x; }      int main() {
int main(){ return square(5); }           return 25;  // inline → 5*5 → fold → 25
                                      }

Cena: -O3 vkládá funkce inline příliš agresivně → nafouknutí instrukční cache → někdy pomalejší než -O2.

💡 Pasy se skládají do dramatické transformace

Pasy se navzájem živí: mem2reg povýší allocy → licm vytáhne invarianty → SCEV rozpozná uzavřenou formu smyčky → instcombine zjednoduší. U funkce square se smyčka úplně eliminuje a výsledek se spočítá vzorcem select(x>1, x*(x-1), 0) + x. Vyzkoušejte: clang -S -emit-llvm -O2 -o - square-for.c.

restrict, undefined behavior, volatile

Motivace: -O2 dává až 300% zrychlení i na triviální smyčce.

// veclib.c
void vecadd(int *a, int *b, int *c, size_t n) {
    for (size_t i = 0; i < n; ++i) {
        a[i] += c[i];
        b[i] += c[i];
    }
}
gcc -O0 -o vecadd *.c && ./vecadd   # time = 0.37s
gcc -O2 -o vecadd *.c && ./vecadd   # time = 0.12s  → ~300% speedup!

Ale ani na -O2 se assembler nevektorizuje a c[i] se načítá dvakrát za iteraci:

vecadd:
    test   rcx, rcx
    je     .exit
    xor    eax, eax
.loop:
    mov    r8d, [rdx+rax*4]   ; load c[i]
    add    [rdi+rax*4], r8d   ; a[i] += c[i]
    mov    r8d, [rdx+rax*4]   ; load c[i] AGAIN!
    add    [rsi+rax*4], r8d   ; b[i] += c[i]
    add    rax, 1
    cmp    rax, rcx
    jne    .loop
.exit:
    ret

Pointer aliasing a restrict

Proč dvakrát? Kompilátor neví, jestli se ukazatele nepřekrývají. Co když a == c? Zápis do a[i] by změnil c[i], takže ho musí znovu načíst. Slíbíme-li kompilátoru pomocí restrict, že se ukazatele nikdy nepřekrývají, druhý mov zmizí a smyčka jde vektorizovat.

void vecadd(int *restrict a,
            int *restrict b,
            int *restrict c,
            size_t n) { ... }

⚠️ restrict je smlouva

restrict = váš slib kompilátoru, že se tyto ukazatele nikdy nepřekrývají. Když slib porušíte, program má nedefinované chování.

Undefined behavior (UB)

Co udělá kompilátor s tímto kódem na -O2?

if (x + 1 > x) {
    do_fast_path();
} else {
    do_slow_path();  // Může tohle vůbec nastat?
}

Přetečení znaménkového int je v C undefined behavior. Kompilátor předpokládá, že nemůže nastat, takže x + 1 > x je vždy pravda – větev else celá zmizí. Kompilátorův pohled: if (true) do_fast_path();.

volatile

volatile nutí kompilátor vygenerovat každý load a store ze zdroje a hodnotu nedržet v registru (necachovat ji). Nezbytné u memory-mapped hardwarových registrů:

volatile int *status = (volatile int *)0x40021000;
while (*status & 0x1) {  // Musí se číst znovu pokaždé!
    // wait for hardware ready
}

Bez volatile by kompilátor hodnotu načetl jednou do registru → nekonečná smyčka. Použití: memory-mapped registry, signal handlery, komunikace mezi vlákny (raději ale atomiky).

Linker, LTO a PGO

Soubory math.c a main.c se kompilují odděleně. Kompilátor vidí volání sqrt(), ale nemá jeho kód → nechá symbolický zástupce. Úkol linkeru:

  1. Spojí všechny .o soubory.
  2. Přiřadí finální paměťové adresy.
  3. Doplní symboly → reálné adresy.
  4. Odstraní nepoužité funkce (dead-code elimination) – vyžaduje -ffunction-sections + -Wl,--gc-sections.

Dynamické linkování přes PLT/GOT (lazy binding): první volání jde přes resolver (pomalu, doplní GOT), další volání jsou přímý skok (rychle). LD_BIND_NOW=1 vynutí eager rozlišení a odstraní penalizaci prvního volání.

call printf@plt

GOT entryadresa printf
1První volání (pomalu)skok na resolver → Dynamic Linker (ld.so) rozliší symbol, opraví GOT
2Další volání (rychle)přímý skok → libc.so: printf()
PLT/GOT lazy binding: první volání přes resolver, další přímým skokem.

Link-Time Optimization (LTO)

LTO dá kompilátoru globální pohled tím, že do .o souborů zapíše IR. Linker všechno IR načte → globální pohled → může inlinovat napříč soubory (odstraní „zeď“ mezi main.c a math.c). Dvě varianty:

Full LTOThinLTO
Všechna IR se sloučí do jednoho modulu a optimalizuje jednovláknově → sekvenční úzké hrdlo.Ke každému .bc se přidá summary, vznikne kombinovaný index a optimalizace běží vysoce paralelně po jednotlivých modulech.

Profile-Guided Optimization (PGO)

PGO nahradí kompilátorovo hádání naměřenými daty o běhu:

  1. Zkompiluj s instrumentací (-fprofile-generate).
  2. Spusť s reprezentativní zátěží.
  3. Překompiluj s profilovými daty (-fprofile-use).

Přínosy: lepší predikce skoků; inlining podle skutečně horkých cest; optimalizace rozložení kódu. PGO pomáhá tam, kde kompilátor jinak jen hádá, které větve a volání jsou časté – tedy u reálných programů se složitým, daty řízeným tokem řízení. Optimalizace se dají vrstvit (kumulativní zrychlení nad -O2):

⚠️ Co kompilátor NEUMÍ

Kompilátor umí: register allocation, inlining, loop unrolling & hoisting, constant folding, dead code elimination. Neumí: změnit vaše datové struktury (hustá matice → řídká, 99 % nul), změnit algoritmickou složitost (O(n²) → O(n·log n)), využít vaši znalost rozložení dat, opravit cache-nepřátelské rozložení (AoS → SoA), paralelizovat složitou logiku. Tohle zůstává na programátorovi. Pravidlo: nikdy nehádejte, co kompilátor dělá – podívejte se (godbolt.org / Compiler Explorer).

Profiling

💬 Donald Knuth

„Předčasná optimalizace je kořen všeho zla.“ Kompilátor optimalizuje, jak kód běží; profiling řekne, kde je pomalý. Reálný SW je příliš složitý na „mentální vykonání“; platí Paretovo pravidlo (80 % času v 20 % kódu). Chceme optimalizovat úzká hrdla, ne všechen kód.

Úzká hrdla se mohou skrývat kdekoli v zásobníku: aplikační kód, knihovny třetích stran, jádro OS, vzory přístupu do paměti, latence sítě, disk I/O. Najít skutečný zdroj bývá těžké.

Performance counters a perf

Většina moderních procesorů má hardwarové performance counters, které počítají hardwarové události (takty, vykonané instrukce, cache reads/hits/misses, branch misses, memory references…). Linux perf tyto čítače čte – program se kvůli profilování nemusí nijak upravovat (na rozdíl od instrumentace). Čítač je nastaven tak, aby každých N událostí vyvolal přerušení; jádro při něm uloží instruction pointer, takže vznikne histogram, kde se čas tráví. perf umí sledovat HW události (cycles, cache misses, branch misses) i SW události (page faults, context switches, tracepoints), analyzovat jednu aplikaci (proces + jádro) i celý systém, a při přepnutí kontextu čítače ukládá a obnovuje (count „per-thread“).

perf stat sečte čítače přes celý běh (rychlý přehled):

perf stat -e cycles,branch-misses,branches,cache-misses,cache-references ./vecadd

 Performance counter stats for './vecadd':
     1,898,543,656      cycles                                     (79.98%)
           267,572      branch-misses       #  0.08% of all branches
       348,090,074      branches                                   (79.95%)
        20,232,628      cache-misses        # 75.6% of all cache refs
        26,767,103      cache-references                           (80.09%)

75% cache-miss → program je memory-bound. perf list vypíše dostupné události. perf record + perf report ukáže, kde se čas tráví:

perf record -e cycles ./program     # → samples IP → perf.data
perf report                          # → interaktivní TUI: které funkce jsou horké?
perf record --call-graph dwarf ./program  # → také zaznamená call stacky

Kompilujte s -g! Bez debug info perf neumí přiřadit samply jménům funkcí. Vždy -O2 -g. Call graph lze získat třemi způsoby: --call-graph=fp (frame pointer, vyžaduje -fno-omit-frame-pointer), --call-graph=dwarf (z debug info, vyžaduje -g), --call-graph=lbr (hardwarový Last Branch Record na moderních Intel CPU, bez flagů, ale omezený hloubkou bufferu).

💬 Z praxe (lab)

Čtení čítačů standardně vyžaduje práva (proto perf nejde spustit na školních počítačích bez root). Lze povolit: echo 1 | sudo tee /proc/sys/kernel/perf_event_paranoid (na Ritchie je hodnota 0). Čítače jde číst i přímo z C/C++ přes perf_event_open, případně knihovny libpfm nebo PAPI – nevýhoda přímého přístupu je, že není jasné, kdo událost způsobil; perf to ví, protože běží v jádře a „context switchuje“ čítače per-thread.

Flamegraphy: čtěte osy správně

  • Šířka = strávený čas (jediná osa, na které záleží).
  • Výška = hloubka volání (ne pomalost!).
  • X-osa = abecedně, ne čas.
  • Barva = náhodná (žádný význam).

Časté chyby: myslet si, že vysoký zásobník = pomalý; že zleva doprava = pořadí v čase; ignorovat off-CPU čas (I/O, zámky). Nástroje: perf (Linux), samply (web UI), Hotspot (GUI nad perf daty). Pro krátké měření času lze použít C clock(), C++ high_resolution_clock (chrono) nebo Linux clock_gettime (přes rychlý virtual syscall).

zkoušková otázka Popiš fáze kompilace C/C++ – vstup a výstup každé fáze, co se děje uvnitř. Vysvětli několik optimalizací. Co je PGO a kdy pomáhá? (4 b)

Fáze:

  • Frontend – vstup: zdrojový kód; výstup: AST a posléze IR. Parsuje syntaxi, řeší symboly a metaprogramování (šablony, makra, constexpr), desugaruje do základních konstrukcí, vytvoří AST.
  • Middle-end (optimalizátor) – vstup i výstup: IR (LLVM IR / GIMPLE). Machine-independent optimalizace nad CFG (inlining, CSE, LICM, constant folding, DCE/DSE…).
  • Backend (code generator) – vstup: IR; výstup: assembler / strojový kód. Instruction selection, register allocation, instruction scheduling, vektorizace.
  • Linker – vstup: .o soubory; výstup: spustitelný soubor. Spojí objekty, přiřadí adresy, doplní symboly, odstraní mrtvý kód.

Optimalizace: constant folding/propagation (vyhodnocení konstant při překladu), dead store elimination (smaž zápisy, které se nepřečtou), LICM (vytáhni invariant ze smyčky), inlining (nahraď volání tělem a odemkni další pasy).

PGO (Profile-Guided Optimization): 1) zkompiluj s instrumentací, 2) spusť na reprezentativní zátěži, 3) překompiluj s profilem. Zlepší predikci skoků, inlining horkých cest a rozložení kódu. Pomáhá tam, kde kompilátor jinak jen hádá, které větve/volání jsou časté – tj. u reálných programů s daty řízeným tokem řízení (~10–20 % nad -O2).

zkoušková otázka Daný kód dvou funkcí: 3 optimalizace kompilátoru (jméno + popis), 1 manuální, kterou kompilátor neudělá, a proč.

3 optimalizace kompilátoru:

  • Inlining – nahradí volání druhé funkce jejím tělem, odpadne režie volání a umožní optimalizace napříč hranicí.
  • Constant folding / propagation – konstantní výrazy vyhodnotí už při překladu.
  • Strength reduction – nahradí drahou operaci levnější, např. násobení mocninou dvou (x * 2) za bitový posun (x << 1), nebo dělení za posun; dále CSE, LICM, dead store elimination.

1 manuální optimalizace, kterou kompilátor neudělá: změna datové struktury nebo algoritmu – např. AoS → SoA, hustá matice → řídká (CSR), nebo O(n²) → O(n·log n). Kompilátor to neudělá, protože nezná sémantiku ani rozložení dat – garantuje zachování pozorovatelného chování a nesmí měnit datové struktury ani algoritmickou složitost. (Stejně tak ručně přidaná rychlá cesta nebo využití znalosti, že většina hodnot je nulová.)

zkoušková otázka Napiš AST z úryvku (funkce s ifem). K čemu slouží AST při kompilaci?

AST je strom, kde každý uzel = jedna sémantická konstrukce. Pro funkci s podmínkou typicky:

FunctionDeclf : …
ParmVarDecl…
CompoundStmt → IfStmt
podmínkaBinaryOp '>' / '==' … → DeclRefExpr …
then / elsethen: CompoundStmt → ReturnStmt / … · else: CompoundStmt → …
AST funkce s podmínkou: IfStmt má podmínku, větev then a else.

K čemu slouží: je to výstup frontendu – strukturovaná reprezentace zdroje, ze které se odstraní formátování, ale zachová struktura a sémantika (typy, odkazy na deklarace). Slouží jako základ pro sémantickou analýzu (typová kontrola, řešení symbolů) a pro generování mezireprezentace (IR), nad níž pak běží optimalizace. Zdrojový kód je vlastně jen serializační formát AST.

zkoušková otázka Performance counters – jak fungují a jak se používají při profilování.

Jak fungují: jsou to hardwarové čítače v CPU, které počítají hardwarové události – takty (cycles), vykonané instrukce, cache reads/hits/misses, branch misses, memory references atd. Čítač lze nastavit tak, aby každých N událostí vyvolal přerušení; jádro při něm uloží instruction pointer (a volitelně call stack). Z mnoha vzorků vznikne histogram, kde se čas/události tráví.

Jak se používají: přes Linux perf, který program nijak neupravuje (na rozdíl od instrumentace). perf list vypíše dostupné události; perf stat -e ... sečte čítače přes celý běh (rychlý přehled, např. 75% cache-miss → memory-bound); perf record + perf report ukáže horké funkce (vzorkuje IP do perf.data); --call-graph dwarf přidá call stacky. Kompilovat je nutné s -g (mapování na jména funkcí). Čítače jde číst i přímo přes perf_event_open / libpfm / PAPI; výhoda perf je, že běží v jádře a počítá události per-thread. Režie samplingu je < 2 %.

P2 Co je IR a jaký problém řeší (M×N)?

Intermediate Representation (LLVM IR, GCC GIMPLE) je „assembler“ virtuálního CPU mezi frontendem a backendem. Bez IR potřebuje M jazyků × N architektur M×N kompilátorů; se sdíleným IR stačí M+N (každý jazyk se do IR přeloží, každý cíl z IR). Machine-independent optimalizace v middle-endu pak prospějí všem cílům najednou.

P2 Proč může restrict zrychlit kód a co znamená?

Bez restrict kompilátor předpokládá možný aliasing ukazatelů – zápis přes jeden ukazatel může změnit data za jiným, takže je musí znovu načítat a nemůže vektorizovat (klasické dvojí načtení c[i] ve vecadd). restrict je slib kompilátoru, že se ukazatele nepřekrývají → odstraní opětovné načtení → umožní vektorizaci. Porušení slibu = undefined behavior.

P2 Jaký je rozdíl mezi instrumentací a samplingem?

Instrumentace (gprof, -pg) přidá sondy do každé funkce, sleduje každé volání i časování, má režii 20–100 % a trpí observer effectem (mění to, co měříte). Sampling (perf, HW čítače) periodicky přeruší běh a zaznamená instruction pointer – režie < 2 %, bez úpravy binárky, statisticky přesné pro horké cesty.

📌 Co si zapamatovat

  • Bentleyho pravidla ve 4 rodinách snižují práci: datové struktury (packing/bit-fieldy, zarovnání velký→malý, AoS→SoA, augmentace, precomputace, caching, sparsity/CSR), logika (constant folding, CSE, porovnání mocnin místo sqrt, short-circuit, řazení testů, fast path, lookup), smyčky (hoisting/LICM, sentinely, unrolling, fúze, zúžení mezí), funkce (inlining, koncová rekurze → iterace, zhrubnutí rekurze).
  • Redukce práce je jen heuristika – kvůli ILP, cache, SIMD a predikci nemusí méně instrukcí znamenat rychleji.
  • Kompilátor = frontend (parse → AST, symboly, desugar) + middle-end (machine-independent optimalizátor nad IR / virtuálním CPU, CFG) + backend (machine-specific: instruction selection, register allocation, scheduling).
  • IR mění problém M×N jazyků/cílů na M+N. Naivní backend dá pro square 7 instrukcí; register allocation je největší výhra (→ 3 instrukce, 0 přístupů do paměti).
  • Optimalizace = pattern match & replace řízené NP-těžkými heuristikami; pasy se skládají (mem2reg → licm → SCEV → instcombine smaže celou smyčku).
  • Páky programátora: restrict (ukazatele nealiasují → vektorizace), nespoléhat na UB (znaménkové přetečení smaže větve), volatile (každý load/store pro MMIO/signal handlery).
  • Kompilátor nezmění datové struktury, algoritmickou složitost, rozložení (AoS→SoA) ani nevyužije znalost dat – to zůstává programátorovi.
  • Flagy jsou presety – vždy měř. -O2 -g je praktický default; -O3 může být pomalejší (I-cache); -march=native odemkne AVX, ale není přenositelné; -ffast-math mění výsledky. LTO (Full vs Thin) = inlining napříč soubory; PGO/AutoFDO/BOLT se vrství na ~30–60 % nad -O2.
  • Profiling: nehádej, měř (80/20). Instrumentace (gprof, 20–100 %, observer effect) vs sampling (perf, HW čítače, < 2 %). perf stat pro přehled (75% cache-miss → memory-bound), perf record/report + --call-graph dwarf na horké funkce, vždy -O2 -g. Flamegraph: šířka = čas (jediná podstatná osa), výška = hloubka, x = abecedně, barva náhodná.
