Konkurence dat v JVM: vícevláknový přístup a neblokující algoritmy

Proč korektní a rychlý souběžný kód na JVM vyžaduje pochopit, co se děje pod úrovní Java zdroje: superskalární a out-of-order procesor přehazuje paměťové operace, a tak data race vede k překvapivým výsledkům. Postupně si vybudujeme nástroje pro řízení viditelnosti a atomicity – volatile a paměťové bariéry, synchronized a ReentrantLock (a jak JVM zámky implementuje přes mark word, thin/fat a biased lock), a balík atomik postavený na hardwarovém CAS. Nakonec neblokující (lock-free) algoritmy: Treiberův zásobník, ABA problém a konkurenční kolekce. Všechny ukázky strojového kódu jsou z JVM 8 (64bit, halda <32 GB), Intel Haswell, syntaxe AT&T.

Data race a přehazování instrukcí

Data race (datový souběh) nastane, když dvě nebo více vláken přistupují ke stejné sdílené proměnné bez synchronizace, přičemž alespoň jeden přístup je zápis. Výsledek pak závisí na nepředvídatelném pořadí operací a procesor i kompilátor smí nezávislé čtení a zápisy přeházet, takže může vzniknout výsledek, který by žádné sekvenční prokládání vláken nedovolilo.

Klasický příklad: dvě vlákna sdílejí čtyři pole inicializovaná na 0:

public int A = 0;
public int B = 0;
public int C = 0;
public int D = 0;
Vlákno 1Vlákno 2
 
public void method1() {
    int r2 = A;
    B = 1;
    D = r2;
}
 
public void method2() {
    int r1 = B;
    A = 2;
    C = r1;
}

Otázka: jaké hodnoty mohou mít C a D? Intuitivně čekáme C=0, D=0, C=1, D=0 nebo C=0, D=2. Možný je ale i čtvrtý výsledek C=1, D=2, protože procesor i kompilátor smí v každé metodě prohodit nezávislé čtení a zápis.

Přehazování provádí i C2 (server) JIT kompilátor

Bytecode pro method1 (před přehozením):

 0: aload_0
 1: getfield      #2   // Field A:I
 4: istore_1

 7: putfield      #3   // Field B:I
10: aload_0

12: putfield      #5   // Field D:I

Instrukce přeházené C2 kompilátorem (RSI ukazuje na instanci objektu):

0x0000000010639924c: movl   $0x1,0x10(%rsi)    ;*putfield B
                                               ; - datarace.DataRace::method1@7  (line 11)

0x0000000010639925d: mov    0xc(%rsi),%r11d
0x00000000106399257: mov    %r11d,0x18(%rsi)   ;*putfield D

Zápis do B (putfield B) je vygenerován před čtením A (mov 0xc(%rsi),%r11d) – store byl posunut před load. Stejné přehození nastane v method2, a tak vznikne čtvrtý výsledek C=1, D=2.

CPU paměťový model a Java Memory Model (JMM)

Java Memory Model definuje pravidla, za kterých je zápis jednoho vlákna viditelný pro čtení jiného vlákna, a které přehazování je povolené. Bez explicitní synchronizace (volatile, zámky, atomiky) JMM nezaručuje, že jiné vlákno uvidí zápisy ve stejném pořadí, v jakém byly napsány ve zdrojovém kódu. Důvodem je samotná architektura procesoru.

Pipelining a superskalární vykonávání

Procesor zpracovává instrukce v pipeline – jednotlivé fáze (IF – fetch, ID – decode, EX – execute, MEM – paměťový přístup, WB – write back) se u po sobě jdoucích instrukcí překrývají:

Instrukce \ takt123456789
Instrukce iIFIDEXMEMWB
Instrukce i+1IFIDEXMEMWB
Instrukce i+2IFIDEXMEMWB
Instrukce i+3IFIDEXMEMWB
Instrukce i+4IFIDEXMEMWB

Každá fáze je navíc paralelizovaná (Haswell zvládne až 4 instrukce za takt). Jádro má 8 výpočetních jednotek (4× celočíselná ALU, 2× FPU násobička, 2× čtení paměti …) a μops (mikrooperace) s různou latencí běží v každém taktu paralelně.

  • Nezávislé instrukce lze provést mimo pořadí nebo paralelně – pokud nepoužívají stejný registr nebo paměťovou adresu.
  • Čtení z paměti se nepřehazují mezi sebou (paralelní nezávislá čtení jsou ale povolena).
  • Pozdější nezávislé čtení smí předběhnout zápis – serializují se jen zápisy (stores zůstávají v pořadí, loads je smí předbíhat). Právě toto umožňuje výsledek C=1, D=2.

CPU vs. jádro vs. vlákno – paměťová hierarchie

Procesor obsahuje více jader; každé jádro má privátní L1/L2 cache a běží na něm HW vlákna, všechna jádra sdílejí L3 cache a hlavní paměť.

Všechny zápisy do hlavní paměti běží v režimu write-back: standardní zápis nejprve vyžaduje data v cache (drahý cache miss, pokud chybí); non-temporal zápisy (vhodné pro velké bloky) jdou přímo do paměti bez cachování; k dispozici jsou i prefetch instrukce.

Volatile a paměťová bariéra

Pokud označíme A a B jako volatile:

public volatile int A = 0;
public volatile int B = 0;
public int C = 0;
public int D = 0;

public void method1() {
    int r2 = A;
    B = 1;
    D = r2;
}

vznikne tento strojový kód:

0x000000010710e08c: mov    0xc(%rsi),%r11d
0x000000010710e090: movl   $0x1,0x10(%rsi)
0x000000010710e097: lock addl $0x0,(%rsp)

💬 Jak se volatile projeví v nativním kódu

Paměťové operace okolo zápisu do volatile proměnné se v C1/C2 nepřehazují. Bariérou je instrukce lock addl $0x0,(%rsp) – přičte 0 k vrcholu zásobníku, takže nedělá žádnou užitečnou práci; podstatný je jen vedlejší účinek prefixu lock: zakáže přehazování instrukcí a zajistí, že všechny předchozí zápisy jsou viditelné ostatním procesorům. Je to nejrychlejší zápisová paměťová bariéra. Čtecí bariéra není potřeba – čtení se při běhu CPU nepřehazují.

K jakým operacím je volatile užitečné

  • Nikdy se necachuje lokálně ve vlákně – všechny přístupy jdou přímo do hlavní paměti.
  • Zaručuje atomické čtení a zápis jednoho pole (definuje zápisovou paměťovou bariéru).
  • Lze použít pro primitivy i reference na objekty.
  • Neblokuje běh vlákna.

⚠️ Cena a omezení

  • Volatile zápisy jsou kvůli vyprazdňování cache výrazně pomalejší (~100×).
  • Volatile čtení (když probíhají zápisy) jsou také pomalejší (~25×, podle počtu jader) – kvůli invalidaci cache.
  • Stále je to rychlejší než zámky / synchronizace.

Volatile není potřeba pro: neměnné (immutable) objekty; proměnnou, k níž přistupuje jen jedno vlákno (context switch už cache vyprázdní); proměnnou uvnitř složené synchronizované operace.

Volatile není pro čtení–úprava–zápis

public class VolatileCounter {
    private volatile int cnt = 0;

    public int get() {
        return cnt;
    }

    public void increment() {
        cnt++;
    }
}

Bude to fungovat ve vícevláknovém prostředí? NE. increment se přeloží na tři samostatné instrukce (RSI je this):

0x000000010911544c: mov    0xc(%rsi),%edi
0x000000010911544f: inc    %edi
0x0000000109115451: mov    %edi,0xc(%rsi)
0x0000000109115457: lock addl $0x0,(%rsp)

Čtení–inkrement–zápis jsou tři instrukce, takže dvě vlákna se mohou proložit a ztratit aktualizaci. Bariéra lock sice zviditelní zápis, ale neudělá z celého cnt++ atomickou operaci.

💡 Kdy je volatile naopak nutné

Užitečné pro zápis jediným vláknem (např. terminační příznak). Pokud příznak nastavuje jiné vlákno, musí být volatile – jinak by C2 kompilátor vytáhl čtení z cyklu ven (hoisting) a vznikla by nekonečná smyčka, která příznak nikdy znovu neotestuje.

Volatile pole (arrays)

public class VolatileIntArray {
    private volatile int[] array;

    public VolatileIntArray(int capacity) {
        array = new int[capacity];
    }

    public int get(int index) {
        return array[index];
    }

    public void put(int index, int value) {
        array[index] = value;
    }
}

Je zápis do prvku pole obsloužen jako volatile? NE – volatile je jen reference na pole, nikoli jednotlivé prvky. V assembleru put je sice kontrola meze (cmp/jae → ArrayIndexOutOfBoundsException), ale žádná lock bariéra kolem zápisu:

# this:  rsi:rsi = 'datarace/VolatileIntArray'
# parm0: rdx     = int
# parm1: rcx     = int
0x000000011170bcc: mov     0xc(%rsi),%esi
0x000000011170bcf: shl     $0x3,%rsi              ;*getfield array
0x000000011170bd3: movslq  %edx,%rdi
0x000000011170bd6: cmp     0xc(%rsi),%edx         ; implicit exception -> 0x...bbf9

Řešení: po zápisu prvku znovu přiřadit (stejnou) referenci pole, aby volatile zápis reference vygeneroval bariéru:

private volatile int[] array;

public void put(int index, int value) {
    array[index] = value;
    array = array;          // dodatečné (zbytečné) přiřazení reference pole
}
0x000000010db21a67: mov    %r8d,0xc(%rsi)      ;*putfield array
0x000000010db21a6f: lock addl $0x0,(%rsp)      ; - VolatileIntArray::put@12 (line 16)

Prefix lock zakáže veškeré přehazování kolem sebe a zviditelní předchozí zápisy ostatním CPU. Stále to ale není vhodné pro čtení–úprava–zápis (aktualizace prvku sama o sobě není atomická).

Při použití zámků nejsou s read-modify-write problémy. Klíčové slovo synchronized zajišťuje vzájemné vyloučení:

    }
}
  • metoda vs. blok
  • instance objektu vs. instance třídy (u statických metod se zamyká instance třídy).

Implementace zámku v JVM – mark word

Stav zámku objektu je uložen v jeho mark word (64bit JVM). Layout podle stavu zámku:

Obsah (high → low)Tag bityStav
Reference na vytlačenou (displaced) hlavičku00Thin locked (tenký zámek)
NULL (0)00Inflating (nafukování)
Unused · Hashcode · Age · 001Unlocked (odemčeno)
Reference na Java vlákno · epoch · Age · 101Biased / biasable (odebráno od Java 15)
Reference na monitor10Fat locked (tlustý zámek)
(rezervováno)11Rezervováno pro GC

🔑 Druhy zámků v JVM

Thin lock (tenký zámek)
Nekonkurovaný zámek. Mark word se mění přes CAS instrukce. Lock record na zásobníku obsahuje 8B vytlačení původní mark word (rekurzivní zámek má 0) a 4B/8B komprimovaný OOP na zamčený objekt. Před přechodem na fat lock se používá spin-locking (10 taktů s volatile čtením + NOPy).
Fat lock (tlustý zámek)
Používá monitor objekt na haldě (nafouknutí ho vytvoří, vyfouknutí zničí). Spustí ho konkurovaný zámek nebo volání wait()/notify(). Monitor drží původní mark word, OS zámek, podmínky a množinu vláken; podporuje parking (uspání vlákna).
Biased lock (předpojatý zámek)
Optimalizace pro případ, kdy objekt zamyká stále totéž vlákno – zámek se „nakloní“ (bias) k tomu vláknu a další lock/unlock týmž vláknem je zdarma. Odebráno od Java 15. Assembler byl optimalizován pro thin locking.
Reentrant lock (znovuvstupitelný zámek)
Vlákno, které zámek už drží, ho může získat znovu (rekurzivně) bez deadlocku; viz ReentrantLock níže a rekurzivní lock record s hodnotou 0 v mark word.

Stavový automat biased lockingu (rovněž odebrán od Java 15): objekt začne jako biasable; první zamčení vláknem T ho udělá biased k T s aktuální epochou (další lock/unlock vláknem T je zdarma). Hromadný rebias (bulk rebias) ho udělá re-biasable (bias smí převzít nové vlákno T'); hromadné odebrání (bulk revoke) ho posune do prostého unlocked. Z unlocked se nekonkurovaným zámkem stane thin lock; konkurence nebo wait/notify ho nafoukne na fat (inflated) monitor a vyfouknutí (deflate) ho vrátí zpět.

ReentrantLock

public class ReentrantCounter {
    private int cnt = 0;
    private ReentrantLock lock = new ReentrantLock();

    public int get() {
        return cnt;
    }

    public void increment() {
        lock.lock();
        try {
            cnt++;
        } finally {
            lock.unlock();   // ! vždy uvolnit ve finally
        }
    }
}

⚠️ unlock() vždy ve finally

unlock() musí být v bloku finally, aby se zámek uvolnil i při výjimce.

Proti synchronized nabízí rozšířené operace:

  • lock(), unlock()
  • lockInterruptibly() – vyhodí InterruptedException
  • boolean tryLock()
  • boolean tryLock(long timeout, TimeUnit unit) – vyhodí InterruptedException

Férovost (fairness): blokovaná vlákna lze uspořádat pro férové zamykání. new ReentrantLock(boolean fair) – ve výchozím stavu neférový; synchronized je rovněž neférový. Neférový ReentrantLock je o něco rychlejší než synchronized, žije ale jako další instance na haldě. Férové zámky jsou pomalejší (~100×).

Atomické operace a CAS

Nejrychlejší čítač pro vícevláknové použití je založen na AtomicInteger:

public class AtomicCounter {
    private AtomicInteger cnt = new AtomicInteger(/* initialValue: */ 0);

    public int get() {
        return cnt.get();
    }

    public void increment() {
        cnt.incrementAndGet();
    }
}

Vnitřní implementace používá neblokující vzor = CAS smyčku:

Co je CAS (compare-and-swap)

CAS atomicky porovná hodnotu v paměti s očekávanou; pokud souhlasí, zapíše novou hodnotu a vrátí úspěch, jinak nezmění nic a vrátí neúspěch. C2 kompilátor celou CAS smyčku z increment zkolabuje do jediné instrukce lock addl (RSI je this, R12 = 0):

0x000000010b108d4c: mov    0xc(%rsi),%r11d   ;*getfield cnt
0x000000010b108d53: test   %r11d,%r11d       ; kontrola null pointeru
0x000000010b108d5d: je     0x000000010b108d68
0x000000010b108d55: lock addl $0x1,0xc(%r12,%r11,8)  ;*invokevirtual getAndAddInt
                                             ; - AtomicInteger::incrementAndGet (line 186)
  • Smyčka while je nahrazena jedinou instrukcí (lock addl).
  • Prefix lock zakáže veškeré přehazování a zviditelní předchozí zápisy ostatním CPU.
  • Prefix lock zajistí jádru výhradní vlastnictví příslušného cache řádku po dobu operace. Cache koherence je zajištěna protokolem MESIF (Haswell) se záložním zámkem paměťové sběrnice.
  • Čítač přes AtomicInteger je nejrychlejší ze všech pro vícevláknové použití.

Hardwarový CAS – atomické instrukce

  • 32bit CPU podporují 64bitové CAS: cmpxchg src, dst – s implicitním prefixem lock.
  • 64bit CPU podporují 128bitové CAS: cmpxchg16b – pracuje s páry registrů RDX:RAX a RCX:RBX.
  • Java v balíku java.util.concurrent.atomic používá jen 64bitové CAS: AtomicBoolean, AtomicInteger, AtomicLong, AtomicReference, AtomicIntegerArray, AtomicLongArray, AtomicReferenceArray.

Atomic field updatery a složené typy

Field updatery jsou vhodné pro velký počet objektů daného typu – šetří paměť, protože každá instance nemusí mít vnořený extra (atomický) objekt. Odkazují přímo na volatile proměnnou bez getterů/setterů, ale jejich operace jsou méně efektivní.

Před (vnořený AtomicInteger v každém objektu) → po (jeden sdílený statický updater, pole je jen volatile int):

public class ObjectWithAtomic {
    private static AtomicIntegerFieldUpdater<ObjectWithAtomic>
        valueUpdater = AtomicIntegerFieldUpdater.newUpdater(ObjectWithAtomic.class, "value");
    private volatile int value = 0;

    public void method1() {
        if (valueUpdater.compareAndSet(this, 1, 2)) {
            // ...
        }
    }
}

Existují AtomicIntegerFieldUpdater, AtomicLongFieldUpdater a AtomicReferenceFieldUpdater. Žádné field updatery pro pole (arrays) neexistují.

Atomické složené typy a ABA problém

  • AtomicMarkableReference – reference na objekt s mark bitem.
  • AtomicStampedReference – reference na objekt s celočíselným „razítkem“ (stamp).

💡 ABA problém

Hodnota se změní A → B a pak zpět B → A. Prosté CAS na hodnotě nepozná, že se mezitím něco stalo – v obou případech vidí A a chybně uspěje. Mark/stamp oba stavy odliší (razítko se mezitím zvýší), takže ABA odhalí. Tyto typy nepoužívají dvojnásobně široký CAS (CAS2/CASX) – jsou proto výrazně pomalejší kvůli alokaci objektu.

Protože reference + stamp/mark je větší než 64 bitů a Java používá jen 64bitové CAS, zabalí se obě hodnoty do neměnného objektu Pair a CAS se provede na (volatile) referenci na pár:

public class AtomicStampedReference<V> {

    private static class Pair<T> {
        final T reference;
        final int stamp;
        private Pair(T reference, int stamp) {
            this.reference = reference;
            this.stamp = stamp;
        }
        static <T> Pair<T> of(T reference, int stamp) {
            return new Pair<T>(reference, stamp);
        }
    }

    private volatile Pair<V> pair;

    public boolean compareAndSet(V expectedReference,
                                 V newReference,
                                 int expectedStamp,
                                 int newStamp) {
        Pair<V> current = pair;
        return
            expectedReference == current.reference &&
            expectedStamp == current.stamp &&
            ((newReference == current.reference &&
              newStamp == current.stamp) ||
             casPair(current, Pair.of(newReference, newStamp)));
    }
}

Neblokující (lock-free) algoritmy jsou založeny na CAS / CMPXCHG a instrukcích s prefixem LOCK. Jsou lock-free, ale typicky ne wait-free (kvůli neomezeným retry smyčkám).

Sdílené zdroje chráněné zámky mají nevýhody:

  • Vlákno s vysokou prioritou může být zablokováno (např. obsluha přerušení).
  • Hrubozrnné zamykání snižuje paralelismus (neférové zámky).
  • Jemnozrnné a férové zamykání zvyšuje režii.
  • Hrozí deadlock a inverze priorit (vlákno s nízkou prioritou drží zdroj, který potřebuje vlákno s vysokou prioritou).

Vlastnosti neblokujících algoritmů:

  • Předčí blokující algoritmy, protože většina CAS / CMPXCHG uspěje napoprvé.
  • Odstraňují cenu synchronizace, uspávání vláken a přepínání kontextu.

🔑 Lock-free vs. wait-free

Non-blocking (lock-free)
Zaručuje, že systém jako celek vždy pokročí – vždy alespoň jedno vlákno dokončí operaci v konečném počtu kroků. Jednotlivé vlákno ale může teoreticky opakovat svou retry smyčku neomezeně dlouho (jeho CAS stále selhává, protože je jiná vlákna předbíhají). Neuvázne (žádné deadlocky), ale negarantuje strop na počet kroků pro konkrétní vlákno.
Wait-free
Silnější záruka: každé vlákno dokončí svou operaci v konečném (předem omezeném) počtu kroků bez ohledu na chování ostatních. Žádné neomezené opakování. Real-time systémy vyžadují wait-free algoritmy – lock-free nestačí.

Neblokující zásobník (Treiberův algoritmus, 1986)

Zásobník je jednosměrně zřetězený seznam s atomickou referencí head; push i pop jsou CAS smyčky, které se opakují, pokud se head mezitím změnila:

static class Node<E> {
    final E item;
    Node<E> next;

    public Node(E item) { this.item = item; }
}

AtomicReference<Node<E>> head = new AtomicReference<Node<E>>();

public void push(E item) {
    Node<E> newHead = new Node<E>(item);
    Node<E> oldHead;
    do {
        oldHead = head.get();
        newHead.next = oldHead;
    } while (!head.compareAndSet(oldHead, newHead));
}

public E pop() {
    Node<E> oldHead;
    Node<E> newHead;
    do {
        oldHead = head.get();
        if (oldHead == null)
            return null;
        newHead = oldHead.next;
    } while (!head.compareAndSet(oldHead, newHead));
    return oldHead.item;
}

Push (vlož nový vrchol C před A → B) a pop (odeber vrchol A, zůstane B):

1head ► A ► B ► (konec)výchozí stav zásobníku
2C (připravený uzel)C.next = oldHead = head.get() = A
head.compareAndSet(A, C)
3head ► C ► A ► B ► (konec)po úspěšném CAS
PUSH — vlož nový vrchol C před A
1head ► A ► B ► (konec)výchozí stav zásobníku
head.compareAndSet(A, B)
2head ► B ► (konec)po úspěšném CAS, odebrán vrchol A
POP — odeber vrchol A, zůstane B

⚠️ ABA problém v zásobníku

Push po pop může způsobit ABA problém, pokud se znovu použije adresa! Adresa uvolněného uzlu může být realokována, takže head se může rovnat staré referenci, i když se zásobník mezitím změnil – CAS pak chybně uspěje. Řešením je AtomicStampedReference / AtomicMarkableReference.

Neblokující kolekce a mapy

Blokující kolekce a mapy jsou synchronizované wrappery:

static <T>   Collection<T> Collections.synchronizedCollection(Collection<T> c)
static <T>   List<T>       Collections.synchronizedList(List<T> list)
static <K,V> Map<K,V>      Collections.synchronizedMap(Map<K,V> m)
static <T>   Set<T>        Collections.synchronizedSet(Set<T> s)
// také pro SortedSet a SortedMap

Neblokující kolekce a mapy:

  • ConcurrentLinkedQueue (Collection, Queue): peek(), poll(), offer(E) v FIFO pořadí.
  • ConcurrentLinkedDeque (Collection, Deque): offer/poll/peek na obou koncích.
  • ConcurrentHashMap (Map): putIfAbsent(K,V), remove(Object,Object), replace(K, oldV, newV).
  • ConcurrentSkipListMap (SortedMap), ConcurrentSkipListSet (SortedSet).

💬 Pozor na jedno vlákno

Neblokující kolekce a mapy jsou pomalejší pro jednovláknový přístup – kvůli použití CAS instrukcí.

ConcurrentHashMap – struktura a konkurence

  • Souběžná čtení – get, iterator (čtení bez zamykání).
  • Minimalizace konkurence při zápisu: výchozí úroveň konkurence 16 (počet zapisujících vláken). Vložení do prázdného segmentu používá CAS; pozdější změny jsou na zámcích segmentů.
  • Konkurence v segmentu: pro méně než 8 prvků se použije seznam, jakmile bin naroste, převede se na vyvážený strom (udržuje next pro iteraci).
  • Změna velikosti tabulky při překročení load factoru 0,75: rozšiřování o mocniny dvojky; každý záznam se přesune na stejný nebo o mocninu dvojky posunutý index; znovupoužití interního Node, pokud se next nemění (většina případů); kterékoli vlákno smí pomoci s přesunem místo blokování; forward nodes upozorní na přesunuté biny. Pokud znáš odhad, zadej initialCapacity (vyhne se opakovanému zvětšování).

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Co je data race v programu? K jakým operacím je užitečná proměnná volatile v JVM a jak se to projeví v nativním kódu? Jmenuj 2 atomické instrukce.

Data race = dvě či více vláken přistupují bez synchronizace ke stejné sdílené proměnné, přičemž alespoň jeden přístup je zápis; výsledek závisí na nepředvídatelném pořadí, které CPU i JIT smí přehazovat (klasický příklad C=1, D=2).

Volatile je užitečné pro viditelnost a atomické čtení/zápis jednoho pole, typicky terminační příznak nastavovaný jiným vláknem (bez něj by C2 vytáhl čtení z cyklu, takže by vznikla nekonečná smyčka). Není vhodné pro read-modify-write (cnt++ jsou 3 instrukce).

V nativním kódu se zápis do volatile projeví zápisovou paměťovou bariérou lock addl $0x0,(%rsp) – nedělá žádnou práci, podstatný je jen prefix lock, který zakáže přehazování a zviditelní předchozí zápisy ostatním CPU. Je to nejrychlejší zápisová bariéra; čtecí bariéra není potřeba.

Dvě atomické instrukce: cmpxchg (compare-and-swap, implicitní lock) a cmpxchg16b (128bit CAS); resp. lock addl / lock xadd.

zkoušková otázka Napiš neblokující zásobník (push a pop). Jaký je rozdíl mezi non-blocking a wait-free?

Treiberův zásobník – jednosměrně zřetězený seznam s AtomicReference<Node> head, push i pop jsou CAS smyčky:

public void push(E item) {
    Node<E> newHead = new Node<E>(item);
    Node<E> oldHead;
    do {
        oldHead = head.get();
        newHead.next = oldHead;
    } while (!head.compareAndSet(oldHead, newHead));
}

public E pop() {
    Node<E> oldHead, newHead;
    do {
        oldHead = head.get();
        if (oldHead == null) return null;
        newHead = oldHead.next;
    } while (!head.compareAndSet(oldHead, newHead));
    return oldHead.item;
}

Non-blocking (lock-free): systém jako celek vždy pokročí (vždy aspoň jedno vlákno dokončí operaci), ale konkrétní vlákno může opakovat retry smyčku neomezeně. Wait-free: každé vlákno dokončí operaci v konečném, předem omezeném počtu kroků bez ohledu na ostatní. Treiberův zásobník je lock-free, ne wait-free. Real-time systémy potřebují wait-free.

P8 Vysvětli thin lock, fat lock a biased lock. Kde je stav zámku uložen?

Stav zámku je v mark word objektu. Thin lock = nekonkurovaný zámek, mark word se mění přes CAS, lock record na zásobníku, před fallbackem krátký spin (10 taktů). Fat lock = monitor objekt na haldě (OS zámek, podmínky, parking), nastane při konkurenci nebo wait()/notify() (nafouknutí/vyfouknutí). Biased lock = optimalizace, kdy se zámek „nakloní“ k jednomu vláknu a další lock/unlock týmž vláknem je zdarma – odebráno od Java 15.

P8 Co je CAS a co je ABA problém? Jak se ABA řeší?

CAS (compare-and-swap) atomicky porovná hodnotu v paměti s očekávanou a jen při shodě zapíše novou hodnotu (HW instrukce cmpxchg s prefixem lock). Je základem atomik a lock-free algoritmů (CAS smyčka).

ABA problém: hodnota se změní A → B → A; prosté CAS vidí v obou okamžicích A a chybně uspěje, i když se mezitím něco stalo (např. v zásobníku byla realokována adresa uvolněného uzlu). Řešení: AtomicStampedReference (razítko) nebo AtomicMarkableReference (mark bit) – razítko/mark se mezitím zvýší a odliší stavy. Tyto typy alokují objekt Pair (nepoužívají dvojširoký CAS), a jsou proto pomalejší.

P8 Proč nestačí volatile na cnt++ a co použít místo toho?

cnt++ je čtení–inkrement–zápis = tři instrukce; dvě vlákna se mohou proložit a ztratit aktualizaci. Volatile bariéra zápis zviditelní, ale celou operaci neudělá atomickou. Řešením je synchronized/ReentrantLock, nebo nejlépe (a nejrychleji) AtomicInteger.incrementAndGet(), který C2 zkolabuje do jediného lock addl.

📌 Co si zapamatovat

  • Data race = nesynchronizovaný souběžný přístup (aspoň jeden zápis); CPU i C2 JIT přehazují operace → výsledek C=1, D=2. Zápisy se serializují, pozdější čtení smí předběhnout zápis.
  • volatile dá viditelnost + atomické čtení/zápis jednoho pole přes bariéru lock addl $0x0,(%rsp) (nejlevnější bariéra); není pro read-modify-write a u polí je volatile jen reference, ne prvky. Volatile zápis ~100×, čtení ~25× dražší, stále levnější než zámky.
  • JVM zámky v mark word: nekonkurovaný → thin lock (CAS + krátký spin), konkurovaný / wait/notify → fat lock (monitor na haldě, parking). Biased lock odebrán od Java 15.
  • ReentrantLock přidá tryLock, časované/přerušitelné zamykání a férovost; neférový je o málo rychlejší než synchronized (ale objekt na haldě), férový ~100× pomalejší.
  • Atomiky stojí na HW CAS (cmpxchg, implicitní lock; cmpxchg16b pro 128 bit); Java používá jen 64bit CAS. AtomicInteger čítač je nejrychlejší.
  • AtomicStampedReference/MarkableReference řeší ABA problém, ale alokují Pair → pomalé. Field updatery šetří paměť, ale jsou pomalejší; pro pole neexistují.
  • Lock-free = systém pokročí, ale vlákno může opakovat neomezeně; wait-free = každé vlákno v konečném počtu kroků (real-time potřebuje wait-free).
  • Treiberův zásobník (1986) – lock-free LIFO (CAS na head), pozor na ABA při realokaci adres. ConcurrentHashMap: čtení bez zámků, CAS na prázdný bin pak zámky segmentů (konkurence 16), seznam <8 jinak strom, resize při load factoru 0,75 po mocninách dvojky.
