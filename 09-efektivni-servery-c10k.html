Efektivní servery, problém C10k a neblokující I/O

Jak postavit server, který zvládne desítky tisíc souběžných spojení. Klasický problém C10k: model „vlákno na spojení“ (styl Apache, blokující I/O) proti událostmi řízenému modelu (styl Nginx, neblokující I/O). Projdeme systémové primitivy pro hlášení připravenosti (select, poll, epoll), blokující sítě v Javě (Socket, ServerSocket), neblokující Java NIO (kanály, buffery, selektory) i moderní virtuální vlákna (Project Loom, Java 21), která vracejí jednoduchý návrh „vlákno na spojení“ s téměř optimální škálovatelností pro I/O zátěž.

💬 Klíč k zapouzdření

Síťové (IP) adresy zůstávají konstantní po celé cestě end-to-end, kdežto linkové (Ethernet/MAC) adresy se přepisují na každém směrovači (hop-by-hop). Každá vrstva si přidává a odebírá jen svou vlastní hlavičku.

Problém C10k (a C10M)

C10k je problém z konce 90. let: jak na jediném serveru obsloužit velký počet klientů (10 000+) najednou. Jde o velký počet souběžných spojení, jejichž obsluhu je třeba efektivně plánovat. Pozor: nejde o počet požadavků za sekundu, ale o počet současně otevřených spojení.

  • Dnes se mluví o C1M nebo C10M (miliony spojení).
  • Reálný příklad: WhatsApp v roce 2014 zvládal přes 2 miliony souběžných TCP spojení na jeden server (Erlang/BEAM + ladění FreeBSD).

Existují dva základní přístupy ke zpracování spojení:

Škálováníneškáluje – vysoká cena přepínání kontextuvícejádrová škálovatelnost s řízeným počtem worker vláken

Procesy vs. vlákna

Procesy i vlákna umožňují souběžné vykonávání. Proces má jedno nebo více vláken.

  • Vlákna jednoho procesu sdílejí stejný adresní prostor (data i kód).
  • Přepínání kontextu mezi vlákny je obvykle levnější než mezi procesy.
  • Mezivláknová komunikace je relativně efektivní přes sdílenou paměť.
  • Každé vlákno má vlastní zásobník (stack) s rámci (t.getStackTrace()), vlastní lokální proměnné a parametry metod.

V JVM lze vlákno vytvořit jako podtřídu java.lang.Thread nebo implementací rozhraní java.lang.Runnable.

Model „vlákno na spojení“ a jeho limity

Naivní řešení obsluhy mnoha spojení: vytvořit pro každé spojení (např. TCP) jedno vlákno a uvnitř používat blokující operace. Kód je jednoduchý a sekvenční, ale škálovatelnost je omezená:

  • Vytvoření a ukončení vlákna není levná operace.
  • Každé vlákno potřebuje nezanedbatelně paměti (minimálně na svůj zásobník).
  • Plánovač OS spravuje jen platformová vlákna a přepínání kontextu má vysokou cenu.

Proto obsloužit miliony spojení tímto způsobem prakticky nelze. Jádrem výkonných síťových aplikací je naopak smyčka událostí (event loop) zpracovávající události z mnoha zdrojů; typicky běží jedno vlákno na každém jádru CPU. Tento návrh používá Nginx, Node.js (knihovna libuv) i běhová prostředí jazyků se stylem async/await.

Blokující vs. neblokující I/O

Blokující I/O: vlákno zavolá operaci (např. read) a uspí se, dokud nejsou data připravena. Jednoduché, ale jedno vlákno = jedno spojení. Neblokující I/O: operace se vrátí ihned, případně bez přečtení dat; vlákno se mezitím může věnovat jiným spojením. Jak se ale dozvíme, že je I/O připravené? Existuje několik mechanismů, každý se svým háčkem:

Událostmi řízený model: select, poll, epoll

Tato systémová volání hlásí, které deskriptory jsou připravené k I/O. Postupně rostou ve škálovatelnosti.

select()

  • definováno v POSIX (Portable Operating System Interface);
  • původně určeno pro blokující I/O;
  • předané seznamy deskriptorů nelze znovu použít v dalších voláních – systémové volání je totiž modifikuje;
  • neškáluje – omezený počet deskriptorů a navíc se přes ně musí iterovat, aby se našly události.
int
select(int nfds, fd_set *restrict readfds, fd_set *restrict writefds, fd_set *restrict errorfds,
       struct timeval *restrict timeout);

void  FD_CLR(int fd, fd_set *fdset);
void  FD_COPY(fd_set *fdset_orig, fd_set *fdset_copy);
int   FD_ISSET(int fd, fd_set *fdset);
void  FD_SET(int fd, fd_set *fdset);
void  FD_ZERO(fd_set *fdset);

poll()

  • počet sledovaných deskriptorů není omezen;
  • deskriptory lze znovu použít;
  • lepší, ale stále se přes deskriptory musí iterovat, aby se našly události.
int
poll(struct pollfd fds[], nfds_t nfds, int timeout);

struct pollfd {
    int    fd;       /* file descriptor */
    short  events;   /* events to look for */
    short  revents;  /* events returned */
};

epoll

  • jen Linux (např. Windows má IOCP – I/O Completion Ports; BSD/macOS má kqueue);
  • škáluje;
  • sledované události lze měnit i během čekání (přes systémové volání);
  • vrací jen ty události, které nastaly (není třeba iterovat přes vše).

API: epoll_create / epoll_create1 (inicializace instance v jádře), epoll_ctl (přidá/změní/odebere deskriptory), epoll_wait (čeká na události s timeoutem).

Režimy:

  • level triggered – wait se vrátí ihned, pokud je událost dostupná;
  • edge triggered (EPOLLET) – připravenost se hlásí jen při příchozí události; musíte zpracovat všechny čekající události před dalším čekáním.

Události: EPOLLIN, EPOLLOUT, EPOLLPRI, EPOLLRDHUP, EPOLLHUP, EPOLLERR.

typedef union epoll_data
{
  void        *ptr;
  int          fd;
  __uint32_t   u32;
  __uint64_t   u64;
} epoll_data_t;

struct epoll_event
{
    __uint32_t   events; /* Epoll events */
    epoll_data_t data;   /* User data variable */
};

Inicializace a registrace deskriptoru:

int epfd = epoll_create1(0);
...
struct epoll_event ev;
int client_sock;
...
ev.events = EPOLLIN | EPOLLPRI | EPOLLERR | EPOLLHUP;
ev.data.fd = client_sock;
int res = epoll_ctl(epfd, EPOLL_CTL_ADD, client_sock, &ev);

Typická smyčka událostí:

while (1) {
    // wait for something to do...
    int nfds = epoll_wait(epfd, events,
                          MAX_EPOLL_EVENTS_PER_RUN,
                          EPOLL_RUN_TIMEOUT);
    if (nfds < 0) die("Error in epoll_wait!");

    // for each ready socket
    for(int i = 0; i < nfds; i++) {
      int fd = events[i].data.fd;
      handle_io_on_socket(fd);
    }
}

💡 Vývoj škálovatelnosti

select() (POSIX, omezený počet, seznamy ničené/měněné při každém volání) → poll() (bez limitu, znovupoužitelné, stále O(n) průchod) → epoll (jen Linux, škáluje, vrací jen vzniklé události, level vs. edge EPOLLET). Ekvivalent na Windows je IOCP, na BSD/macOS kqueue.

Blokující sítě v Javě

TCP klient – Socket

  • Socket – klientský konec TCP/IP spojení; je svázán s konkrétní cílovou IP a portem;
  • každé spojení je jednoznačně určeno svými dvěma konci;
  • poskytuje vstupní a výstupní proudy (streams).
try (
    Socket echoSocket = new Socket("localhost", 7);
    PrintWriter out = new PrintWriter(echoSocket.getOutputStream(), true); // autoFlush = true
    BufferedReader in = new BufferedReader(new InputStreamReader(echoSocket.getInputStream()));
    BufferedReader stdIn = new BufferedReader(new InputStreamReader(System.in))
) {

    String userInput;

    while ((userInput = stdIn.readLine()) != null) {
        out.println(userInput);
        System.out.println("echo: " + in.readLine());
    }
}

TCP server – ServerSocket

  • ServerSocket – serverový socket reprezentující naslouchající TCP/IP koncový bod;
  • v konstruktoru zadáte port, volitelně i IP, na kterou se má navázat;
  • na navázání spojení čeká metoda Socket accept().

Server „vlákno na spojení“ – každý handler běží ve vlastním vlákně s blokujícím I/O:

ExecutorService clientRunner = Executors.newCachedThreadPool();
try (
    ServerSocket serverSocket = new ServerSocket(7);
) {
    while (true) {
        final Socket s = serverSocket.accept();
        clientRunner.execute(() -> {
            try (
                BufferedReader in = new BufferedReader(new InputStreamReader(s.getInputStream()));
                PrintWriter out = new PrintWriter(s.getOutputStream(), true); // autoFlush = true
            ) {
                String line;
                while (s.isConnected()) {
                    if ((line = in.readLine()) != null) {
                        out.println(line);
                    }
                }
            } catch (IOException e) {
                e.printStackTrace();
            }
        });
    }
} catch (Exception e) {
    e.printStackTrace();
} finally {
    clientRunner.shutdownNow();
}

UDP – DatagramPacket, DatagramSocket, MulticastSocket

  • DatagramPacket – samostatná, soběstačná zpráva (jako síťový paket): cílová InetAddress address a int port, data byte data[], int length, int offset, odesílatel SocketAddress sa;
  • DatagramSocket – koncový bod pro odesílání/příjem paketů; metody send(DatagramPacket p) a receive(DatagramPacket p) jsou blokující; connect(InetAddress, int) omezí komunikaci na daného hosta;
  • MulticastSocket – navíc připojování/odpojování od multicastových skupin (IGMP, rozsah IP 224.0.0.0 – 239.255.255.255).

Java NIO – kanály, buffery, selektory

Java NIO je neblokující, blokově orientované I/O. Umožňuje škálovatelné asynchronní I/O a polling, vysokorychlostní binární i znakové I/O včetně mapování souborů do paměti. Základní abstrakcí je kanál (channel) pracující s buffery: kanál čte data do bufferu a zapisuje data z bufferu.

Buffer – vlastnosti

  • java.nio.Buffer – lineární konečná posloupnost prvků jednoho primitivního typu (ByteBuffer, CharBuffer, IntBuffer, LongBuffer, MappedByteBuffer z FileChannel.map(…) …);
  • není thread-safe; jeden buffer má více režimů (čtení i zápis);
  • invariant: 0 <= mark <= position <= limit <= capacity:
    • capacity – počet prvků, nikdy se nemění;
    • limit – index prvního prvku, který se nemá číst/zapisovat;
    • position – index dalšího prvku ke čtení/zápisu;
    • mark – index, na který se position nastaví po reset().

Klíčové operace:

Typické použití bufferu: (1) zapiš data do bufferu, (2) zavolej buffer.flip(), (3) přečti data z bufferu, (4) zavolej buffer.clear() nebo buffer.compact().

💬 compact()

compact() zkopíruje bajty mezi position a limit na začátek bufferu a připraví ho na další zápis (nedopsaná data se nezahodí).

Přímé (direct) buffery a nativní paměť

  • ByteBuffer.allocateDirect(…) – buffer je uložen mimo Java heap, v nativní paměti;
  • umožňuje nativnímu i Java kódu sdílet data bez kopírování – stejná paměť se předává jádru OS při voláních (užitečné pro soubory a sokety);
  • více bufferů může sdílet nativní paměť: slice() / duplicate() mají nezávislou position/limit/mark, ale sdílený obsah; asReadOnlyBuffer() dává pohled jen pro čtení;
  • ladění a sledování (JVM flagy): -XX:MaxDirectMemorySize=N (výchozí: bez limitu), -XX:NativeMemoryTracking=off|summary|detail, -XX:+PrintNMTStatistics.

💡 Proč přímé buffery

Použití heap bufferů znamená kopírování obsahu do a z Java heapu, protože GC může objekty přemisťovat. Přímé buffery v nativní paměti se předají jádru přímo, bez kopie. JNI a nativní NIO buffery žijí v „Other native memory“ (Non Heap), mimo zásobník i Java heap.

Channel a Selector

  • jedno vlákno obsluhuje více kanálů najednou – interně přes epoll, pokud ho OS podporuje;
  • kanály pokrývají síťové i souborové I/O: FileChannel, DatagramChannel, MulticastChannel, SocketChannel, ServerSocketChannel;
  • Channel podporuje současné čtení i zápis (proudy jsou jen jednosměrné) a vždy čte/zapisuje z/do bufferu.

Selector a SelectionKey

Do selektoru lze registrovat jen kanály v neblokujícím režimu:

Selector selector = Selector.open();
channel.configureBlocking(false);
SelectionKey key = channel.register(selector, SelectionKey.OP_READ);

FileChannel neblokující režim nepodporuje. Události k naslouchání (SelectionKey, lze kombinovat): OP_CONNECT, OP_ACCEPT, OP_READ, OP_WRITE.

  • SelectionKey (vrací register) nese interest set (vámi nastavené operace), ready set (které operace jsou připravené – sk.isReadable(), sk.isWritable() …), kanál, selektor a volitelně připojený objekt (sk.attach(Object) / sk.attachment()).
  • Výběrové metody selektoru: int select() blokuje, dokud není připraven alespoň jeden kanál; select(long timeout) s časovým limitem; selectNow() vůbec neblokuje; vracejí počet kanálů připravených od posledního volání.
  • selector.selectedKeys() vrátí množinu klíčů; wakeup() probudí vlákno blokované v select(); close() zruší selektor (kanály se nezavřou).

Vzor Reactor + ThreadPool: výpočetní krok process se odloží jako úloha do fronty a zpracuje ho worker vlákno z poolu, čímž se odděluje výpočetní práce od smyčky událostí.

Thread pooling

Thread pooling je princip, kdy se místo vytváření nového vlákna pro každou úlohu udržuje fond (pool) předvytvořených vláken, který úlohy zpracovává. Hodí se pro vykonání velkého počtu asynchronních úloh (např. obsluhu požadavků na serveru) a snižuje režii vytváření vláken a přepínání kontextu.

  • Rozhraní java.util.concurrent.ExecutorService: execute(Runnable r), Future<?> submit(Runnable r), Future<T> submit(Callable<T> c), shutdown(), shutdownNow(), awaitTermination.
  • Future<T>: cancel(boolean), isCancelled(), isDone(), V get(), V get(long timeout, TimeUnit unit).
  • Tovární metody Executors: newSingleThreadExecutor(), newFixedThreadPool(nThreads), newCachedThreadPool() (výchozí keep-alive 60 sekund).

Virtuální vlákna (Project Loom)

Virtuální vlákno je odlehčená implementace Java vlákna, dostupná od Javy 21 (preview od Javy 19 s --enable-preview). Vrací jednoduchý návrh „vlákno na spojení“ s téměř optimálním využitím hardwaru pro I/O zátěž.

Klasické (platformové) vlákno je tenký obal kolem vlákna spravovaného OS – základní jednotky plánování OS. Jeho vytvoření a zrušení jsou drahé a náročné na zdroje, má pevnou velikost zásobníku (hrozí StackOverflowError) a neškáluje. Alternativou býval styl async/await (reaktivní, callbacky), ten ale trpí špatně čitelnými stack trace, obtížným laděním a složitou sekvenční kompozicí.

Jak fungují (mounting)

  • umožňují thread-per-request s téměř optimálním využitím HW;
  • jsou kompatibilní s Thread API a podporují ladění/profilování stávajícími nástroji;
  • rámce zásobníku jsou v heapu a velikost zásobníku se dynamicky mění (roste i klesá);
  • OS spravuje stále jen platformová vlákna; virtuální vlákno se pro běh namountuje (mount) na nosné (carrier) vlákno – jeho rámce se zkopírují z heapu na zásobník nosného vlákna;
  • při blokování (I/O, zámek, jiný zdroj) se vlákno odmountuje (unmount); mountování/odmountování je z Java kódu neviditelné, thread dump neobsahuje rámce nosného vlákna;
  • nosná vlákna pocházejí z ForkJoinPool v režimu FIFO, o velikosti počtu logických jader CPU.

API a škálovatelnost

  • Thread::ofVirtual(); Executors.newVirtualThreadPerTaskExecutor() (každá úloha běží ve vlastním virtuálním vlákně);
  • Thread::currentThread() vrací virtuální vlákno (ne nosné); ThreadLocal, přerušení i procházení zásobníku fungují stejně;
  • virtuální vlákno je vždy daemon (setDaemon nemá efekt) a nelze měnit prioritu;
  • rychlé vytvoření, malá paměťová stopa; efektivita běhu je stejná jako u platformových vláken;
  • výborně škáluje pro I/O zátěž (i krátkodobé úlohy), hodí se pro serverové aplikace; pro CPU zátěž nepřináší žádnou výhodu.

Příklad spuštění 100 000 virtuálních vláken:

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 100_000).forEach(i -> {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        });
    });
}

Po zahřátí to trvá asi 1,1 sekundy; s Executors.newFixedThreadPool(1000) by to trvalo asi 1000 sekund.

⚠️ Nevýhody virtuálních vláken

  • synchronized připne (pin) virtuální vlákno k nosnému → používejte ReentrantLock;
  • vykonání JNI také připíná;
  • nosné vlákno se uvolní jen při blokujících operacích – není preempce!

💡 Výhoda vs. nevýhoda (zkoušková dvojice)

Výhoda oproti systémovým vláknům: levné vytvoření a malá paměťová stopa → škálují na statisíce souběžných I/O úloh při zachování jednoduchého kódu „vlákno na spojení“. Nevýhoda: žádný přínos pro CPU zátěž a riziko pinningu (synchronized, JNI) a chybějící preempce.

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Co je thread-pooling? Popiš virtuální vlákna. Jmenuj jednu výhodu a jednu nevýhodu oproti systémovým vláknům.

Thread-pooling = místo vytváření vlákna pro každou úlohu se udržuje fond předvytvořených vláken (ExecutorService, Executors.newFixedThreadPool / newCachedThreadPool / newSingleThreadExecutor), který zpracovává velký počet asynchronních úloh; snižuje režii vytváření vláken a přepínání kontextu.

Virtuální vlákna (Java 21) jsou odlehčená Java vlákna s rámci v heapu, jejichž zásobník se dynamicky mění. OS spravuje jen platformová (nosná) vlákna; virtuální vlákno se mountuje na nosné vlákno z FIFO ForkJoinPool a při blokování se odmountuje. Umožňují jednoduchý návrh „vlákno na spojení“.

Výhoda: levné vytvoření a malá paměťová stopa → škálují na statisíce souběžných I/O úloh (100k vláken ≈ 1,1 s vs. ≈ 1000 s u poolu 1000 vláken). Nevýhoda: žádný přínos pro CPU zátěž a riziko pinningu nosného vlákna (synchronized, JNI) plus chybějící preempce (carrier se uvolní jen při blokování).

zkoušková otázka C10k: popis problému, dva přístupy ke zpracování spojení, neblokující I/O v Javě a v C.

C10k = obsloužit na jednom serveru 10 000+ souběžných spojení najednou (dnes C1M/C10M); jde o efektivní plánování a počet spojení, ne o požadavky/s.

Dva přístupy: (1) vlákno na spojení (Apache) – každé spojení = vlastní vlákno s blokujícím I/O, plánování řeší OS, neškáluje kvůli ceně přepínání kontextu; (2) událostmi řízený I/O (Nginx) – malý počet worker vláken se smyčkou událostí a neblokujícím I/O, vícejádrová škálovatelnost.

Neblokující I/O v C: systémová volání select() (POSIX, limit, ničí seznamy), poll() (bez limitu, O(n)) a epoll (Linux, škáluje, vrací jen vzniklé události; epoll_create1, epoll_ctl, epoll_wait; level/edge EPOLLET). Na Windows IOCP, na BSD/macOS kqueue.

Neblokující I/O v Javě: Java NIO – kanály (SocketChannel, ServerSocketChannel) v neblokujícím režimu (configureBlocking(false)), registrované u Selectoru (select() hlásí připravené kanály přes SelectionKey), data se přenášejí přes buffery. Interně NIO používá epoll.

P9 Jaký je rozdíl mezi select, poll a epoll?

select() – POSIX, omezený počet deskriptorů, předané seznamy systémové volání modifikuje (nelze znovu použít), přes deskriptory se musí iterovat → neškáluje. poll() – počet deskriptorů bez limitu, deskriptory znovupoužitelné, ale stále se přes ně musí iterovat O(n). epoll – jen Linux, škáluje, sledované události lze měnit za běhu a vrací jen ty události, které nastaly (level vs. edge triggered EPOLLET). Ekvivalenty: Windows IOCP, BSD/macOS kqueue.

P9 Vysvětli rozdíl mezi blokujícím a neblokujícím I/O. Jak se v Javě realizuje neblokující čtení?

Blokující I/O: operace (např. read) uspí vlákno, dokud nejsou data připravena → jedno vlákno obsluhuje jedno spojení (model Apache, Socket/ServerSocket). Neblokující I/O: operace se vrátí ihned, případně bez přečtení dat; jedno vlákno tak obslouží mnoho spojení (model Nginx).

V Javě to zajišťuje NIO: kanál se přepne configureBlocking(false), zaregistruje u Selectoru na operaci (OP_READ …), selector.select() vrátí připravené kanály a podle SelectionKey (isReadable()) se z kanálu čte do bufferu. read(…) se může vrátit, aniž by cokoli přečetl.

📌 Co si zapamatovat

  • C10k = 10 000+ souběžných spojení na jednom serveru (dnes C1M/C10M); jde o plánování a počet spojení, ne o požadavky/s.
  • Dva přístupy: vlákno na spojení (Apache) – blokující I/O, neškáluje; událostmi řízený (Nginx) – malý pool worker vláken, neblokující event loop, škáluje.
  • Připravenost I/O: select() (limit, ničí seznamy) → poll() (bez limitu, O(n)) → epoll (Linux, škáluje, jen vzniklé události); Windows IOCP, BSD kqueue.
  • Java NIO = neblokující blokové I/O přes Buffer + Channel + Selector; buffer: 0 ≤ mark ≤ position ≤ limit ≤ capacity, clear()/flip().
  • Direct buffery v nativní paměti se sdílí s jádrem bez kopie (sokety/soubory); heap buffery se kopírují, protože GC přemisťuje.
  • Thread pooling amortizuje cenu vytváření vláken; virtuální vlákna (Java 21) vrací jednoduchý thread-per-request a škálují pro I/O (ne CPU), pozor na pinning (synchronized, JNI) a chybějící preempci.
