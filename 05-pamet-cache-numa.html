Přístup do paměti: cache, dynamická alokace a NUMA

Proč je hlavní paměť (DRAM) pomalá, jak ji cache procesoru zakrývá a jak psát kód, který cache co nejvíc využije. Projdeme vnitřek DRAM a její časování, organizaci cache (přímo mapovaná / asociativní), zápisové politiky, TLB, cache-friendly datové struktury a alokátory. Ukázkový příklad násobení matic ukáže, že pouhým přeskupením přístupů (transpozice, dlaždice, rekurze) lze dosáhnout více než 10× zrychlení při stejné složitosti O(N³). Na závěr vícejádrová témata: cache koherence (MESI(F)), true a false sharing, NUMA, připínání vláken a libnuma.

Proč je DRAM pomalá

SRAM vs DRAM

Existují dva základní typy paměti RAM, které se liší cenou, rychlostí i způsobem uložení bitu:

TypVlastnostiUložení bituPoužití
SRAM (statická) Rychlá, ale drahá. 6 tranzistorů na bit. Registry, cache, TLB…
DRAM (dynamická) Pomalá – (vy)nabíjení kondenzátoru není okamžité; čtení navíc ničí hodnotu (po čtení se musí obnovit zápisem). Kompromis kapacita / velikost / spotřeba. 1 tranzistor + kondenzátor na bit. Hlavní paměť.

SRAM buňka má 6 tranzistorů (M1–M6) řízených slovem WL, napájených z Vdd, s komplementárními bitovými linkami BL/B̄L. DRAM buňka je naproti tomu jediný tranzistor M (řízený linkou AL), který připojuje úložný kondenzátor C k datové lince DL. Nabití kondenzátoru na 100 % trvá zhruba 5·RC a vybití zpět na 0 dalších ~4·RC – proto (vy)nabíjení DRAM buňky není okamžité.

DRAM v počítači

  • Procesor obsahuje paměťový řadič (MC, memory controller).
  • Řadič komunikuje s DRAM čipy přes paměťovou sběrnici („Memory Bus“) protokolem s příkazy: ACTIVATE, READ / WRITE, PRECHARGE.

Na schématu platformy Intel P55 je CPU uprostřed: k DDR modulům ho připojuje Memory Bus, ke GPU (s vlastní GDDR pamětí) sběrnice PCIe x16 a k čipsetu PCH (Platform Controller Hub) sběrnice DMI. PCH dále připojuje disk přes SATA a rozšiřující karty.

Jak DRAM čip funguje

  • Adresovat každou buňku zvlášť je nepraktické (potřebovalo by to příliš mnoho vodičů).
  • Čip je uspořádaný do řádků a sloupců (a bank); adresa je multiplexovaná.
  • Multiplexery řádků a sloupců vybírají linky podle bitů adresy.
  • Čtení/zápis probíhá v mnoha čipech paralelně, aby se složilo celé datové slovo (64 bitů).

Uvnitř DRAM banky

Přístup do DRAM probíhá ve třech analogových fázích:

Cell Array Row 0 / Row 1 / Row 2 ★aktivní / Row 3 · bitové linky BL, B̄L
Sense Amps = „Row Buffer“ ACTIVATE · tRCD ≈ 13 ns · (≈ obvod SRAM)
Column Decoder / Mux READ · CL ≈ 10 ns
I/O Buffer → piny datová slova ven
Tři analogové fáze přístupu do DRAM banky
  • ACTIVATE: otevře řádek → snímací zesilovače (sense amps) detekují a zachytí data. Sense amps jsou ten „row buffer“ (ne samostatná SRAM).
  • READ: sloupcový multiplexer vybere, které bity půjdou na I/O piny.
  • PRECHARGE: obnoví buňky a vynuluje bitové linky pro další řádek.
  • Každý krok trvá ≈ 10–13 ns (čas analogového ustálení).
  • Toto se za 20 let téměř nezlepšilo – roste pouze propustnost (bandwidth).

Klíčové latence: tRCD ≈ 13 ns (ACTIVATE → připraveno) a CL ≈ 10 ns (CAS latency, READ).

Komunikační protokol SDRAM

  • Přístupový protokol je synchronní – existuje hodinový signál → SDRAM (Synchronous DRAM).
  • CLK dodává paměťový řadič (frekvence FSB – typicky 800–1600 MHz).
  • Double/Quad-pumped – data se přenášejí na obou hranách hodin / ve více podfázích.
  • Max. rychlost: 64 bit × 8 × 200 MHz = 12,8 GB/s (v reálu nedosažitelná).
  • Technologie DRAM vyžaduje zpoždění tRCD a CL – nelze je zkrátit.
  • Data se posílají v dávkách (bursts) o velikosti cache line – poslat jediné slovo by bylo velmi neefektivní kvůli zpožděním tRCD a CL.

Časový diagram má signály CLK, R̄ĀS̄, C̄ĀS̄, Address a DQ (data). Adresa nejprve nese řádek/banku (Activate), po mezeře tRCD sloupec; po mezeře CL pak DQ přenese souvislou dávku datových slov.

Časovací parametry DDR4

U každého standardu je memory clock skutečné hodiny DRAM buněk, I/O bus clock je 4× vyšší a data rate (MT/s) je 8× memory clock (double data rate přes quad-pumped sběrnici). Lokalita přístupů

Cache funguje, protože reálné programy přistupují do paměti se dvěma druhy lokality:

Prostorová lokalita
Přistupované objekty leží blízko sebe. Kód: vnitřní smyčky. Data: struktury (čtení jednoho pole bývá následováno čtením dalších polí).
Časová lokalita
Tatáž data se použijí vícekrát v krátkém čase. Kód: smyčky. Data: např. koeficienty digitálního filtru čtené každou vzorkovací periodu.

Cache procesoru – celkový obraz

  • Všechny load/store operace jdou přes cache.
  • CPU ↔ cache: rychlé propojení.
  • Cache ↔ hlavní paměť: sběrnice FSB.
  • Oddělené cache pro instrukce a data (L1i a L1d).

Hierarchie bývá víceúrovňová a per-jádro: CPU jádro → L1d / L1i → L2 → sdílená L3 → sběrnice → hlavní paměť. Ve vícejádrovém uspořádání má každé jádro vlastní rozdělené L1 cache, které napájejí sdílené vyšší úrovně připojené ke společné sběrnici a hlavní paměti.

Terminologie a typy miss

Cache hit
Paměťový požadavek je obsloužen z cache, bez přístupu do vyšší (pomalejší) úrovně.
Cache miss
Opak hitu – požadavek musí do pomalé hlavní paměti. Má více možných příčin (viz níže).
Cache line eviction
Vystěhování cache line, aby se uvolnilo místo pro nová data.
Cache replacement policy
Politika, která rozhoduje, kterou line vystěhovat: LRU, pseudo-LRU, random…

Typy cache miss (zdroje, proč k němu dojde):

  • Cold miss (compulsory) – první, povinný přístup k datům, která v cache ještě nebyla.
  • Capacity miss – pracovní množina (working set) se nevejde do cache.
  • Conflict miss – data sice kapacitně stačí, ale mapují se na stejnou line/set a vytěsňují se.
  • True sharing miss – jiné jádro zapsalo do téže sdílené line.
  • False sharing miss – jiné jádro zapsalo do jiné proměnné, která ale náhodou leží ve stejné line (viz dále).

Organizace a asociativita

Existují tři hlavní organizace cache podle toho, kam se daná adresa smí umístit:

OrganizaceMapováníKompromisVerdikt
Přímo mapovaná (direct-mapped)Každá adresa se mapuje přesně na jednu cache line.Jednoduchý HW, ale náchylná na conflict miss.Jednoduchá
Plně asociativní (fully associative)Adresa může jít kamkoli.Ideální miss rate, ale drahý komparátor.Ideální
Set-asociativní (set associative)N-cestné sety vyvažují cenu HW a miss rate.Používá se ve všech moderních CPU.Reálný kompromis

Přímo mapovaná cache

  • Každé místo v paměti má přiřazenu jen jednu cache line.
  • Místa vzdálená o násobky velikosti cache spolu vždy kolidují! (zde o násobky 8 × 64 B).
  • Kromě dat cache ukládá i tag.
  • Typická velikost cache line: Intel/AMD 64 B, Apple M1–M4 128 B.

Adresa se dělí na tři pole:

Tag (T) bity 31…
Cache Line (S) výběr line
Offset (O) …0 uvnitř line
Rozdělení 32bitové adresy v přímo mapované cache

Více vzdálených bloků paměti (např. adresy 8·64 B od sebe) ukazuje na tutéž cache line – proto v přímo mapované cache snadno dochází ke konfliktům.

Samovytěsňující se kód (direct-mapped)

void outer_func() {
  for (int i = 0; i < 1000; i++)
    inner_func();
}
void inner_func() {
  // do something
}
  • Pokud se outer_func() a inner_func() náhodou mapují na tutéž cache line, dostanete dva cache miss každou iteraci (fetch instrukcí)!
  • Řešení: dej související (hot) funkce k sobě, aby je linker umístil vedle sebe:
__attribute__((hot)) void outer_func();
__attribute__((hot)) void inner_func();

Set-asociativní cache

  • Většina dnešního hardwaru.
  • Typicky 8–16 cest (ways).
  • 2-cestná set-asociativní cache ≈ přímo mapovaná cache dvojnásobné velikosti.
  • Cache ≈ hardwarová hash-tabulka s omezenou velikostí bucketu (limit = počet cest).
  • Replacement policy určuje, která cesta se při conflict miss vystěhuje (LRU, pseudo-LRU, random…).

Dělení adresy má stejný tvar, ale prostřední pole teď vybírá set, ne jedinou line:

Tag (T) bity 31…
Cache Set (S) výběr setu
Offset (O) …0 uvnitř line
Rozdělení adresy v set-asociativní cache – prostřední pole vybírá set

Blok paměti lze umístit do kterékoli cesty svého setu, takže dva kolidující bloky mohou existovat současně (po jednom v každé cestě). Velikost cache line 64 B.

💬 Existuje samovytěsňování i u set-asociativní cache?

Ano, ale je méně pravděpodobné. Při N cestách může v jednom setu koexistovat až N kolidujících line, takže k samovytěsnění dojde, až když se do jednoho setu mapuje více než N hot line – to je mnohem vzácnější než u přímo mapované cache, kde už dvě kolidující line jsou v konfliktu.

Zápisové politiky

Cíl: vyhnout se zbytečnému vystěhování cachovaných dat.

Write-back (nejčastější)
Zapsaná data se cachují pro pozdější opětovné použití a do systémové paměti se nepíšou hned → snižuje provoz na sběrnici.
Write-through
Zapsaná data se cachují i zapisují do systémové paměti zároveň. Hodí se pro frame buffery, kde chcete zápis „vidět“ jako pixely na obrazovce. Lze vyvolat explicitně přes SSE streaming stores.
Write-combining
Zápisy se necachují, ale ukládají do write-combining bufferu. Jakmile je buffer plný, data se zapíšou přímo do paměti najednou. Často pro frame-buffer paměť – bez něj by aktualizace obrazovky vytěsnila všechna ostatní cachovaná data.

Streaming stores (obejití cache při zápisu)

SSE/AVX intrinsiky pro obejití cache při zápisu:

#include <emmintrin.h>
void _mm_stream_si32(int *p, int a);
void _mm_stream_si128(int *p, __m128i a);
void _mm_stream_pd(double *p, __m128d a);

#include <xmmintrin.h>
void _mm_stream_pi(__m64 *p, __m64 a);
void _mm_stream_ps(float *p, __m128 a);

#include <ammintrin.h>
void _mm_stream_sd(double *p, __m128d a);
void _mm_stream_ss(float *p, __m128 a);

💡 Kdy streaming stores

Používej je pro velké sekvenční zápisy, které se brzy znovu nečtou – zabráníš tím znečištění cache (cache pollution).

Benchmark: sekvenční vs náhodný přístup

#define REP 1000000
char A[65536*1024];

for (rep = 0; rep < REP; rep++)
  for (i = 0; i < WSS; i += 64)
    A[i]++;            // sekvenční

Cena za přístup (CPU cykly) roste schodovitě, jak pracovní množina přetéká z jedné úrovně cache do další. I při sekvenčním přístupu zůstává díky prefetchingu a prostorové lokalitě nízká (≤ ~7 cyklů):

for (rep = 0; rep < REP; rep++) {

Náhodný přístup ztrácí prostorovou lokalitu a je mnohem dražší – vrcholí kolem ~230 cyklů na přístup (vs. ~7 u sekvenčního). Při velkých pracovních množinách se k L3 miss přidávají i TLB miss jako významný přispěvatel:

Eviction a replacement policy

Při zaplnění setu se nějaká cache line musí vystěhovat (eviction). O výběru oběti rozhoduje replacement policy:

LRU (Least Recently Used)
Vystěhuje nejdéle nepoužitou line. Ideální aproximace lokality, ale přesné LRU je u mnoha cest drahé na realizaci.
Pseudo-LRU
Levnější přiblížení LRU (např. stromové bity), používané v reálném HW.
Random
Náhodný výběr oběti – minimální HW, překvapivě obstojné výsledky.

Cache-related preemption delay

  • Když je vlákno odebráno (preempted), nahrazující vlákno vytěsní data z cache.
  • Po skončení preempce zažije původní vlákno dávku cache miss.
  • Některé (starší) architektury musí při přepnutí adresního prostoru vyprázdnit TLB (flush). Moderní architektury označují TLB identifikátorem adresního prostoru (ASID, PCID…).
  • Vysoce výkonný software omezuje preempce (např. připínáním vláken). Kompromis: omezení preempce zvyšuje dobu odezvy!

TLB a překlad adres

TLB (Translation Lookaside Buffer) cachuje překlad virtuální → fyzická adresa.

  • Při TLB miss se musí provést page walk (procházení tabulek stránek, 2 až 5 úrovní).
  • Intel i7-2600 má 512 položek L2 TLB ⇒ 512 × 4 kB = 2 MB pokrytí.
  • Vylepšení: tzv. huge pages (1 stránka = 2 MB, PS=1). V Linuxu někdy automaticky, nebo explicitně přes hugetlbfs.

Při IA-32e stránkování se 48bitová lineární adresa dělí na PML4 (47:39) / Directory Ptr (38:30) / Directory (29:21) / Table (20:12) / Offset (11:0) a překlad prochází CR3 → PML4E → PDPTE → PDE → PTE → 4 KiB stránka → fyzická adresa (9bitový index na úroveň). Při velké stránce (PS=1) se procházení zkrátí.

  • Pro většinu datových struktur/algoritmů existují cache-optimalizované varianty.
  • Jsou složitější než učebnicové příklady – ale často 10× a víc rychlejší.
  • Viz též přednáška 2: Struct of Arrays (SoA) dosahuje 100 % využití cache line oproti 33 % u Array of Structs (AoS).

💡 SoA vs AoS

U Array of Structs (klasické pole objektů) se při čtení jednoho pole natáhne celý 64B řádek, ale využijí se z něj jen bajty toho jednoho pole (~33 %). U Struct of Arrays leží stejnojmenná pole pohromadě, takže sekvenční zpracování využije celý řádek (100 %).

Dynamická alokace (malloc(), new)

  • Paměťové alokátory se snaží udržovat prostorovou a časovou lokalitu.
  • Prostorová lokalita je obtížná, když je halda fragmentovaná (po mnoha operacích new/delete).
  • Časová lokalita – když se paměť uvolní, následná alokace ji znovu použije, protože je cache-hot.

💬 Vlastní alokátor

Pokud heuristiky alokátoru pro vaši zátěž selhávají, zvažte vlastní alokátor pro hot path.

Příklad: násobení matic

Násobíme C = A · B, kde cij = Σk=1..N aik · bkj.

💡 Pointa příkladu

Cache-optimalizovaná verze je 10× rychlejší než naivní implementace při stejné složitosti O(N³) – pouze přeskupením pořadí přístupů do paměti.

Předpoklady animované simulace cache na slidech: prvek matice double (8 B), velikost cache line 16 B (2 prvky), plně asociativní cache, L2 = 128 B, L1 = 32 B, každá matice má v příkladu vlastní cache.

Naivní implementace

for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    for (k = 0; k < N; ++k)
      C[i][j] += A[i][k] * B[k][j];

Matice A se prochází po řádcích (dobrá lokalita), B po sloupcích (každý přístup na novou cache line → špatná lokalita). Uváděné součty pro daný krok: mem:27, cache hits:12 ≈ 44 %.

Implementace s transpozicí

double B[N][N];
for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    B[i][j] = Bsrc[j][i];

for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    for (k = 0; k < N; ++k)
      C[i][j] += A[i][k] * B[j][k];

Transpozicí B prochází vnitřní smyčka B po řádcích (B[j][k]) místo po sloupcích, čímž se obnoví prostorová lokalita. Výkon: naivní = 100 %, transpozice = 23,4 % – 4× zrychlení jen díky transpozici. Součty kroku: mem:27, cache hits:16 ≈ 59 %.

Dlaždicová (tiled) implementace

for (k1 = 0; k1 < N; k += tile)
  for (j1 = 0; j1 < N; j += tile)
    for (i1 = 0; i1 < N; i += tile)
      for (i = i1; i < i1 + tile; ++i)
        for (j = j1; j < j1 + tile; ++j)
          for (k = k1; k < k1 + tile; ++k)
            C[i][j] += A[i][k] * B[j][k];
  • Každá dlaždice (tile) se vejde do cache, takže se její data mnohokrát použijí před vystěhováním.
  • Výkon: 17,3 % výkonu naivní verze (9,5 % se SIMD) – 6× zrychlení.
  • Dlaždice se vejdou do L2, ale ne do L1 – jen 75 % hitů v L1.

Součty kroku: mem:384, cache hits:344 ≈ 89 %.

Dvouúrovňové dlaždicování

for (k2 = 0; k2 < N; k2 += tile2)
  for (j2 = 0; j2 < N; j2 += tile2)
    for (i2 = 0; i2 < N; i2 += tile2)
      for (k1 = k2; k1 < k2 + tile2; k += tile1)
        for (j1 = j2; j1 < j2 + tile2; j += tile1)
          for (i1 = i2; i1 < i2 + tile2; i += tile1)
            for (i = i1; i < i1 + tile1; ++i)
              for (j = j1; j < j1 + tile1; ++j)
                for (k = k1; k < k1 + tile1; ++k)
                  C[i][j] += A[i][k] * B[j][k];
  • Vnější dlaždice (tile2) cílí na L2, vnitřní (tile1) na L1.
  • 79 % hitů v L1 (oproti 75 % u jednoúrovňového dlaždicování).

Součty kroku: mem:378, L1 hits:298 ≈ 78 %, L2 hits:40 ≈ 10 %, cache hits:338 ≈ 89 %.

Rekurzivní násobení matic

  • Zobecnění na libovolný počet úrovní cache (cache-oblivious přístup).
  • Násobení N × N = 8 multiply-add operací nad maticemi (N/2) × (N/2).

Bloky se rozdělí na čtvrtiny (A11…A22 atd.) a výsledné bloky se skládají např. jako C11 = A11B11 + A12B21. Animace: YouTube playlist „Matrix multiplication cache animations“, zdrojový kód GitHub: wentasah/mmul-anim.

Cache koherence (MESI)

  • V symetrických multiprocesorových (SMP) systémech nemohou cache jednotlivých CPU pracovat nezávisle.
  • Udržení jednotného pohledu na paměť pro všechny procesory se nazývá cache koherence.
  • Pokud nějaký procesor zapíše do cache line, ostatní procesory musí odpovídající line ze svých cache vyčistit. (Pozor: komunikace mezi jádry/sokety je „pomalá“.)
  • Synchronizační protokol: MESI(F).
    • Dirty cache line není přítomna v žádné jiné cache procesoru.
    • Čisté kopie téže cache line mohou být v libovolně mnoha cache.

Každé CPU má store buffer, privátní L1$ a L2$, vše připojené ke sdílenému CPU interconnectu (sběrnici), který vede i k paměti.

💬 Hot / dirty cache line

Hot line je nedávno použitá (cache-hot) – přístup k ní je rychlý. Dirty line obsahuje zápis, který ještě nebyl propsán do paměti; v MESI smí existovat jen v jedné cache a při sdílení se musí nejdřív propsat do paměti a přenést do ostatních cache.

True a false sharing

True sharing

  • Program je pomalý, protože cache line se sdílenými daty putují z jednoho CPU na druhý.
  • Typický příklad: každý mutex je sdílený mezi CPU.
  • Když je to problém (příliš velká contention): jemnější zamykání, nebo změna datové struktury (např. per-CPU data) a/nebo cache-friendly algoritmy.
std::atomic_int32_t counter;

void thread_cpu0() {
  while (true)
    counter++;
}
void thread_cpu1() {
  while (true)
    counter++;
}

Obě vlákna inkrementují tentýž counter, takže jeho cache line při každém inkrementu „ping-ponguje“ (poskakuje) mezi cache obou jader.

Když všechna CPU atomicky inkrementují globální proměnnou, každé jádro táhne jedinou sdílenou line přes interconnect → každé CPU má cache miss a inkrementy se fakticky serializují. Když inkrementují per-CPU proměnnou, každé jádro sahá jen na svou lokální line → žádný cache miss, mnohem rychlejší a paralelní.

💬 AMD Ryzen 5000 – NUMA i v jednom soketu

Chiplet design: 2 CCD (Core Complex Dies), každý se 2 CCX (Core Complexes). Latence napříč CCD je vyšší než uvnitř CCX ⇒ NUMA-like efekty i na jednom soketu. Software musí znát topologii.

False sharing

Data přistupovaná z různých CPU nejsou sdílená, ale náhodou leží v jedné cache line:

// Per-CPU counters
std::atomic_int32_t counter_cpu0;
std::atomic_int32_t counter_cpu1;

void thread_cpu0() {
  while (true)
    counter_cpu0++;
}
void thread_cpu1() {
  while (true)
    counter_cpu1++;
}
  • Problém není vidět ve zdrojovém kódu! counter_cpu0 a counter_cpu1 jsou různé proměnné, přesto mohou skončit v téže 64bajtové line, která pak poskakuje mezi jádry.
  • Pomoct může kombinace více HW výkonnostních čítačů – to dělá linuxový perf c2c (cache-to-cache).

Řešení: zarovnat každou proměnnou na hranici cache line:

std::atomic_int32_t counter_cpu0 __attribute__((aligned(64)));
std::atomic_int32_t counter_cpu1 __attribute__((aligned(64)));

Velikost cache line zjistíte za běhu přes sysconf(_SC_LEVEL1_DCACHE_LINESIZE) nebo při překladu přes gcc -DLEVEL1_DCACHE_LINESIZE=$(getconf LEVEL1_DCACHE_LINESIZE) ....

Příklad: __thread brání false sharingu

Per-thread seed přidělený kompilátorem/linkerem (dobře – thread-local storage):

__thread unsigned int seed;

void gen_rnd_key(char *key, int length)
{
    int len = strlen(charset);
    for (int i = 0; i < length; i++) {
        int r = rand_r(&seed);
        key[i] = charset[r];
    }
    key[length] = '\0';
}

__thread v praxi zajistí, že každá seed leží v jiné cache line. Naproti tomu pole přidělené programátorem způsobí false sharing:

unsigned int seed[NUM_CPUS];

void gen_rnd_key(char *key,
    int length, int thread_idx)
{
    int len = strlen(charset);
    for (int i = 0; i < length; i++) {
        int r = rand_r(&seed[thread_idx]);
        key[i] = charset[r];
    }
    key[length] = '\0';
}

Sousední položky seed[] sdílejí cache line. Tato verze je asi o 40 % pomalejší na serveru ritchie při běhu na 32 CPU.

💡 Fyzikální mez komunikace mezi jádry

Rychlost světla za 1 takt @ 3 GHz urazí round-trip ≈ 5 cm; rychlost elektronů v tranzistorech ≈ 0,03–0,3·C. Tím je shora omezeno, jak rychlá vůbec může komunikace mezi jádry být.

NUMA a rozložení vláken

  • Každý socket CPU má vlastní lokální paměť.
  • Přístup do vzdálené paměti jde přes interconnect (QPI, Infinity Fabric).
  • Lokální: ~100 ns, vzdálená: ~150–200 ns.
  • First-touch politika: OS alokuje paměť na uzlu, kde do ní vlákno poprvé zapíše.

Schéma: čtyři procesory propojené křížem; každý má vlastní banky paměti a I/O řadiče. Přístup do paměti přímo připojené k procesoru je rychlý (lokální), dosažení paměti jiného procesoru znamená jeden či více skoků přes interconnect (vzdálené).

Migrace vláken mezi jádry

Load balancing OS migruje vlákna mezi jádry, přičemž migrovaná vlákna ztratí svou cache stopu i NUMA lokalitu.

Řešení: připnout vlákna k CPU (nastavit CPU affinity):

cpu_set_t cpuset;
pthread_t thread;
thread = pthread_self();
/* Set affinity mask to include
   only CPU 3 */
CPU_ZERO(&cpuset);
CPU_SET(3, &cpuset);
s = pthread_setaffinity_np(
  thread, sizeof(cpu_set_t),
  &cpuset);

💡 Size matters

I když máme terabajty paměti, na velikosti a rozložení datových struktur stále záleží – jen pár kilobajtů paměti je rychlých, zbytek je pomalý! Cache-optimalizované algoritmy mohou být 10–100× rychlejší než naivní. Když profiler hlásí spoustu cache miss a vy nevidíte žádná sdílená data, zkontrolujte false sharing.

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Daný kus kódu má vysoké cache misses – navrhni úpravy a odůvodni je (funkčnost musí zůstat).

Postup: nejdřív zjistit, jaký miss (cold/capacity/conflict, true/false sharing) – např. přes perf stat, perf c2c. Typické úpravy zachovávající funkčnost:

  • Změnit pořadí přístupů / smyček, aby byl přístup sekvenční po řádcích (prostorová lokalita) – např. transpozice nebo prohození vnitřních smyček (viz násobení matic, 4×).
  • Blocking / tiling – zpracovávat data po dlaždicích, které se vejdou do cache, aby se data před vystěhováním mnohokrát použila (časová lokalita).
  • Změnit rozložení dat: Array of Structs → Struct of Arrays (100 % využití line místo 33 %); nahradit spojový seznam polem.
  • Při sdílení napříč jádry zarovnat / oddělit proměnné (aligned(64), __thread, per-CPU data) proti false sharingu.
  • Velké sekvenční zápisy, které se brzy znovu nečtou → streaming stores (necachovat).

Vždy: jedna změna → změř, že to pomohlo a výsledek je stejný.

zkoušková otázka Kód s funkcí, kde perf zaměřený na TLB našel hotspot – kde je problém a jak ho zlepšit při zachování funkčnosti?

Vysoké TLB miss znamenají, že přístupy sahají na příliš mnoho různých stránek – typicky náhodný / roztroušený přístup přes velkou pracovní množinu, takže se nevejde do TLB (např. i7-2600 pokryje 512 × 4 kB = 2 MB). Každý miss spustí page walk (2–5 úrovní). Řešení:

  • Zlepšit prostorovou lokalitu – přístup sekvenčně, blocking/tiling, kompaktnější datová struktura (pole místo pointer-chasing), aby se sahalo na méně stránek.
  • Huge pages (2 MB) – jedna položka TLB pokryje 2 MB místo 4 kB; v Linuxu automaticky (THP) nebo přes hugetlbfs.
  • Zmenšit pracovní množinu / zhustit data, aby se vešla do dosahu TLB.
3 b perf record -e l1i_miss ./myprog ukázal zpomalení ve process_data, která v cyklu volá tři static inline funkce do_a/do_b/do_c – vysvětli příčinu a navrhni změnu zachovávající funkčnost.

Událost l1i_miss = miss v L1 instrukční cache. Pokud se kód process_data a kód volaných funkcí mapuje na kolidující cache line (zejména u přímo mapované / málo cestné L1i), vzájemně se v cyklu vytěsňují – jde o variantu samovytěsňujícího se kódu: každá iterace platí miss na fetch instrukcí. (I když jsou označené inline, nemusí se reálně inline-ovat, takže se skáče na vzdálené adresy.)

Řešení: dostat související horký kód blízko k sobě, aby se nevytěsňoval:

  • Označit funkce __attribute__((hot)), aby je linker umístil pohromadě.
  • Vynutit skutečný inlining (__attribute__((always_inline))) – tělo do_a/do_b/do_c se vloží přímo do smyčky a zmizí skoky na vzdálené instrukce.

Funkčnost zůstává, mění se jen rozmístění/inlining kódu.

zkoušková otázka Kód s cache misses na poli partial_sums – v čem je problém a jak ho vyřešit při zachování funkčnosti?

Pole mezisoučtů indexované per-vlákno/per-CPU (partial_sums[thread_idx]) je klasický false sharing: sousední položky leží v téže 64B cache line, a protože do nich různá jádra zapisují, line neustále poskakuje mezi jejich cache (koherenční provoz) → spousta miss, přestože vlákna logicky nic nesdílejí. Ve zdrojáku to není vidět; odhalí to perf c2c.

Řešení: oddělit položky do různých cache line:

  • Zarovnat / vypolstrovat položky na velikost line: __attribute__((aligned(64))), padding ve struktuře.
  • Použít thread-local (__thread) mezisoučet a sečíst až na konci.
  • Velikost line zjistit přes sysconf(_SC_LEVEL1_DCACHE_LINESIZE).
zkoušková otázka Jak je organizovaná cache v dnešních CPU? Co je true a false sharing a proč je důležité na sdílení dbát?

Organizace cache: víceúrovňová hierarchie (L1i/L1d per jádro, L2, sdílená L3), data se přenášejí po cache line (typicky 64 B, Apple 128 B). Dnešní cache jsou set-asociativní (typicky 8–16 cest) – ≈ hardwarová hash-tabulka s bucketem o velikosti počtu cest; adresa se dělí na tag / set / offset, eviction řídí replacement policy (LRU/pseudo-LRU/random). Koherenci mezi jádry zajišťuje MESI(F) (dirty line jen v jedné cache).

True sharing: více jader přistupuje k témuž údaji (globální čítač, mutex) – jeho line ping-ponguje mezi jádry a přístupy se serializují.

False sharing: jádra přistupují k různým proměnným, které ale náhodou leží v jedné line – ta poskakuje stejně, ačkoli logicky se nic nesdílí; ve zdrojáku neviditelné.

Proč na tom záleží: komunikace mezi jádry/sokety je drahá; oba jevy zabíjejí škálování. Řeší se jemnějším zamykáním, per-CPU/thread-local daty a zarovnáním na cache line (aligned(64)).

P5 Proč je DRAM pomalá a proč se data přenášejí v dávkách velikosti cache line?

Přístup probíhá ve třech analogových fázích – ACTIVATE (tRCD ≈ 13 ns), READ (CL ≈ 10 ns), PRECHARGE – a ty se za 20 let téměř nezlepšily (roste jen propustnost). Poslat jediné 64bitové slovo by tato zpoždění platilo zbytečně, proto se přenáší celá dávka (burst) o velikosti cache line, aby se latence amortizovala.

P5 Vysvětli přímo mapovanou, plně asociativní a set-asociativní cache. Co je samovytěsňující se kód?

Direct-mapped: každá adresa má 1 line – jednoduché, ale conflict miss při kolizích (násobky velikosti cache kolidují). Fully associative: kamkoli – ideální miss rate, drahý komparátor. Set-associative: N cest na set – reálný kompromis (8–16 cest), ≈ hash-tabulka. Samovytěsňující se kód: dvě hot funkce (outer/inner) se mapují na tutéž line a v každé iteraci se vzájemně vytěsňují (2 miss/iter). Řešení: __attribute__((hot)), aby je linker dal vedle sebe; u set-asociativní je problém méně pravděpodobný (potřeba > N kolidujících line).

P5 Co je NUMA a first-touch? Jak zabránit ztrátě lokality při migraci vláken?

NUMA = každý socket má lokální paměť (~100 ns), vzdálená přes interconnect je pomalejší (~150–200 ns). First-touch: OS alokuje paměť na uzlu, kde do ní vlákno poprvé zapíše. Migrace vlákna mezi jádry ztratí cache stopu i NUMA lokalitu → propad. Řešení: připnout vlákna (pthread_setaffinity_np / cpu_set_t) a řídit alokaci přes libnuma (numa_set_localalloc, numa_bind…). I jednosoketové chiplety (Ryzen, 2 CCD × 2 CCX) mají NUMA-like efekty.

📌 Co si zapamatovat

  • DRAM je principiálně pomalá: ACTIVATE/READ/PRECHARGE ~10–13 ns (tRCD ≈ 13 ns, CL ≈ 10 ns), latence se 20 let nezlepšila – roste jen propustnost; data jdou v dávkách velikosti cache line (64 B / Apple 128 B).
  • Cache těží z prostorové a časové lokality. Hit = z cache, miss = do paměti; druhy: cold/compulsory, capacity, conflict, true/false sharing.
  • Organizace: direct-mapped (jednoduchá, conflict miss, samovytěsňování), fully associative (ideál, drahá), set-associative (8–16 cest, reálný HW). Adresa = tag / set / offset.
  • Zápis: write-back (default), write-through (frame buffer, _mm_stream_*), write-combining. Eviction řídí replacement policy (LRU/pseudo-LRU/random).
  • TLB cachuje virtuální→fyzické překlady; miss → page walk (2–5 úrovní); huge pages (2 MB) zvyšují pokrytí.
  • Cache-friendly: pole dobrá, spojové seznamy/stromy špatné (kromě B-stromů); SoA 100 % vs AoS 33 % využití line.
  • Násobení matic O(N³): transpozice (4×), tiling (6×, 75 % L1), dvouúrovňové (79 % L1), rekurze → 10×+ jen přeskupením přístupů.
  • Koherence MESI(F): dirty line jen v jedné cache. True sharing (týž údaj) serializuje; false sharing (různé proměnné v jedné line) je skrytá brzda – detekce perf c2c, oprava aligned(64) / __thread / per-CPU data.
  • NUMA: lokální ~100 ns, vzdálená ~150–200 ns, first-touch; migrace ztrácí cache i NUMA lokalitu → připnout vlákna (pthread_setaffinity_np) a řídit alokaci libnuma.
  • Pointa: jen pár kilobajtů je rychlých; cache-optimalizované algoritmy bývají 10–100× rychlejší. Hodně miss bez viditelného sdílení → hledej false sharing.
