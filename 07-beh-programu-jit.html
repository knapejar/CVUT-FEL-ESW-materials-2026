Běh programu – virtuální stroj, bytecode, JIT

Jak vlastně běží program v Javě: zdrojový kód se přeloží do platformně nezávislého bytecodu, který JVM vykonává v hybridním modelu – nejprve interpretovaně, později pomocí JIT zkompilovaně do nativního assembleru. Projdeme rozložení paměti JVM (zásobník a frames vs. halda), zásobníkový instrukční model, prohlížení class souborů disassemblerem (javap) a dekompilátorem (procyon), tiered JIT kompilaci (C1/C2, OSR, de-optimalizace), safepointy a Time-To-Safe-Point a nakonec profiling aplikací (sampling vs. tracing). Vracející se motiv: kompilátor není kouzelník – piš efektivní algoritmy a pak měř → analyzuj → optimalizuj.

Model virtuálního stroje

Model virtuálního stroje používá .NET i JVM (na JVM běží i jazyky Scala, Kotlin, Jython, Clojure, …). Zdrojový kód se přeloží do bytecodu virtuálního stroje a ten běží v hybridním modelu vykonávání:

  • interpretovaně, nebo
  • zkompilovaně do nativního assembleru s automatickými optimalizacemi specifickými pro danou platformu (např. SIMD).
VýhodyNevýhody
Platformní nezávislost (architektura RISC/CISC, velikost slova, OS), reflexe (inspekce a úprava za běhu), menší velikost binárky. Pomalejší start (interpretace, latence kompilace), menší kontrola nad nízkoúrovňovými optimalizacemi.

Rozložení paměti JVM

Paměť JVM se dělí na oblasti specifické pro vlákno (zásobník) a oblasti sdílené všemi vlákny (Non-Heap a Heap).

🔑 Co má každé vlákno vlastní a co je sdílené

Vlastní pro každé vlákno (zásobník)
Program Counter (PC), Java zásobník (drží frames) a nativní zásobník. Žádné jiné vlákno do nich nevidí.
Sdílené – Non-Heap
Ostatní nativní paměť, Metaspace (obsahuje Code Cache a Method Area) a Class Data s běhovým constant poolem (Run-Time Constant Pool: řetězcové a číselné konstanty, odkazy na třídy, pole, metody, name & type, invoke dynamic) plus Method Code (kód metod).
Sdílené – Heap (halda)
Drží objekty i internované řetězce. Young Generation = Eden Space + Survivor Spaces; Old / Tenured Generation. Minor GC čistí mladou generaci, Major GC povyšuje/čistí starou generaci.

💡 Vyčerpání zásobníku vs. haldy

Přetečení zásobníku → StackOverflowError; vyčerpání haldy → OutOfMemoryError.

Frame (rámec) a stack-oriented execution

Každé vlákno má svůj vlastní zásobník frames (pevné velikosti). Frame vzniká při volání metody a zaniká při návratu.

  • Interpretovaný frame odpovídá přesně jedné metodě.
  • Zkompilovaný frame zahrnuje všechny inlinované metody.
  • Velikost framu je dána v době kompilace (uložena v class souboru pro interpretované vykonávání).

Co frame obsahuje:

  • Pole lokálních proměnných (libovolného typu), indexované 0 1 2 3 4 5 …:
    • {this} – jen u instanční metody! (zabírá lokální proměnnou 0)
    • {argumenty metody}
    • {lokální proměnné}
  • Operandový zásobník (libovolný typ) – LIFO; roste, jak se hodnoty vkládají.
  • Odkaz na běhový constant pool (definici třídy).

Bytecode a zásobníkové vykonávání

JVM používá pro většinu operací zásobníkový (stack-oriented) model: instrukce pracují přímo s operandovým zásobníkem.

Příklad – výpočet (2 + 3) × 11 + 1. Vstupní proud instrukcí a stav operandového zásobníku v čase:

Vstup23add11mul1add
Zásobník (vrchol)3111
Zásobník (dno)2255555556

Tedy: vlož 2, vlož 3, add → 5, vlož 11, mul → 55, vlož 1, add → 56.

Opcody

Jeden JVM opcode má 1 bajt. Instrukce = opcode (kód operace) + případné parametry. Kategorie:

  • load a store (aload_0, istore, aconst_null, …)
  • aritmetika a logika (ladd, fcmpl, …)
  • typové konverze (i2b, d2i, …)
  • manipulace s objekty (new, putfield, getfield, …)
  • správa zásobníku (swap, dup2, …)
  • řízení toku (ifeq, goto, …)
  • volání metod (invokespecial, areturn, …) – manipulace s framy
  • výjimky a monitory/konkurence (athrow, monitorenter, …)

Prefixy/sufixy značí typ – i, l, s, b, c, f, d a a (reference). Lokální proměnné fungují jako registry – např. istore_1 (proměnná 0 je this u instanční metody).

Srovnání fragmentu CPU assembleru a ekvivalentního JVM bytecodu (AT&T syntaxe):

; CPU assembler (AT&T syntaxe)
mov   %rax,%r8
shl   $0x5,%eax
sub   %r8d,%eax
add   %ecx,%eax
inc   %edx
# JVM bytecode
iconst_0
istore_3
iload_3
bipush  100

CPU assembler používá reálné registry; JVM bytecode vkládá/odebírá z operandového zásobníku a číslované lokální proměnné používá jako pseudo-registry.

Class soubor, disassembler a dekompilátor

Výstupem kompilace Javy je class soubor (.class): jeden class soubor = jedna třída; obsahuje bytecode metod, constant pool a metadata (fields, metody). Struktura ClassFile dle specifikace JVM:

ClassFile {
    u4              magic;
    u2              minor_version;
    u2              major_version;
    u2              constant_pool_count;
    cp_info         constant_pool[constant_pool_count - 1];
    u2              access_flags;
    u2              this_class;
    u2              super_class;
    u2              interfaces_count;
    u2              interfaces[interfaces_count];
    u2              fields_count;
    field_info      fields[fields_count];
    u2              methods_count;
    method_info     methods[methods_count];
    u2              attributes_count;
    attribute_info  attributes[attributes_count];
}

Prvními 4 bajty class souboru je magické číslo CA FE BA BE. V ASCII jsou v binárce čitelné řetězce z constant poolu (Employee, java/lang/Object, update, employeeData, <init>, SourceFile, …).

Příklad 1 – Employee<Type>

public class Employee<Type> {
    private Type data;
    public int id;

    public Employee(Type data, int id) {
        update(data, id);
    }

    private void update(Type data, int id) {
        this.data = data;
        this.id = id;
    }

    public Type employeeData() {
        return data;
    }
}

Generika existují jen v době kompilace (type erasure) – deskriptory používají Ljava/lang/Object;, zatímco záznamy Signature (např. #10 = TType;) generickou informaci zachovávají. Konstanty jsou indexované a odkazují se přes #n.

Dekompilátor – procyon

procyon je open-source Java dekompilátor: převádí bytecode zpět do čitelného kódu podobného zdroji. Výstup (Procyon v0.5.30) je funkčně shodný s originálem; kosmetické rozdíly: modifikátory final, přejmenované parametry (type, n) a explicitní this.:

// Decompiled by Procyon v0.5.30
package employee;

public class Employee<Type> {
    private Type data;
    public int id;

    public Employee(final Type type, final int n) {
        this.update(type, n);
    }

    private void update(final Type data, final int id) {
        this.data = data;
        this.id = id;
    }

    public Type employeeData() {
        return this.data;
    }
}

Příklad 2 – daysInMonth / compute

Cíl: pochopit, jak se jednoduchá logika překládá do bytecodu, a vidět neefektivity před zásahem JIT. daysInMonth vrací počet dní v měsíci, compute sečte dny přes milion měsíců.

private static Integer daysInMonth(int month, int year) {
    int retVal;
    switch (month) {
        case 1: case 3: case 5: case 7: case 8: case 10: case 12:
            retVal = 31; break;
        case 2:
            retVal = (year % 4 == 0 && (year % 100 != 0 || year % 400 == 0)) ? 29 : 28;
            break;
        case 4: case 6: case 9: case 11:
            retVal = 30; break;
        default:
            throw new IllegalArgumentException("Unknown month: " + month);
    }
    return new Integer(retVal);
}

private static int compute() {
    int month = 4;
    int year = 2000;
    int o = 0;
    for (int i = 0; i < 1_000_000; i++) {
        o += daysInMonth(month, year);
    }
    return o;
}

⚠️ javac neoptimalizuje

Při kompilaci zdroje se neoptimalizuje! Bytecode věrně odráží strukturu zdrojového kódu – neefektivní zdroj vede k neefektivnímu bytecodu. Offset 10 je začátek smyčky, kde jsou čtyři int lokály a prázdný zásobník.

Kompilace zdrojového kódu (javac)

Kompilace zdroje (zdroj ⇒ bytecode) nedělá optimalizace:

  • Bytecode není sám o sobě efektivnější než váš zdrojový kód.
  • Invarianty smyček se neodstraňují.
  • Konkrétně žádné: loop unrolling, algebraické zjednodušení, strength reduction.

Volitelně lze bytecode transformovat před spuštěním. Příklad ProGuard – obfuskátor včetně bytecode optimalizací:

  • shrinker – odstraní nepoužitý kód,
  • optimizer – optimalizuje bytecode (mění přístupový vzor private/static/final, inlinuje),
  • obfuscator – obfuskuje názvy, mění rozložení,
  • preverifier – zajistí načtení tříd.

Obfuskace = ztížení čitelnosti pro člověka při zachování stejné funkcionality. Pokud se compute zjednoduší → rychlejší interpretace a lepší výstup JIT.

JIT kompilace – C1, C2 a tiered přístup

Just-in-time (JIT) kompilace překládá bytecode na nativní kód za běhu. JIT dělá adaptivní optimalizaci (tiered kompilace od verze 7):

  • Vyvažuje kompromis mezi JIT kompilací a interpretací.
  • Identifikuje často vykonávané části ("hot spots"), včetně dat o vztahu volající–volaný pro virtuální volání metod.
  • Spouští dynamickou rekompilaci podle aktuálního profilu vykonávání.
  • Inlining pro snížení režie volání, optimalizace větvení.
  • Může dělat spekulativní předpoklady (např. přeskočit kód) → později může být nutné kód de-optimalizovat.

💬 AOT (Ahead-of-Time)

AOT kompilace překládá do assembleru před spuštěním virtuálního stroje a odstraňuje fázi warm-up. V Javě 17 byla odstraněna.

JIT kompilátory C1 a C2

JIT kompilátory jsou asynchronní (běží na více vláknech).

C1 – rychlá kompilaceC2 – plně optimalizující, vysoký výkon
  • zjednodušený inlining a efektivní využití CPU registrů,
  • window-based optimalizace nad malou množinou instrukcí,
  • intrinsic funkce s vektorovými operacemi SIMD (Math, arraycopy, …).
  • dead code elimination, loop unrolling, loop invariant hoisting, common sub-expression elimination, constant propagation,
  • plný inlining, plná de-optimalizace (zpět na úroveň 0),
  • escape analysis, null check elimination,
  • pattern-based loop vectorization a super word packing (SIMD).

Tiered compilation – úrovně 0–4

CompLevel_none            = 0,   // Interpret
CompLevel_simple          = 1,   // C1
CompLevel_limited_profile = 2,   // C1, čítače invokací & backedge
CompLevel_full_profile    = 3,   // C1, čítače invokací & backedge + mdo
CompLevel_full_optimization = 4, // C2

💡 Typická cesta kompilace

0 → 3 → 4 (interpret → C1 s plným profilováním → C2). V příkladu: tier 3 nastoupí při count=256; práh pro spuštění tier 4 (C2) je 1024 invokací (0x1ff8 >> 3 = 1024) a tier 4 se reálně zkompiluje při count=5376.

OSR, de-optimalizace a warm-up

On-stack replacement (OSR) je optimalizace metody za běhu: začíná na cílech skoků v bytecodu (goto, if_*). Díky OSR lze nahradit právě běžící metodu (typicky uvnitř dlouhé smyčky) její zkompilovanou verzí, aniž by se čekalo na další zavolání.

Log tiered kompilace zapneš přes -XX:+PrintCompilation (a volitelně -XX:+PrintInlining). Sloupec flags ukazuje n (native), s (synchronized), % (OSR); sloupec tier číslo úrovně:

67   3       java.lang.String::hashCode (55 bytes)
...
84   3       SwitchTest::daysInMonth (144 bytes)
85   3       SwitchTest::compute (35 bytes)
88   21 %    SwitchTest::compute @ 10 (35 bytes)
89   21 %    SwitchTest::compute @ -2 (35 bytes)   made not entrant
93   24 %    SwitchTest::compute @ 10 (35 bytes)
94   22      SwitchTest::compute (35 bytes)

Cesta optimalizace na assembleru (tier 3 → tier 4)

Příklady běží na JVM 8 (64-bit), Intel Haswell, AT&T syntaxe (assembler se zapne přes -XX:+UnlockDiagnosticVMOptions -XX:+PrintAssembly).

  • Tier 3 = C1 s čítači invokací & backedge + MethodDataOop (MDO). Tableswitch je řetěz cmp/je porovnání proti %esi (vstup month), s default skokem. Alokace Integer je inlinovaná v TLAB (0x10 = 16 B velikost instance); pokud v TLABu není místo → nový TLAB + externí alokace.
  • Tier 4 = C2 bez profilovacích čítačů. Tableswitch je jump přes lookup tabulku (jeden indexovaný nepřímý skok místo řetězu cmp/je). Konstruktory Integer.<init> → Number.<init> → Object.<init> jsou inlinované (hot), je vidět komprimovaný OOP na třídu Integer a prefetch 3 cache lines dopředu.

🔑 Struktura objektu na haldě (64-bit JVM)

Objekt 8B mark word 4B/8B Klass ref. ... data objektu ... hlavička 12 nebo 16 B
Pole (array) 8B mark word 4B/8B Klass ref. 4B délka pole sekvence hodnot hlavička 16 nebo 20 B

Hlavička = mark word (8 B) + Klass ref (4 B / 8 B). Instance Integer má 0x10 = 16 B. U pole hlavička navíc obsahuje 4 B délku.

💡 De-optimalizace a warm-up

Cesta default (výjimka IllegalArgumentException) nemá profil → je považována za nepravděpodobnou (uncommon) → návrat (fallback) do interpretu (de-optimalizace této větve, uncommon_trap), tier 3 verze se zahodí (made not entrant). U compute proběhne tier 4 OSR @10 (backedge_count=101376) a nakonec C2 celou milionovou smyčku díky full inline, dead code elimination, escape analysis, loop invariant hoisting a strength reduction složí na konstantu – výsledek 30 000 000 se vrátí bez jediné iterace:

mov    $0x1c9c380,%eax         ; 30 000 000
add    $0x10,%rsp
pop    %rbp
test   %eax,-0x214f0dc(%rip)   ; {poll_return}
retq                           ; RAX = návratová hodnota (primitivní int)

To je odměna za warm-up. JVM proto vyžaduje warm-up, aby využila výhody C2 (či C1); kompilátor ale neudělá všechno → piš efektivnější algoritmy.

💬 32-bit vs 64-bit JVM a komprimované OOP

32-bit (max ~3 GB haldy): menší paměťová stopa, ale pomalejší operace s long & double. 64-bit do ~32 GB: rychlejší long/double, mírně větší stopa; komprimované OOP jsou při použití trochu pomalejší, ale zaberou méně paměti → méně častý GC → lepší výkon. Nad ~32 GB: rychlé reference, ale plýtvání pamětí (48 GB ≈ 32 GB s komprimovanými OOP).

Safepoint je stav, kdy jsou všechny OOP (Ordinary Object Pointers) v dobře definovaném stavu popsaném OopMap (GC mapy) – OOP může JVM bezpečně manipulovat, zatímco je vlákno pozastaveno.

Kde safepoint nastává:

  • v interpretovaném módu – mezi libovolnými dvěma bytecode instrukcemi,
  • ve zkompilovaném C1/C2 kódu – na konci všech (neinlinovaných) metod, na back-edge nepočítané smyčky a během běhového volání JVM,
  • při parkování, blokování na IO, monitoru či zámku,
  • při běhu JNI – nevyžaduje pozastavení vlákna (nativní vlákna jsou de facto už na safepointu).

🔑 Použití safepointu (globální vs. lokální)

Globální safepoint (všechna vlákna) – stop-the-world pauza
GC, výpis vláken, thread dump, heap dump, získání všech stack trace, redefinice tříd (instrumentace), debug.
Lokální safepoint (jen běžící vlákno)
de-optimalizace, OSR.

Time To Safe Point (TTSP)

TTSP je čas, který potřebují všechna vlákna, aby dosáhla safepointu. Ovlivňuje délku GC pauz i přesnost profilování. Měří se přes flagy -XX:+PrintSafepointStatistics -XX:+PrintGCApplicationStoppedTime.

okno = TTSP
2Vlákna běží k safepointukaždé Java vlákno běží, dokud nedosáhne safepointu
3@Safepointposlední vlákno (J1) dorazilo – konec okna TTSP
4Actual GetStackTrace workteprve teď profiler sbírá stack trace
5resumeAll()vlákna se rozběhnou dál
Okno od žádosti, dokud nedorazí poslední vlákno, je TTSP. Běžící (zelená) i odplánovaná (žlutá) Java vlákna do TTSP přispívají; nativní (červená, JNI) jsou už na safepointu.

Okno od žádosti, dokud nedorazí poslední vlákno, je TTSP. Běžící (zelená) i odplánovaná (žlutá) Java vlákna do TTSP přispívají; nativní (červená, JNI) jsou už na safepointu. Proto na TTSP záleží: než profiler zavolá GetStackTrace, musí počkat na poslední vlákno.

Profiling: CPU a paměť, sampling vs. tracing

Profiling sleduje CPU (kde se tráví čas) a paměť (alokace a využití). Dva základní módy:

SamplingTracing (instrumentace)
  • periodické vzorkování zásobníků běžících vláken,
  • žádné počty invokací, není 100% přesné (vzorkovací chyby),
  • žádná modifikace bytecodu/assembleru,
  • dopad 1–2 % na výkon (TTSP, thread dumpy, analýza).
  • sleduje vstup/výstup metod, traceObjAllocations,
  • instrumentovaný bytecode → ovlivněný výkon → ovlivněné optimalizace kompilátoru,
  • kód se může chovat jinak než v produkci.

⚠️ Tracing zabíjí JIT optimalizace

Instrumentace vkládá volání (rootMethodEntry/rootMethodExit) – v příkladu 749 bajtů assembleru na jeden rootMethodEntry u daysInMonth, 313 bajtů u vnitřních konstruktorů. CPU tracing compute je výrazně pomalejší: žádný object escape, žádný invariant hoisting, žádná strength reduction (smyčka zůstane celá). Ve VisualVM se pak daysInMonth i konstruktory volají 1 000 000× a dominují běhu.

Nástroje pro profiling v Javě

  • VisualVM (jvisualvm) – monitoring, troubleshooting a profiling; sampling i instrumentace, statistiky objektů na haldě, heap dumpy, monitoring GC a vláken. Dříve v JDK 6–8, dnes samostatný nástroj.
  • JMC (Java Mission Control) + JFR (JDK Flight Recorder) – součást JDK, sampling přes Flight Recorder s téměř nulovou režií (i v produkci); zaznamenává kompilaci, GC, I/O, výjimky, statistiky objektů. Spustí se i přes -XX:StartFlightRecording=duration=100s,filename=myrecording.jfr.
  • async-profiler – nízkorežijní profiler pro Linux/macOS (využívá perf), integrovaný do IntelliJ IDEA Ultimate.
  • JConsole – základní monitoring (halda, CPU, vlákna) přes JMX.
  • Profiler integrovaný v IDE (IntelliJ IDEA Ultimate od 2019.3).
  • CLI nástroje: jcmd, jhat, jinfo.

💡 Postup při optimalizaci Java aplikace

  1. Začni samplingem (nízká režie, neovlivní JIT).
  2. Najdi úzká hrdla (kde se tráví nejvíce času) – může to být i mimo JVM (latence externí DB, souborový systém).
  3. Tracing použij jen na hotspoty.
  4. Profiluj i paměť – hledej objekty, které vznikají nejčastěji (autoboxing int vs. Integer, zbytečné alokace), případně analyzuj heap dump.
  5. Zkoušej různé nástroje a metody (sampling vs. instrumentace) – kvůli odlišné chybovosti dají odlišný pohled.
  6. Měř → analyzuj → optimalizuj a opakuj.

Mikro- vs. makrobenchmark

Testuj reálné aplikace, ideálně za realistických podmínek.

MikrobenchmarkMakrobenchmark
Měří velmi malé jednotky. Pozor na:
  • warm-up – aby se měřil kód, ne kompilátor,
  • efekty kompilátoru – výsledek je nutné použít, hrozí přeřazení operací,
  • synchronizace u vícevláknových benchmarků,
  • měnící se předpočítané parametry ovlivňující složitost → v praxi různé optimalizace.
 Používá realistickou zátěž. Nejméně výkonná komponenta ovlivní celou aplikaci.
  • Rozuměj throughputu, elapsed a response time; počítej s outliery (např. GC). Používej existující generátory, ne vlastní.
  • Rozuměj variabilitě (vnitřní stav, systémový šum – zátěž/síť, pravděpodobnostní analýza).
  • Testuj brzy a často: opakuj měření, automatizuj testy, dobré pokrytí funkcí a vstupů, testuj na cílovém systému (JIT efekty, caching, systémový šum).

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Popiš paměť v JVM. Co má každé vlákno vlastní a co je společné? Co je stack-oriented execution? Co obsahují frames?

Paměť JVM: zásobník (vlastní pro vlákno) + Non-Heap a Heap (sdílené).

Vlastní pro každé vlákno: Program Counter, Java zásobník (frames) a nativní zásobník. Sdílené: Non-Heap = Metaspace (Code Cache + Method Area) a Class Data (běhový constant pool + Method Code); Heap = Young (Eden + Survivor) a Old/Tenured (drží i internované Stringy). Přetečení zásobníku → StackOverflowError, vyčerpání haldy → OutOfMemoryError.

Stack-oriented execution: JVM je zásobníkový stroj – instrukce pracují přímo s operandovým zásobníkem (LIFO), např. (2+3)×11+1 = postupné push/add/mul. Lokální proměnné fungují jako registry.

Frame obsahuje: pole lokálních proměnných (this na indexu 0 u instanční metody, pak argumenty a lokály), operandový zásobník (LIFO, roste) a odkaz na běhový constant pool. Vzniká při volání, zaniká při návratu; interpretovaný = 1 metoda, zkompilovaný = všechny inlinované.

zkoušková otázka Které optimalizace probíhají v C1, C2 a tiered kompilaci.

C1 (rychlá kompilace): zjednodušený inlining, efektivní využití registrů, window-based optimalizace nad malou množinou instrukcí, intrinsic funkce s SIMD (Math, arraycopy).

C2 (plně optimalizující): dead code elimination, loop unrolling, loop invariant hoisting, common sub-expression elimination, constant propagation, plný inlining, plná de-optimalizace, escape analysis, null check elimination, pattern-based loop vectorization a super word packing (SIMD).

Tiered kompilace (od Javy 7): adaptivní úrovně 0 (interpret), 1 (C1 simple), 2 (C1 + čítače invokací/backedge), 3 (C1 + čítače + MDO), 4 (C2). Typická cesta 0 → 3 → 4; JIT je asynchronní, identifikuje hot spoty a podle profilu rekompiluje.

zkoušková otázka Safepoint – uveď alespoň 2 použití; co je TTSP a proč na něm záleží.

Safepoint = stav, kdy jsou všechny OOP v dobře definovaném stavu (OopMap) a JVM s nimi může bezpečně manipulovat, zatímco je vlákno pozastaveno. V interpretu mezi libovolnými dvěma bytecody, v C1/C2 kódu na konci neinlinovaných metod / na back-edge nepočítané smyčky / během runtime volání.

Použití (alespoň 2): globální (stop-the-world) – GC, thread dump, heap dump, redefinice tříd (instrumentace), debug; lokální (jen běžící vlákno) – de-optimalizace, OSR.

TTSP (Time To Safe Point) = čas, než všechna vlákna dosáhnou safepointu. Záleží na něm, protože ovlivňuje délku GC pauz a přesnost profilování – stop-the-world operace nemůže začít, dokud nedorazí poslední vlákno. JNI/nativní vlákna jsou už na safepointu, ale běžící Java vlákna do TTSP přispívají.

zkoušková otázka Co je sampling a tracing? Jmenuj alespoň 2 programy pro profiling v Javě. Jak bys postupoval při optimalizaci Java aplikace?

Sampling: periodické vzorkování zásobníků běžících vláken; nemodifikuje bytecode, žádné přesné počty invokací, není 100% přesný (vzorkovací chyby), dopad jen ~1–2 %.

Tracing (instrumentace): vkládá kód na vstup/výstup metod (a traceObjAlloc); přesné počty, ale instrumentovaný bytecode ovlivní výkon i optimalizace JIT (vypne escape analysis, invariant hoisting, strength reduction) → kód se chová jinak než v produkci.

Nástroje (alespoň 2): VisualVM, JMC + JFR (Flight Recorder), async-profiler, JConsole, profiler v IntelliJ IDEA.

Postup optimalizace: 1) začni samplingem; 2) najdi úzká hrdla (i mimo JVM – DB, filesystem); 3) tracing použij jen na hotspoty; 4) profiluj i paměť (autoboxing, zbytečné alokace, heap dump); 5) zkoušej různé nástroje/metody pro odlišné pohledy; 6) měř → analyzuj → optimalizuj a opakuj.

P7 Co je model virtuálního stroje a jaké má výhody a nevýhody?

Zdroj se přeloží do bytecodu VM, ten běží v hybridním modelu (interpretovaně, nebo zkompilovaně do nativního assembleru). Používá JVM i .NET. Výhody: platformní nezávislost, reflexe, menší binárka. Nevýhody: pomalejší start a menší kontrola nad nízkoúrovňovými optimalizacemi.

P7 Co je OSR a de-optimalizace? Proč JVM potřebuje warm-up?

OSR (on-stack replacement) nahradí právě běžící metodu (typicky uvnitř dlouhé smyčky) zkompilovanou verzí na cíli skoku (back-edge), aniž by se čekalo na další zavolání. De-optimalizace = spekulativně zkompilovaný kód padne zpět do interpretu (uncommon trap), když se poruší předpoklad (např. neočekávaná větev), a daná zkompilovaná verze se zahodí. Warm-up je potřeba, aby JIT nasbíral profil a stihl zkompilovat hot spoty do C2 – teprve pak se projeví plné optimalizace (např. compute se zhroutí na konstantu 30 000 000).

P7 Jaký je rozdíl mezi disassemblerem, dekompilátorem a obfuskací?

Disassembler (javap) vytvoří čitelnou textovou podobu class souboru (constant pool, bytecode). Dekompilátor (procyon) převede bytecode zpět do kódu podobného zdroji (funkčně shodného, jen kosmetické rozdíly). Obfuskace (ProGuard) naopak ztíží čitelnost pro člověka při zachování stejné funkcionality (přejmenování, změna rozložení) a může bytecode i optimalizovat/zmenšit.

📌 Co si zapamatovat

  • VM model (JVM, .NET): zdroj → bytecode → hybridní běh (nejdřív interpret, pak JIT na nativní kód). Výhody: nezávislost, reflexe, malá binárka; cena: pomalejší start.
  • Paměť: vlastní pro vlákno = PC + Java/nativní zásobník (frames); sdílené = Non-Heap (Metaspace, Class Data + constant pool) a Heap (Young = Eden+Survivor, Old). Zásobník → StackOverflowError, halda → OutOfMemoryError.
  • Frame = pole lokálů (this = index 0) + operandový zásobník (LIFO) + odkaz na constant pool. JVM je zásobníkový stroj, opcode = 1 bajt.
  • .class = jedna třída; javap (disassembler), procyon (dekompilátor), ProGuard (obfuskátor). Generika jen v době kompilace (erasure): descriptor (JVM) bez nich, Signature (kompilátor) s nimi.
  • javac neoptimalizuje – bytecode kopíruje zdroj (žádný unrolling, invarianty, strength reduction).
  • JIT: adaptivní, tiered (od Javy 7), asynchronní. Úrovně 0–4, cesta 0→3→4. C1 = rychlý/lehký; C2 = plný optimizér (escape analysis, SIMD…). OSR mění běžící metodu na back-edge; spekulace se může de-optimalizovat.
  • Safepoint = OOP v OopMap. Globální = stop-the-world (GC, dumpy, redefinice tříd), lokální = de-opt/OSR. TTSP ovlivňuje GC pauzy a přesnost profilování.
  • Komprimované OOP (32-bit, ≤~32 GB) šetří paměť → méně GC; nekomprimované 64-bit jsou rychlejší, ale větší.
  • Profiling: sampling (≤2 %, nepřesné, bez změny bytecodu) vs. tracing (přesné, ale ničí JIT optimalizace). Nástroje: VisualVM, JMC+JFR, async-profiler. Postup: nejdřív sampling → najdi hotspoty → tracing jen na ně → měř → analyzuj → optimalizuj.
  • Testuj reálně: mikro- (warm-up, použij výsledek) vs. makrobenchmark; AOT (odstraňuje warm-up) byl v Javě 17 odstraněn.
