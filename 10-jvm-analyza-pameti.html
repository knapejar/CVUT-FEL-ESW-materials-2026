Faktory výkonu a analýza paměti

Výkon aplikace ovlivňují dvě hlavní veličiny, a datové struktury figurují v obou:

  • Celková doba běhu
    • algoritmy (složitost, počet instrukcí, synchronizace),
    • režie správy paměti (garbage collection),
    • datové struktury – rychlost přístupu k datům, efektivita cache, tlak na GC.
  • Spotřeba paměti
    • datové struktury – paměťová efektivita.

Analýza paměti přichází ve dvou podobách:

Statická analýzaDynamická analýza
analyzuje spotřebu paměti v daném okamžiku analyzuje chování paměti v čase
vhodná pro rozbor efektivity datových struktur a prohlížení obsahu paměti vhodná pro analýzu alokace objektů a identifikaci memory leaků

Heap dump

Pořízení heap dumpu rovněž vyžaduje globální safepoint (stop the world). Možnosti:

  • -XX:+HeapDumpOnOutOfMemoryError – automatický dump při OutOfMemoryError,
  • jmap -dump:live,format=b,file={name}.hprof {PID} – ruční dump,
  • nástroje pro analýzu: VisualVM, YourKit, …

Analyzátor heap dumpu vypisuje pro každý objekt jeho Retained Size a Shallow Size, umožňuje procházet cesty od GC Roots, prohlížet Allocations, Ages, Class Hierarchy a Incoming References. V příkladu má vybraný objekt java.lang.Thread ("main") shallow size 176 B, ale retained size 309 803 648 B (vlákno drží celý dosažitelný graf objektů); celá halda má ~295 MB silně dosažitelných objektů.

🔑 Dvě velikosti objektu

Shallow size
paměť alokovaná pro samotný objekt – jen jeho vlastní hlavička a pole.
Retained size
množství paměti, které objekt drží a které GC nemůže uvolnit, dokud objekt žije; tedy paměť, která by se uvolnila, kdyby byl objekt uvolněn GC. Rovná se vlastní shallow size + shallow size všech objektů přímo či nepřímo dosažitelných POUZE z tohoto objektu.

Klíčový rozdíl: objekt je v retained setu jiného objektu, jen pokud každá cesta od GC roots k němu vede přes tento objekt. Pokud existuje druhá cesta od jiného kořene, sdílený podgraf se z retained size vyloučí.

GC Roots
obj1drží {obj2, obj3, obj4}
obj2 → obj4
obj3dosažitelný JEN přes obj1
Vlevo: obj3 je dosažitelný jen přes obj1 → obj1 drží {obj2, obj3, obj4} (i přes pomocnou cestu obj5 → obj1)
GC Roots
obj1 → obj2 → obj4
obj3 / obj4druhá cesta přes obj5
Vpravo: obj3 je dosažitelný i přes obj5 (druhá cesta ke sdílenému podgrafu) → obj3/obj4 nedrží jen obj1, jsou z retained setu vyloučeny

Plýtvání pamětí, memory leak, výkon

Plýtvání pamětí (wasting memory)

Paměť neobsahuje užitečná data:

  • duplicitní řetězce – sdílet instance řetězců poolingem nebo intern();
  • duplicitní objekty se shodným obsahem polí – sdílet je, použít líné vytváření, dočasné (neperzistentní) použití;
  • pole nulové délky – zbytečná režie pro garbage collector; použít na třídu jeden sdílený singleton prázdného pole (např. ve static poli třídy);
  • null pole – objekty s mnoha null poli; pro zřídka přiřazovaná pole použít podtřídy;
  • řídká pole (sparse arrays) – mnoho null, nulových nebo shodných prvků; použít jinou datovou strukturu (např. mapu) nebo přepracovat algoritmus;
  • neefektivní datová struktura – velká režie na zbytečný obsah; použít jinou datovou strukturu.

Memory leak

Objekty, které se už nepoužívají, ale zůstávají dosažitelné kvůli přetrvávajícím referencím:

  • objekt držený zpětnou referencí z nestatické vnitřní třídy – implicitní zpětná reference z instance vnitřní třídy (i anonymní), např. u callback objektů;
  • řešení: minimalizovat použití instancí nestatických vnitřních tříd.

Výkon

Propustnost čtení/zápisu dat:

  • hashovací tabulky s nerovnoměrně rozloženými hash kódy – zhoršený výkon kvůli kolizím; řešení je lepší implementace hashCode.

Dynamická analýza (GC, allocation tracking)

GC telemetrie

Analýza GC telemetrie – např. VisualVM s pluginem VisualGC:

  • vytížení Eden space v čase,
  • GC cykly (sběry) a jejich trvání,
  • s minimálním dopadem na výkon sledované aplikace.

Pohled VisualGC vykresluje generace JVM jako stohované časové řady. Příklad hodnot ze slidu:

  • GC Time: 11 sběrů, 4,021 s; Last Cause: Allocation Failure
  • Eden Space (1,332 G, 234,000 M): 203,527 M, 8 kolekcí, 987,665 ms
  • Survivor 0 (455,000 M, 254,000 M): 176,657 M
  • Survivor 1 (455,000 M, 265,500 M): 0
  • Old Gen (2,667 G, 635,500 M): 319,685 M, 3 kolekce, 3,033 s

Řada Eden vykazuje charakteristický „pilový“ tvar (zaplní se, pak ji minor GC srazí dolů).

Porovnání heap dumpů

Porovnání heap dumpů:

  • rozdíly v počtech a velikostech objektů mezi stavy aplikace;
  • dumpy se všemi objekty (nejen živými) pomohou analyzovat alokace, pokud mezi nimi neproběhl GC cyklus;
  • každý heap dump vyžaduje globální safepoint (čas závisí na velikosti haldy).

Nástroj zobrazí per-třídu diff tabulku mezi „starým snímkem (baseline)“ a „tímto snímkem“ se sloupci Objects (+/-) a Size (+/-), kde zvýrazní, které třídy mezi oběma stavy narostly nebo se zmenšily.

Allocation tracking

Prezentovány jsou dva přístupy:

1) Memory profiler s bytecode instrumentací

  • sleduje každou n-tou alokaci objektu (kompromis mezi přesností a rychlostí);
  • ovlivňuje výkon profilované aplikace – vkládá bytecode traceObjAlloc:
    • přidává značné množství bytecode navíc a spotřebovává paměť,
    • snižuje možnosti JIT optimalizací.
  • profiler ukáže alokační Call Tree s počtem Objects a Size na jednotlivá místa volání (např. AppMain.main(String) v kořeni, detail do Lambda.java / Comparator.compare), plus záložky Packages, Object Explorer, Generations, Ages, Reachability, Class Loaders, Web Applications, Callee List, Quick Info.

2) Flight recording přes JMC – BEZ bytecode instrumentace

  • cíl: identifikovat velké alokace objektů mimo TLAB (Thread Local Allocation Buffer);
  • JMC (Java Mission Control) prezentuje:
    • statistiky TLAB – příklad: TLAB Count 393; Maximum TLAB Size 16,26 MB; Minimum TLAB Size 2,05 MB; Average TLAB Size 2,45 MB; Total Memory Allocated for TLABs 961,48 MB; Allocation Rate for TLABs 15,65 MB/s;

    • Allocation by Class s výsečovým grafem (pie chart) Allocation Pressure a tabulkou po třídách. Dominantní třídy: double[] (~329,37 MB), java.lang.Object[], java.util.HashMap$Node[], gnu.trove.list.linked.TIntLinkedList$TIntLink, java.util.LinkedList$Node, plus rozpad Pressure % na třídu (např. double[] 47,54 %, int[] 83,76 %);

Primitiva vs. objekty (hlavička, padding)

Primitiva

Velikosti v bajtech: boolean (1), byte (1), char (2), int (4), long (8), float (4), double (8).

  • bez implicitní alokace,
  • uloženy v proměnných nebo na operand stacku v rámci JVM.

Objekty

Objekty (s nevyhnutelnou režií hlavičky objektu) jsou alokovány na haldě:

  • každý objekt je standardně potomkem Object – metody clone(), equals, getClass(), hashCode(), wait(…), notify(…), finalize();
  • objekty pro primitiva: Boolean, Byte, Character, Integer, Long, Float, Double – mohou být null;
  • objekty s více poli používají zarovnání a padding podle skupin typů, rozložené v tomto pořadí (uvnitř téže skupiny v pořadí deklarace):
    • long a double (8 B),
    • int a float (4 B),
    • short a char (2 B),
    • byte a boolean (1 B),
    • reference (4 B / 8 B).

Struktura objektu (64bitová JVM): hlavička má 12 nebo 16 bajtů, pak následují data objektu (pole nadtřídy první):

mark word8 B
Klass ref.4 B / 8 B
… data objektupole nadtřídy první

Příklad – 64 bit, halda < 32 GB (komprimované OOP)

Třída, jejíž pole se rozkládají v paměti:

💡 Postřeh

Zapnutí/vypnutí komprimovaných OOP (překročení prahu 32 GB haldy) nafoukne každý objekt nesoucí reference – pro tuto třídu o 20 % (80 B → 96 B). Udržet haldu pod 32 GB je běžný způsob, jak ušetřit paměť.

Pole

Jednorozměrná pole:

  • speciální datová struktura, která ukládá více prvků stejného typu v lineárním pořadí; mají pevnou velikost danou při alokaci;
  • Java automaticky kontroluje meze;
  • alokována na haldě;
  • primitiva – hodnoty primitiv uloženy přímo;
  • objekty – ukládají reference na objekty (4 B nebo 8 B).

Vícerozměrná pole:

  • pole polí – ragged array (nestejné délky podúrovní);
  • pomalejší přístup kvůli dereferencím (více čtení z paměti) a kontrole více indexů;
  • zvažte zploštěné (flattened) pole.

Struktura objektu pole (64bitová JVM): hlavička má 16 nebo 20 bajtů (8 B mark word + 4 B / 8 B Klass reference + 4 B délka pole), pak následuje sekvence hodnot:

Autoboxing a jeho cena

Paměťová efektivita obalových objektů

Paměťová efektivita – 100 % efektivita znamená nulovou režii:

velikost užitečného obsahu
÷ retained size
× 100= efektivita [%]
efektivita = (velikost užitečného obsahu / retained size) × 100 [%]
  • koreluje s efektivitou cache – jeden cache line načítá souvislý blok 64 B paměti;
  • datová lokalita dále zrychluje zpracování využitím už cachovaných dat.

Efektivita jednoho obaleného primitiva (užitečná data vs. retained velikost objektu):

Autoboxing / unboxing

  • autoboxing a unboxing se dějí při přiřazení a předávání parametrů – přes metody valueOf({primitive}) (boxing) a {primitive}Value() (unboxing);
  • všechny obalové objekty primitiv jsou neměnné (immutable) – finální hodnoty;
  • pozor na neefektivity způsobené autoboxingem a unboxingem.

Příklad ukazující skrytý boxing/unboxing v bytecode:

public class PrimitiveObject {
    private static Integer integer = 0;

    public static void main(String[] args) {
        integer++;
    }
}

⚠️ Cena jednoho integer++

Jediný integer++ nad obaleným polem vyvolá virtuální volání intValue() a nový obalený objekt přes valueOf – v horké smyčce velmi nákladné.

Konverze v praxi – histogram četnosti slov

Histogram, který při každé aktualizaci tiše boxuje/unboxuje:

public static void main(String[] args) {
    Map<String, Integer> m = new TreeMap<String, Integer>();
    for (String word : args) {
        Integer freq = m.get(word);
        m.put(word, (freq == null ? 1 : freq + 1));
    }
    System.out.println(m);
}

Každé freq + 1 unboxuje freq, přičte a výsledek reboxuje přes Integer.valueOf; u velkých vstupů boxing dominuje.

Identita vs. rovnost

int i = 2;
int j = 2;
ArrayList<Integer> list = new ArrayList<Integer>();
list.add(i);
list.add(j);
System.out.printf(Boolean.toString(i == j));

Jaký je výstup? A co se změní pro i = 2000 a j = 2000?

Výrazi = j = 2i = j = 2000

  • Pro malé hodnoty (2) vrátí autoboxing tutéž cachovanou instanci Integer, takže == na referencích je true.
  • Pro 2000 (mimo rozsah cache) každé add obalí na jiný objekt Integer, takže == je false, zatímco equals je stále true.
  • Pozor: po serializaci je druhé porovnání vždy false – deserializace vytvoří čerstvé instance a obejde cache.

Identitní sémantika – cache ve valueOf

Identitní sémantika pochází z cache používané metodou valueOf({primitive}):

  • Short, Integer, Long – cachují rozsah <-128; +127>;
  • Byte – cachuje všechny hodnoty;
  • Character – cachuje <0; +127>.

To neplatí pro objekty vytvořené konstruktorem (např. new Integer(1)) – ty vždy vytvoří novou instanci.

Zjednodušená implementace cache pro Short (konceptuálně, z JDK):

private static class ShortCache {
    private ShortCache() {}

    static final Short cache[] = new Short[-(-128) + 127 + 1];

    static {
        for (int i = 0; i < cache.length; i++)
            cache[i] = new Short((short)(i - 128));
    }
}

public static Short valueOf(short s) {
    final int offset = 128;
    int sAsInt = s;
    if (sAsInt >= -128 && sAsInt <= 127) { // must cache
        return ShortCache.cache[sAsInt + offset];
    }
    return new Short(s);
}

public short shortValue() {
    return value;
}

Paměťová efektivita a kolekce

Proč jsou standardní Java kolekce neefektivní

Jak standardní JDK kolekce ukládají data:

  • LinkedList<E> – používá objekt Node<E> s obousměrnými odkazy (prev/next);
  • ArrayList<E> – postavený na poli elementData s referencemi na objekty;
  • HashMap<K,V> – zálohovaná hashovací tabulkou Node<K,V> s cachovaným hashCode a zřetězenými kolizemi (separate chaining).

Tři zdroje neefektivity se sčítají: boxing (každý prvek je samostatný obalový objekt), reference (pole referencí místo hodnot) a režie uzlů (Node s hlavičkou a odkazy). Měřená retained size a efektivita pro 1 milion prvků:

ObjektUžitečná velikostRetained (efektivita) – < 32 GBRetained (efektivita) – ≥ 32 GB
LinkedList<Integer>4 MiB34,33 MiB (11,65 %)47,26 MiB (8,46 %)
ArrayList<Integer>4 MiB17,73 MiB (22,56 %)25,72 MiB (15,55 %)
HashMap<Integer,Double>12 MiB70,19 MiB (17,10 %)87,67 MiB (13,69 %)

LinkedList je nejhorší (každý prvek potřebuje Node se dvěma referencemi plus obalený Integer). ArrayList je lepší, ale stále platí za boxing a pole referencí.

Specializované kolekce pro výkon

Dvě známé knihovny:

  • Trove – licence LGPL (Lesser GNU Public License);
  • FastUtil – licence Apache License 2.0.

Existují i další (např. Eclipse Collections). Specializované kolekce poskytují:

  • typově specifické mapy, množiny, seznamy a fronty – eliminují režii autoboxingu a unboxingu;
  • malou paměťovou stopu – mnohem lepší cachování, sekvenční přístup je velmi rychlý;
  • rychlý přístup a vkládání;
  • v mapách používají open addressing hashing místo zřetězování (chaining);
  • podporu velmi velkých kolekcí (> 2³¹ prvků) ve FastUtil;
  • podporu vlastních hashovacích strategií v Trove.

💡 Výsledek

Primitivní array listy dosahují ~100 % efektivity (holé int[], žádný boxing, žádná režie uzlů) a jsou nezávislé na prahu 32 GB (žádné reference ke komprimaci). Primitivní hashovací mapy zhruba zdvojnásobí až ztrojnásobí efektivitu oproti HashMap a jsou rovněž nezávislé na velikosti haldy.

Open hashing a řešení kolizí

Open addressing (otevřené adresování) eliminuje potřebu obalu Map.Entry<K,V>, který chaining vyžaduje. Místo zřetězeného seznamu uzlů se kolize ukládají do jiných slotů téže tabulky. Pro hledání volného slotu existují různá schémata.

Datové pole místo uzlů

  • typovaná pole klíčů a hodnot;
  • stavové pole byte – FREE, FULL, REMOVED (Trove, celkem 3 pole);
  • speciální sledování nulového/null klíče + výchozí návratová hodnota pro prázdné sloty (FastUtil, celkem 2 pole).

Load factor: menší load factor znamená méně konfliktů (Trove 0,5, FastUtil 0,75).

Řešení kolizí

Klasická schémata otevřeného adresování (zkoumáme posloupnost slotů, dokud nenajdeme volný/hledaný):

  • lineární sondování (linear probing) – při kolizi zkusíme následující slot (krok +1); lepší využití cache díky datové lokalitě. Používá FastUtil;
  • kvadratické sondování (quadratic probing) – krok roste kvadraticky (+1², +2², …), což snižuje shlukování (clustering) typické pro lineární probing;
  • dvojí hashování (double hashing) – krok je dán druhou hashovací funkcí; nejméně konfliktů, sekundární hash h₂ nesmí být 0. Používá Trove:
    h₁(k)primární hash
    + i × h₂(k)sekundární hash, h₂ ≠ 0
    mod |T|= h(i, k)
    h(i, k) = ( h₁(k) + i × h₂(k) ) mod |T|

Mazání a velikost tabulky

Složité mazání je nutné, aby zůstalo hledání po kolizích konzistentní:

  • přesun posledního kolizního prvku do uvolněného slotu (FastUtil);
  • ponechání smazaných prvků (označených REMOVED) – znovu využity pozdějšími put (Trove).

Velikost tabulky:

  • prvočíslo jako velikost tabulky snižuje kolize (Trove);
  • mocnina dvojky umožňuje efektivní bitové operace (FastUtil).

🔑 Trove vs. FastUtil – přehled

Trove (LGPL)
double hashing, prvočíselné velikosti tabulky, load factor 0,5, stavové pole FREE/FULL/REMOVED (3 pole), ponechává smazané sloty, podporuje vlastní hashování.
FastUtil (Apache 2.0)
lineární probing, velikosti = mocniny dvojky (bitové operace), load factor 0,75, 2 pole se sledováním 0/null klíče, při mazání přesouvá kolizní prvky, > 2³¹ prvků.

Strategie zvětšování kolekcí

Běhové neefektivity způsobuje zvětšování kolekcí (resizing) → pokud znáte očekávanou kapacitu, explicitně ji zadejte.

Mapy

  • HashMap
    • hashovací tabulka inicializována s prvním prvkem;
    • výchozí velikost 16 (výchozí load factor 0,75); vlastní kapacita se zaokrouhlí nahoru na mocninu dvojky;
    • zvětšení vyžaduje rehashing (iterace + put) všech předchozích prvků – strategie ×2;
    • zmenšování hashovací tabulky není vůbec podporováno.
  • TIntDoubleHashMap (Trove)
    • výchozí velikost 23 (výchozí load factor 0,5); vlastní kapacita se upraví na nejbližší vyšší prvočíslo;
    • růst znamená rehashing všech předchozích prvků – strategie nejbližší vyšší prvočíslo pro velikost × 2;
    • automatická kompakce po určitém počtu mazání – na nejbližší vyšší prvočíslo pro aktuálně uložené prvky; lze dočasně vypnout, pokud plánujete hodně mazání.
  • Int2DoubleOpenHashMap (FastUtil)
    • zálohovací pole alokována ihned;
    • výchozí velikost 16 (výchozí load factor 0,75); vlastní kapacita se zaokrouhlí na mocninu dvojky;
    • růst znamená rehashing – strategie ×2;
    • automatické zmenšení po remove, pokud vytížení klesne pod 25 % → struktura se zmenší faktorem 2; ne pod minimální velikost 16.

Další možné optimalizace: používat zástupné (stub) implementace pro prázdné/jednoprvkové kolekce, pokud aplikace obsahuje mnoho kolekcí (vyhnout se alokaci plných zálohovacích polí u prázdných nebo drobných kolekcí).

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Proč jsou kontejnery v Javě neefektivní? Jak to řeší nestandardní kontejnery? V čem je efektivní open hashing? Uveď 2 příklady řešení kolizí v open hashingu.

Proč jsou neefektivní: sčítají se tři režie – boxing (každý prvek je samostatný obalový objekt Integer/Double s hlavičkou, ~16–24 B na 4–8 B dat), reference (kolekce drží pole referencí, ne hodnoty) a režie uzlů (LinkedList.Node má dva odkazy, HashMap.Node má klíč, hodnotu, cachovaný hash a odkaz na další při kolizích – separate chaining). Měřeno na 1 M prvků: LinkedList<Integer> jen 11,65 %, ArrayList 22,56 %, HashMap<Integer,Double> 17,10 % efektivity.

Jak to řeší nestandardní kontejnery (Trove – LGPL, FastUtil – Apache 2.0): jsou typově specifické (TIntArrayList, Int2DoubleOpenHashMap), takže nepotřebují boxing a ukládají primitiva přímo do int[]/double[]. Tím dosahují malé paměťové stopy a výborného cachování. TIntArrayList/IntArrayList mají ~100 % efektivitu; primitivní mapy 43–48 %.

V čem je efektivní open hashing (open addressing): eliminuje obalové uzly Map.Entry – data jsou v souvislých typovaných polích klíčů a hodnot, kolize se ukládají do jiných slotů téže tabulky. Žádné reference na uzly, lepší datová lokalita a cache.

2 příklady řešení kolizí: lineární probing (FastUtil – zkouší následující slot, dobrá lokalita cache) a dvojí hashování / double hashing (Trove – krok dán druhou hash funkcí, h(i,k) = (h₁(k) + i·h₂(k)) mod |T|, h₂ ≠ 0; nejméně konfliktů). Případně lze uvést i kvadratické sondování (quadratic probing).

zkoušková otázka Profiling: 2 režimy + rozdíl static/dynamic analýza + shallow vs. retained size.

Dva režimy allocation trackingu: (1) memory profiler s bytecode instrumentací – sleduje každou n-tou alokaci, vkládá bytecode traceObjAlloc, zhoršuje výkon a snižuje JIT optimalizace; (2) flight recording přes JMC bez instrumentace – identifikuje velké alokace mimo TLAB, dává statistiky TLAB i alokací mimo TLAB a alokační tlak po třídách.

Static vs. dynamic analýza: statická analyzuje paměť v daném okamžiku (histogram jmap -histo:live, heap dump) – vhodná pro rozbor efektivity datových struktur. Dynamická analyzuje chování paměti v čase (GC telemetrie, porovnání heap dumpů, allocation tracking) – vhodná pro analýzu alokací a hledání memory leaků. Každý histogram/heap dump vyžaduje globální safepoint (stop the world).

Shallow vs. retained size: shallow = paměť samotného objektu (jen jeho hlavička a pole). Retained = shallow + shallow size všech objektů dosažitelných pouze přes tento objekt; tedy paměť, kterou by GC uvolnil, kdyby objekt zemřel. Pokud ke sdílenému podgrafu vede druhá cesta od jiného GC rootu, do retained setu se nepočítá.

P10 Vysvětli autoboxing/unboxing a proč je v horké smyčce drahý.

Autoboxing = automatické zabalení primitiva do obalového objektu přes valueOf({primitive}); unboxing = rozbalení zpět přes {primitive}Value(). Dějí se při přiřazení a předávání parametrů. Obalové objekty jsou neměnné.

Drahé proto, že každá operace nad obalenou hodnotou skrytě rozbalí a znovu zabalí. Např. integer++ nad polem Integer se v bytecode rozvine na intValue() (virtuální volání) + iadd + Integer.valueOf() (alokace nového objektu). V horké smyčce to znamená rychlé vytváření a zahazování objektů (alokační churn) a tlak na GC. Cache ve valueOf pomáhá jen pro malé hodnoty v rozsahu <-128;+127>.

P10 Co je retained size a jak se liší od shallow size?

Retained size objektu je paměť, kterou by garbage collector uvolnil, kdyby tento objekt zemřel – tedy jeho vlastní shallow size plus shallow size všech objektů, které jsou dosažitelné výhradně přes něj (každá cesta od GC roots k nim vede skrz tento objekt). Shallow size je naopak jen velikost samotného objektu (hlavička + jeho pole), bez čehokoli, na co odkazuje. Příklad ze slidu: vlákno "main" má shallow 176 B, ale retained 309 803 648 B, protože drží celý dosažitelný graf.

📌 Co si zapamatovat

  • Statická vs. dynamická analýza: statická = snímek v okamžiku (histogram jmap -histo:live, heap dump jmap -dump:live,format=b, -XX:+HeapDumpOnOutOfMemoryError); dynamická = chování v čase (GC telemetrie, porovnání heap dumpů, allocation tracking) a hledání leaků. Vše vyžaduje globální safepoint (stop the world).
  • Shallow vs. retained: shallow = jen objekt; retained = shallow + vše dosažitelné jen přes něj. Sdílený podgraf s druhou cestou od kořene se nepočítá.
  • Rozložení objektu (64 bit): hlavička 12/16 B (8 B mark word + 4/8 B Klass ref), pole po skupinách 8→4→2→1 B → reference, s paddingem. Komprimované OOP (< 32 GB) = 4B reference, nad 32 GB standardní OOP (8B) → příklad 80 B → 96 B (+20 %). Pole přidávají 4 B délku (16/20 B).
  • Autoboxing je drahý: efektivita obalů je nízká (Long/Double ~33 %, Boolean ~0,78 %); integer++ = intValue() + valueOf(). Cache valueOf dělá == pravdivé jen pro malé hodnoty (<-128;+127>) a po serializaci vždy false – používej .equals().
  • Primitivní kolekce vyhrávají: Trove (LGPL) a FastUtil (Apache 2.0) ruší boxing, používají open addressing místo chainingu. TIntArrayList/IntArrayList ~100 %, primitivní mapy ~43–48 % (vs. 13–17 % u HashMap), nezávislé na prahu 32 GB.
  • Open addressing: Trove = double hashing, prvočíselné velikosti, LF 0,5, pole FREE/FULL/REMOVED (3 pole), ponechává smazané. FastUtil = lineární probing, mocniny dvojky (bitové ops), LF 0,75, 2 pole, přesun kolizí při mazání.
  • Resizing: znej velikosti a předalokuj. ArrayList +~50 % (cap 10, líně), Trove/FastUtil seznamy ×2 (cap 10/16, ihned). HashMap ×2 (16, LF 0,75, bez zmenšení), Trove mapa prvočísla (23, LF 0,5, autokompakce), FastUtil mapa ×2 (16, LF 0,75) + autozmenšení ×2 při vytížení < 25 % (dno 16). Seznamy se zmenšují jen ručně.
