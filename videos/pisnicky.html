📚 Co se z toho naučíš

  • Data race = dvě nebo víc vláken přistupuje na stejné místo v paměti, aspoň jedno zapisuje, a není mezi nimi synchronizace -> nepředvídatelný výsledek (procesor i JIT přehazují pořadí instrukcí).

  • volatile zaručuje viditelnost (visibility) a uspořádání (ordering) jedné proměnné: čte se a píše rovnou z hlavní paměti a kolem zápisu se nepřehazuje pořadí.

  • V nativním kódu na x86: volatile zápis = lock addl $0x0,(%rsp) (bariéra na stacku), volatile čtení je bez bariéry.
  • Dvě atomické instrukce: CAS = compare-and-swap (instrukce cmpxchg) a fetch-and-add (instrukce lock xadd).
  • Obě atomiky dělají "cti-uprav-zapis" jako jeden nedělitelný krok; prefix lock zajistí výhradní cache-line (exkluzivní přístup po dobu operace).

📚 Co se z toho naučíš

  • RCU (read-copy-update) je synchronizační technika pro read-mostly zátěž (hodně čtenářů, málo zápisů).

  • Čtenáři v read-side kritické sekci NESMÍ blokovat ani spát -- jinak by grace period nikdy neskončila a stará verze by se nikdy neuvolnila.
  • RCU vs RW-lock: RW-lock čtenáři pořád berou zámek atomicky, takže pod kontencí kolabuje; RCU read-side je bez atomik -> škáluje, a updater čtenáře neblokuje.
  • Cena RCU: uvolnění staré verze je oddálené na grace period (paměť se neuvolní okamžitě, ale až dočtou všichni čtenáři).

📚 Co se z toho naučíš

  • Proč je paměť úzké hrdlo: CPU je extrémně rychlé (instrukce ~1 cyklus), ale RAM je pomalá (~100 až 300 cyklů). CPU pak jen čeká na data. Cache existuje proto, aby se čekalo míň.
  • Paměťová hierarchie od nejrychlejší/nejmenší po nejpomalejší/největší: registry (~1 cyklus) → L1 (~4) → L2 (~12) → L3 (~40) → RAM (~100–300). Každé patro je řádově pomalejší, ale větší.
  • Cache line: cache pracuje po linkách, typicky 64 B (Intel/AMD). Když čteš jeden bajt, natáhne se celá linka.
  • Lokalita: časová (co jsem použil, brzy použiju znovu) a prostorová (sousední adresy). Cache těží z obojího.
  • Cache miss = data nejsou v cache, musí se z pomalejšího patra → drahé.
  • Cache-friendly vs ne: sekvenční průchod polem (row-major) je rychlý — pomáhá prefetch a prostorová lokalita. Naopak skákání po paměti, pointer chasing a průchod 2D pole po sloupcích místo po řádcích = samé cache misses.
  • Oprava cache-miss kódu (časté u zkoušky): procházej paměť sekvenčně po řádcích (row-major), ne po sloupcích; zlepši rozložení dat; použij blokování / tiling.
  • TLB = malá cache překladu virtuální adresy na fyzickou (z page table). TLB miss → drahý page walk po tabulce stránek. Velké struktury a náhodný přístup způsobují TLB misses.
  • Hlavní message: o výkonu rozhoduje, kolik čekáš na paměť (lokalita a cache misses), ne počet instrukcí. Stejný počet operací může běžet řádově různě podle toho, jak sahá do paměti — sekvenční přístup je řádově rychlejší než náhodný.

📚 Co se z toho naučíš

  • Non-blocking (neblokující) = bez zámků. Žádné vlákno držící zámek (a třeba uspané) nezablokuje ostatní; vlákno se nezasekne kvůli jinému uspanému vláknu. Výhoda: vlákno se nemusí uspávat ani čekat na zámek, nehrozí deadlock.
  • Lock-free = systém jako celek vždy postoupí: aspoň jedno vlákno dokončí svou operaci v konečném počtu kroků. Jednotlivé vlákno ale může hladovět (pořád prohrávat retry smyčku).
  • Wait-free = silnější záruka: každé vlákno dokončí svou operaci v omezeném (konečném) počtu kroků nezávisle na ostatních. Žádné hladovění. Wait-free tedy dává horní mez na počet kroků (vhodné tam, kde potřebuješ záruku latence).
  • Vztah: wait-free je podmnožina lock-free — každý wait-free je i lock-free, ale ne naopak.
  • Staví se na CAS (compare-and-swap): atomicky, když je v paměti očekávaná stará hodnota, zapiš novou, jinak selži a zkus znovu (retry smyčka). Na x86 instrukce cmpxchg.
  • Treiberův non-blocking zásobník (LIFO) přepojuje hlavu (head) přes CAS — kanonický lock-free příklad.
  • ABA problém: vlákno přečte hodnotu A; jiné vlákno mezitím změní A → B → A; první vlákno udělá CAS, vidí pořád A a falešně uspěje, ačkoli se stav mezitím změnil (např. uzel byl uvolněn a jeho adresa recyklována). Stejná hodnota neznamená nezměněný stav.
  • Řešení ABA: verzovací počítadlo / razítko u ukazatele (tagged pointer / stamp), aby CAS porovnával i verzi — v Javě AtomicStampedReference / AtomicMarkableReference; dále hazard pointers nebo dvojitý (double-wide) CAS.

  • Fáze kompilace: 1) lexikální analýza (lexer) → rozseká text na tokeny; 2) syntaktická analýza (parser) → z tokenů postaví AST; 3) sémantická analýza (kontrola typů a pravidel); 4) generování mezikódu IR; 5) optimalizace nad IR; 6) generování cílového strojového kódu + alokace registrů.
  • AST (abstraktní syntaktický strom) = stromová reprezentace struktury programu (uzly = výrazy a příkazy), bez nepodstatných detailů syntaxe jako závorky a středníky.
  • IR (intermediate representation, mezireprezentace) sedí mezi frontendem (jazyk) a backendem (konkrétní CPU). Optimalizace se dělají nad IR jednou, nezávisle na jazyku i na architektuře. Frontend převede jazyk → IR, backend IR → strojový kód. (Příklady IR: LLVM IR, GCC GIMPLE.)
  • Typické optimalizace: inlining, constant folding (spočítání konstant předem), dead code elimination (odstranění mrtvého kódu), optimalizace smyček, vektorizace.
  • PGO (profile-guided optimization): 1) zkompiluj instrumentovaně (-fprofile-generate); 2) spusť na reprezentativních/reálných datech a posbírej profil (které větve a funkce jsou horké); 3) zkompiluj podruhé s profilem (-fprofile-use). Kompilátor pak optimalizuje podle reálného běhu → lepší inlining (podle skutečně horkých cest), uspořádání větví / code layout, lepší predikce skoků.
  • Hlavní message: zdroják jde přes lexer (tokeny) → parser (AST) → IR (kde se optimalizuje) → strojový kód. PGO = změř reálný běh a zkompiluj podruhé chytřeji.

📚 Co se z toho naučíš

  • NUMA (Non-Uniform Memory Access) = víceprocesorový systém, kde má každý socket vlastní lokální paměť. Přístup do lokální paměti je rychlý, do paměti jiného socketu (remote) pomalejší, protože jde přes propojení mezi sockety.
  • Non-uniform znamená, že doba přístupu závisí na tom, kde paměť fyzicky leží.
  • Důsledek: drž data blízko vlákna, které je používá, a alokuj na lokálním uzlu. Špatné rozmístění = vzdálené přístupy = pomalý běh.
  • First-touch policy: stránka se fyzicky alokuje na uzlu toho procesoru, který do ní první zapíše.
  • TLAB (Thread-Local Allocation Buffer) = každé vlákno v JVM dostane svůj malý exkluzivní kousek Edenu (mladé generace haldy), kam alokuje nové objekty jen pro sebe.
  • Alokace v TLABu je jen posun ukazatele (bump pointer), bez zámku, protože ten kousek paměti patří jen jednomu vláknu.
  • Když se TLAB naplní, vlákno si atomicky vezme nový kousek.
  • Proč TLAB existuje: alokace objektů je v Javě extrémně častá; bez TLABu by se vlákna prala o zámek na sdílené haldě. TLAB dělá alokaci rychlou a bez kontence.
  • Společná myšlenka: lokalita a vyhnout se sdílení. NUMA = používej lokální paměť místo vzdálené; TLAB = alokuj do svého lokálního bufferu místo do sdílené haldy.

📚 Co se z toho naučíš

  • Bloom filter = paměťově úsporná pravděpodobnostní datová struktura pro test příslušnosti do množiny (odpovídá na otázku „je prvek v množině?").
  • Mechanismus: bitové pole o m bitech + k nezávislých hashovacích funkcí. Vložení (add): prvek se prožene k hashemi a nastaví se k odpovídajících bitů na 1. Dotaz (test): když jsou všechny příslušné bity 1 → „možná tam je"; když je aspoň jeden bit 0 → „určitě tam není". Add i dotaz jsou O(k).
  • Klíčová vlastnost: žádné false negatives — když filtr řekne „není", tak tam opravdu není. Možné jsou jen false positives (filtr řekne „možná je", i když prvek není) kvůli kolizím bitů od jiných prvků.
  • Základní Bloom filter neumí mazat — bit může sdílet víc prvků, nulování by způsobilo false negatives.
  • Použití: levně a rychle odfiltrovat dotazy, co v množině určitě nejsou, ještě než sáhneš do drahého úložiště (disk, databáze) → ušetříš drahé vyhledávání. Typicky databáze, cache, web crawler („viděl jsem už tuhle adresu?"). V big-data eliminuje I/O tím, že se filtr drží v RAM.
  • Trade-off: víc prvků v menším poli → vyšší pravděpodobnost false positive. Větší pole a víc hashů sníží chybovost za cenu paměti.
  • Příklad ze slidů (ESW11): 1 milion stringů (celkem 25 MB) v Set<String> zabere > 50 MB retained size; Bloom filter s 1% false-positive rate zabere jen 1,13 MB a 7 hashovacích funkcí → více než 44× menší, a 99 % negativních dotazů jsou pravé negativy.
  • Rozšíření (mimo refrén, taky ze slidů): *Counting Bloom filter* — místo bitů má v každé pozici malý čítač (např. 3 bity) → umožní delete a odhad počtu, ale trpí přetečením čítače a delete může zvýšit false-negative rate. *Bitwise Bloom filter* — skládá víc counting filtrů přidávaných dynamicky, aby řešil tyto problémy.

📚 Co se z toho naučíš

  • SIMD (Single Instruction, Multiple Data) = jedna instrukce zpracuje víc datových prvků najednou (vektorově). Opakem je skalární zpracování (jedna instrukce = jeden prvek).
  • Vektorové registry drží víc prvků naráz: SSE = 128 bitů, AVX = 256 bitů, AVX-512 = 512 bitů (arch-specific, x86). Do 256bitového registru se vejde 8 čísel typu float, takže jedna sčítací instrukce sečte 8 dvojic naráz.
  • Vektorizace: kompilátor ji umí udělat sám (auto-vektorizace), nebo ji programátor napíše ručně přes intrinsics.
  • Podmínky vektorizace: iterace smyčky musí být nezávislé, data leží v paměti souvisle a nesmí mezi nimi být datové závislosti.
  • Zrychlení (datový paralelismus): teoreticky tolikrát, kolik prvků se vejde do registru. Reálně bývá menší kvůli přístupu do paměti a zarovnání dat.
  • V Javě: Vector API (modul jdk.incubator.vector). V C/C++: intrinsics nebo auto-vektorizace kompilátoru.
