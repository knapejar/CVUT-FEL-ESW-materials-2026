Serializace: JSON, XML, Protobuf, Avro, Cap'n Proto

Jak si programy vyměňují strukturovaná data – po síti, mezi procesy nebo ve sdílené paměti. Stránka staví textové formáty (XML, JSON) proti binárním přístupům („raw memory“) a serializačním frameworkům (CORBA, Protocol Buffers, Cap'n Proto, FlatBuffers, Apache Avro). Ústřední téma je výkon: parsování textu je drahé (chybné predikce skoků, nízké IPC), zatímco specializované knihovny (RapidJSON, simdjson) nebo binární formáty se zero-copy mohou být o řády rychlejší. Hlavní poselství: žádný formát nevyhrává ve všech ohledech – vyber ten, který sedí na tvé úzké hrdlo.

Co je serializace a k čemu

Serializace je převod strukturovaných dat z paměti (objekty, struktury) do proudu bajtů, který lze přenést po síti, mezi procesy (IPC) nebo uložit. Opačný proces – z bajtů zpět na objekty – je deserializace. Programy si potřebují vyměňovat data ve třech typických scénářích: přes síť, mezi procesy na jednom stroji a mezi vlákny.

Serializaci si můžeme zasadit do celého vykonávacího stacku. Dva identické stroje (každý běží celý stack) komunikují přes síť, jejíž vrstvy leží mezi oběma stacky:

Aplikační protokol (HTTP/gRPC) jede nad transportem (TCP/UDP), který jede nad síťovou/linkovou vrstvou (IP/Ethernet).

RPC / RMI a komunikace programů

Komunikaci programů můžeme rozdělit na dva scénáře.

Přes síť

  • Komunikační protokol (např. přes TCP nebo HTTP).
  • Strukturovaná data → serializace (JSON, protobuf, …).
  • Remote Procedure Call (RPC) – vzdálené volání procedury ve třech krocích:
    1. Serializuj název procedury a argumenty.
    2. Pošli požadavek a čekej na odpověď.
    3. Deserializuj odpověď.
  • Remote Method Invocation (RMI) – téměř totéž co RPC.
  • HTTP-based API – koncepčně podobné.

Na lokálním stroji

  • Jeden adresní prostor (vlákna) – datové struktury přímo v paměti; typový systém jazyka vám pomáhá vyvarovat se chyb.
  • Různé adresní prostory (procesy) – stejné jako „přes síť“. Ideálně zero-copy přes sdílenou paměť. V Linuxu: shm_open(...) a mmap().

Cena parsování textu

Parsování textu je z pohledu mikroarchitektury CPU drahé: rozhoduje se o téměř každém bajtu, což vede k chybným predikcím skoků a nízkému IPC. Demonstrováno na knihovně json-c (jazyk C) nad souborem JSON o velikosti 47 MB (měřeno na HW z roku 2017 – Haswell, glibc 2.24).

#include <json.h>
#include <stdio.h>
int main(int argc, char *argv[]) {
  struct json_tokener *tok = json_tokener_new();
  struct json_object *jobj;
  FILE *f = fopen("test.json", "r");
  char buf[1024 * 1024];
  do {
    size_t len = fread(buf, 1, sizeof(buf), f);
    jobj = json_tokener_parse_ex(tok, buf, len);
  } while (
      json_tokener_get_error(tok) ==
      json_tokener_continue
    );
  fclose(f);
  return 0;
}

RapidJSON je v praxi mnohem rychlejší

Specializovaná C++ knihovna RapidJSON na stejném souboru:

#include <rapidjson/document.h>
#include <rapidjson/filereadstream.h>
using namespace rapidjson;
int main(int argc, char *argv[])
{
  FILE *fp = fopen("test.json", "r");
  char readBuffer[1024 * 1024];
  FileReadStream is(fp, readBuffer, sizeof(readBuffer));
  Document d;
  d.ParseStream(is);
  fclose(fp);
  return 0;
}

Raw memory – nejrychlejší možný formát

Nejrychlejší cesta je poslat bajty z paměti přímo na drát:

struct data {
  char flag;
  long int data;
  char info[3];
};

void sendData(struct data &d) {
  send(sock, &d, sizeof(d));
}
void recvData(struct data &d) {
  recv(sock, &d, sizeof(d));
}

Nejrychlejší možná cesta, ale rozložení v paměti se „prolije“ na drát. Kompilátor navíc vkládá padding, aby byla každá položka přirozeně zarovnaná (offsety vlevo):

Nezabalená struktura (offset vlevo)
flagoffset 00 · 1 B
padding7 B (zarovnání)
dataoffset 08 · long int = 8 B
info[3]offset 16 · 3 B
padding5 B (do násobku 8)

Protože long int musí být zarovnán na 8 bajtů, za jediným char flag je 7 bajtů paddingu; za info[3] je padding, aby velikost struktury byla násobkem 8.

Raw memory potřebuje explicitní opravy rozložení

Aby raw struktury fungovaly mezi stroji, musíte ohlídat tři věci:

  • Typy s pevnou šířkou: int32_t místo int.
  • Endianitu: htole32() / le32toh().
  • Kontrolu rozložení: __attribute__((__packed__)).
struct __attribute__((__packed__)) data {
  char flag;
  int32_t data;
  char info[3];
};

// numerická pole deserializuj explicitně
d.data = le32toh(dd.data);
Zabalená struktura – žádný padding, celková velikost = 8 B
flagoffset 0 · 1 B
dataoffset 1 · int32_t = 4 B
info[3]offset 5 · 3 B

⚠️ Rychlé, ale křehké

Raw memory je extrémně rychlé, ale nepružné. Odesílatel i příjemce se musí shodnout na přesném rozložení; jakékoli přidání/odebrání pole nebo změna typu rozbije kompatibilitu. Funguje jen tam, kde se obě strany vyvíjejí společně. V dlouho žijících systémech vždy okamžitě verzujte protokol (verze nebo feature flagy).

CORBA, IDL a CDR

CORBA (Common Object Request Broker Architecture)

  • Jazykově nezávislý RPC framework z 90. let.
  • Interface Description Language (IDL) jako kontrakt.
  • Z IDL se generuje kód stub/skeleton.
  • Vlivný, ale na dnešní poměry těžkopádný.

Klient volá lokální Proxy (stub), která zabalí (marshaluje) volání přes klientský ORB (Object Request Broker), pošle Request přes síť serverovému ORB, ten přes POA (Portable Object Adapter) předá volání Servantu (skutečné implementaci objektu). Reply putuje zpět.

Interface Description Language (IDL)

  • V mnoha novějších frameworcích nazýván „schéma“.
  • Popisuje pouze typy a rozhraní, ne implementaci.
  • IDL kompilátor generuje jazykové vazby (bindings) a převodní kód do/z CDR (Common Data Representation).
module Finance {
  typedef sequence<string> StringSeq;
  struct AccountDetails {
    string name;
    StringSeq address;
    long account_number;
    double current_balance;
  };
  exception insufficientFunds {};
  interface Account {
    void deposit(in double amount);
    void withdraw(in double amount)
      raises(insufficientFunds);
    readonly attribute AccountDetails details;
  };
};

Common Data Representation (CDR)

  • Wire reprezentace: definuje, jak data vypadají „na drátě“. Většina dnešních serializačních schémat konverguje k něčemu podobnému. Data se zarovnávají (padding) jako v paměti ⇒ efektivní (de)serializace.
  • Endianita: odesílatel volí endian, hlavička zaznamenává, který byl použit; podobní hostitelé tak mohou vynechat prohazování bajtů.
  • TypeCodes: sebepopis IDL datových typů – umožní bezpečně posílat hodnoty Any.

Protocol Buffers (protobuf)

  • Popis dat – koncepčně podobný IDL.
  • Automatické generování kódu.
  • Částečný popis dat se posílá spolu s daty ⇒ méně problémů s verzováním.

syntax = "proto3";

  string query = 1;
  int32  page_number = 2;
  int32  result_per_page = 3;
}

Číslované „tagy“ jednoznačně identifikují pole a zároveň udržují zpětnou kompatibilitu (verzování).

Wire encoding

  • Páry klíč–hodnota.
  • Klíč = tag (číslo pole) + informace o typu ⇒ neznámé klíče (i jejich hodnoty) lze vždy přeskočit.
  • Kódování klíče: (field_number << 3) | wire_type, uloženo jako Varint.

Varint

  • Proměnný počet bajtů; malá čísla zaberou jen jeden bajt.
  • Varint je kompromis mezi velikostí kódovaných dat a rychlostí kódování/dekódování.
  • Nejvyšší (7.) bit je 1 ve všech bajtech kromě posledního (continuation bit). Bity 0–6 nesou hodnotu (skupiny po 7 bitech, little-endian).
9 = 0000 1001b→ Varint: 0000 1001 (1 bajt)
300 = 1 0010 1100b→ Varint: 1010 1100  0000 0010 (2 bajty)
300: skupiny po 7 bitech 0000010 0101100, nejnižší skupina první s nastaveným continuation bitem: 1·0101100  0·0000010

Znaménková čísla (sint) používají Zigzag kódování, aby se čísla blízká nule vešla do jednoho bajtu: n se zakóduje jako (n << 1) ^ (n >> 31) (pro 32 bitů; pro 64 bitů >> 63).

0→ 0
-1→ 1
1→ 2
-2→ 3
…→ …

Řetězce a zprávy (length-delimited)

Varintem kódovaná délka + bajty řetězce/zprávy. Prefix délky umožní parseru efektivně přeskočit neznámá pole.

message Test2 {
  required string b = 6;
}

S b = "testing" je kódování (hex): 32 07 74 65 73 74 69 6e 67

  • 32h = (6 << 3) | 2 // 6 = tag, 2 = length-delimited wire type
  • 07h = délka (7 bajtů)
  • 74 65 73 74 69 6e 67 = ASCII pro testing

Packed repeated pole

message Test4 {
  repeated int32 d = 4 [packed=true];
}
0x22tag · pole 4, wire type 2
0x06velikost payloadu · 6 B
0x031. prvek · varint 3
0x8E 0x022. prvek · varint 270
0x9E 0xA7 0x053. prvek · varint 86942

Díky prefixu délky lze celé pole přeskočit bez dekódování všech hodnot.

💬 Streaming zpráv

Parser neví, kde zpráva začíná a kde končí. Řešení: před zprávu dej její délku, pak přečti přesně tolik bajtů. (Stejný trik vyžaduje i laboratorní úloha – Java strana posílá velikost zprávy před zprávou samotnou.)

Příklad: OpenStreetMap PBF

OSM PBF používá syntaxi proto2 (předchází proto3):

message Node {
  required sint64 id = 1;
  repeated uint32 keys = 2 [packed = true]; // String IDs.
  repeated uint32 vals = 3 [packed = true]; // String IDs.
  optional Info info = 4;
  required sint64 lat = 8;
  required sint64 lon = 9;
}
message Way {
   required int64 id = 1;
   repeated uint32 keys = 2 [packed = true];
   repeated uint32 vals = 3 [packed = true];
   optional Info info = 4;
   repeated sint64 refs = 8 [packed = true];  // DELTA coded
}

Way je uspořádaný seznam odkazů na uzly (refs, delta-kódované pro kompaktnost). Keys/vals jsou paralelní pole ID z tabulky řetězců.

PBF = 881 MB, XML = 21 GB (jako XML ≈ 24× větší).

Cap'n Proto a FlatBuffers

Cap'n Proto

  • Vytvořil původní autor protobufu, o pár let později – zúročil poučení z protobufu.
  • Velmi efektivní pro komunikaci přes sdílenou paměť (i mezi různými jazyky). Použitelné i přes síť.

💡 Klíčová myšlenka – zero-copy

Není potřeba žádné de/enkódování – serializovaná forma je rovnou použitelná jako nativní (in-memory) forma (pokud se nepoužije packing). ⇒ Lze prostě mmap soubor do paměti a pracovat s daty.

Schéma a hex dump serializované zprávy:

  id @0 :UInt32;    # 0xab
  name @1 :Text;    # Alice
  email @2 :Text;   # alice@example.com
  phones @3 :List(PhoneNumber);
  struct PhoneNumber {
    number @0 :Text; # "555-1212"
    type @1 :Type;
  }
  employment :union {
    unemployed @4 :Void;
    employer @5 :Text;  # MIT
    school @6 :Text;
    selfEmployed @7 :Void;
  }
}

Cap'n Proto encoding

  • Bool: 1 bit.
  • Celá čísla: little-endian, nativní velikost, zarovnaná na násobek své velikosti (padding).
  • Výchozí hodnoty: vždy kódovány jako nula, tj. enc = val ^ default (^ = XOR). Pole rovné své výchozí hodnotě se serializuje na samé nuly.
  • Struktury: Pointer (= index) na data a podstruktury – stromová struktura, umožňuje přeskakování neznámých/nepotřebných dat.

Volitelný packing zbaví zprávu nulových bajtů: nastavené bity v prvním (tagovém) bajtu udávají, které z následujících 8 bajtů jsou nenulové; nenulové bajty pak následují.

unpacked08 00 00 00 03 00 02 00  19 00 00 00 aa 01 00 00 (16 B)
packed51 08 03 02   31 19 aa 01 (8 B)

V prvním 8-bajtovém slově jsou nenulové bajty na pozicích 0, 4, 6 → tag 0x51 (binárně 0101 0001), následují 08 03 02. Ve druhém slově dají nenulové bajty tag 0x31 a 19 aa 01. V uvedeném příkladu packing odstraní všech 83 nulových bajtů, ale přidá 17 bajtů packing tagů – jde tedy o směnu odstranění nul za režii tagu na každé slovo.

Čtení v C++ používá Reader typy (zero-copy pohledy na zprávu) a PackedFdMessageReader pro čtení packed zprávy přímo z file descriptoru; union se čte přes .which():

#include "addressbook.capnp.h"
#include <capnp/message.h>
#include <capnp/serialize-packed.h>
void printAddressBook(int fd) {
  ::capnp::PackedFdMessageReader message(fd);
  AddressBook::Reader addressBook = message.getRoot<AddressBook>();
  for (Person::Reader person : addressBook.getPeople()) {
    std::cout << person.getName().cStr() << ": "
              << person.getEmail().cStr() << std::endl;
    // ... phones, employment přes .which() ...
  }
}

FlatBuffers

  • Podobné Cap'n Proto: přístup k serializovaným datům bez parsování/rozbalování (zero-copy).
  • Od Googlu. Ideální pro náhodný přístup do velkých bufferů.

Apache Avro

  • Komunikační model: schéma v JSONu; po navázání spojení proběhne handshake schématu. V datech nejsou žádné tagy, protože schéma znají všechny strany (šetří místo).
  • Uložení do souboru: komprese a bloky umožňující přeskakovat data bez jejich deserializace.

Bezpečnost deserializace

Deserializace je útočná plocha.

⚠️ Nebezpečné formáty – spuštění libovolného kódu

  • Java native serialization – ObjectInputStream.readObject() spouští kód v gadget chains během deserializace. CVE-2015-4852 globálně položila servery WebLogic. Java to odrazuje; allowlisty ObjectInputFilter přidány v Javě 9.
  • Python pickle – pickle.loads() spustí libovolný Python bytecode. Nikdy nedeserializujte z nedůvěryhodného zdroje – bezpečná podmnožina neexistuje.
  • .NET BinaryFormatter – po letech CVE zcela odstraněn v .NET 9 (2024).

💡 Bezpečné už návrhem

Protobuf / Cap'n Proto / FlatBuffers jsou čistá data – žádné ukazatele na kód, žádná resoluce typů. Deserializace nemůže spustit kód; schéma přesně určuje, která pole existují. Zbylá rizika: billion laughs (hluboce vnořené struktury → kvadratická/exponenciální práce), integer overflow (varint pole mohou být nečekaně velká) a malformovaný vstup – po deserializaci vždy validujte.

Anotační serializace

Alternativa bez souboru se schématem a bez generátoru kódu: anotujte vlastní typy.

Schéma + codegen (.proto/.capnp/.fbs)Anotační (Serde, Jackson…)
Postupschéma → generátor → vygenerované typy → serializace zabudovaná vlastní struktura → derive makro / anotace → framework prozkoumá typ → serializace do libovolného backendu
✓ Pluscross-language · max. výkonvlastní typy · ergonomické
✗ Minuskrok navíc v sestavení (build) · cizí typyobvykle jen jeden jazyk
 
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize)]
struct Person { id: u32, name: String, email: String }

let json  = serde_json::to_string(&p)?;   // JSON backend
let bytes = rmp_serde::to_vec(&p)?;        // MessagePack – stejná struktura!
let proto = prost::Message::encode(&p)?;   // Protobuf backend

Obdoby v dalších ekosystémech: Jackson (Java, @JsonProperty), System.Text.Json (C#, [JsonPropertyName]), marshmallow (Python), encoding/json (Go, struct field tags).

RPC / RMI
Vzdálené volání procedury/metody: serializuj jméno + argumenty, pošli, deserializuj odpověď.

Popis typů a rozhraní (ne implementace); kompilátor z něj generuje vazby a (de)serializační kód.

Zkouškové otázky k tématu

Rozbal odpověď až poté, co si ji zkusíš sám zformulovat.

zkoušková otázka Máme zprávu k přenosu – jaký formát použijeme místo JSON? Popiš jeho binární reprezentaci a porovnej s JSON z hlediska rychlosti serializace, paměti a rozšiřitelnosti.

Vhodná volba je Protocol Buffers (protobuf) – binární, schématem definovaný formát s generováním kódu a širokou jazykovou podporou.

Binární reprezentace: zpráva je posloupnost párů klíč–hodnota. Klíč kóduje číslo pole a wire type: (field_number << 3) | wire_type, uložený jako Varint (proměnně dlouhé kódování, nejvyšší bit = continuation; malá čísla = 1 bajt). Znaménková čísla používají Zigzag ((n<<1)^(n>>31)), aby se hodnoty blízké nule vešly do jednoho bajtu. Řetězce a vnořené zprávy jsou length-delimited (varint délka + bajty). Příklad: string b = 6 s hodnotou "testing" → 32 07 74 65 73 74 69 6e 67 (32h = tag 6 + wire type 2, 07h = délka 7, pak ASCII).

Rychlost serializace: protobuf je výrazně rychlejší – odpadá parsování textu s rozhodováním na každém bajtu (json-c mělo IPC 1,59 a 36 % frontend stall na 47 MB JSONu; binární formát se ani neparsuje znak po znaku). Uber uvádí 3× rychlejší kódování, Discord ↓ 30 % CPU.

Paměť / velikost na drátě: menší – varint a kompaktní kódování místo textu; Google uvádí o ~40 % méně než JSON, OSM PBF je vs XML ≈24× menší.

Rozšiřitelnost (schema evolution): výborná díky číselným tagům – neznámé tagy (i s hodnotami) lze vždy přeskočit, takže staré i nové verze schématu spolu fungují bez přepisu. JSON sice tagy nepotřebuje (názvy klíčů jsou v datech), ale za cenu velikosti a pomalého parsování. Poznámka: kdyby šlo hlavně o sdílenou paměť / zero-copy, byl by lepší Cap'n Proto nebo FlatBuffers (serializovaná = paměťová forma, jen mmap).

P6 Jaký je rozdíl mezi textovou a binární serializací?

Textová (XML, JSON): člověkem čitelná, schéma volitelné, ale vysoká velikost i výpočetní režie – parsování se rozhoduje na téměř každém bajtu (chybné predikce skoků, nízké IPC, drahé přeskakování bílých znaků). Binární (protobuf, Avro, Cap'n Proto, raw struct): kompaktní, rychlá, většinou vyžaduje schéma a není čitelná okem. Extrém binárního je raw memory (nejrychlejší, ale křehké) a zero-copy formáty (serializovaná = paměťová forma).

P6 Co přináší IDL / schéma a CDR?

IDL (Interface Description Language), v novějších frameworcích zvané „schéma“, popisuje pouze typy a rozhraní, ne implementaci. Kompilátor z něj generuje jazykové vazby a převodní kód do/z wire formátu. Přínos: jazyková nezávislost, automatické generování kódu, kontrakt mezi stranami. CDR (Common Data Representation) je wire formát CORBy – definuje, jak data vypadají na drátě; odesílatel volí endianitu (hlavička ji zaznamenává), TypeCodes umožní sebepopis a posílání Any. Většina dnešních formátů konverguje k něčemu podobnému.

P6 V čem se liší Protobuf, Avro a Cap'n Proto?

Protobuf: binární, schéma .proto, číselné tagy v datech → snadné verzování a přeskakování neznámých polí; varint/zigzag/length-delimited kódování. Stále se musí (de)serializovat.

Avro: schéma v JSONu vyměněné handshakem, takže v datech nejsou žádné tagy (kompaktnější). Soubory podporují kompresi a skippable bloky. Silné v datových pipeline (Kafka).

Cap'n Proto: od autora protobufu; serializovaná forma = paměťová forma → žádné de/enkódování, jen mmap (zero-copy). Bool = 1 bit, integery little-endian a zarovnané, defaulty kódované jako nula přes enc = val ^ default, struktury přes pointery; volitelný packing odstraní nulové bajty. Ideální pro sdílenou paměť.

📌 Co si zapamatovat

  • Serializace = data z paměti → proud bajtů (síť, IPC, sdílená paměť); RPC/RMI přidají abstrakci volání procedury.
  • Textové formáty (XML, JSON) jsou čitelné, ale drahé – velká režie a pomalé parsování řízené větvením na každém bajtu. XML je nejtěžší.
  • Rychlost parseru je příběh mikroarchitektury: json-c IPC 1,59 / 36 % stall; RapidJSON IPC 2,23 / 18 % a ~8–10× rychleji; simdjson 3,5–4,5 GB/s přes SIMD.
  • Raw struct je nejrychlejší, ale křehký – nutné int32_t, htole32/le32toh, __packed__ a verzování.
  • CORBA přinesla IDL + stub/skeleton + CDR; Protobuf = číselné tagy, varint, zigzag, length-delimited.
  • Cap'n Proto / FlatBuffers = zero-copy (serializovaná = paměťová forma, mmap); Avro = schéma v JSONu přes handshake, bez tagů v datech.
  • Volba formátu je vícerozměrná (CPU, velikost, evoluce, náhodný přístup, čitelnost, schéma, zero-copy) – žádný nevyhrává ve všem.
  • Deserializace je útočná plocha: Java readObject(), Python pickle, .NET BinaryFormatter umí spustit kód; čistá data (Protobuf/Cap'n Proto) ne, ale pozor na billion laughs a overflow.
